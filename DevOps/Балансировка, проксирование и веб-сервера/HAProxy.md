
**HAProxy** (High Availability Proxy) — это одно из самых популярных в мире решений с открытым исходным кодом для балансировки нагрузки (load balancing) и работы в качестве обратного прокси-сервера (reverse proxy).

Он написан на языке **C**, что делает его чрезвычайно быстрым, легковесным и надежным. HAProxy используется тысячами компаний (включая GitHub, Twitter, Instagram, Reddit) для распределения трафика между серверами.

**Основные задачи HAProxy:**

1.  **Балансировка нагрузки:** Распределение входящих запросов пользователей между несколькими серверами (бэкендами), чтобы ни один сервер не был перегружен.
2.  **Обратный прокси:** Клиенты обращаются к HAProxy, а он перенаправляет запросы на внутренние серверы. Клиент не знает реальных адресов бэкендов (это повышает безопасность).
3.  **Высокая доступность (High Availability):** Если один из серверов падает, HAProxy автоматически перестает отправлять на него трафик и направляет запросы на рабочие узлы.
4.  **SSL/TLS Терминация:** HAProxy может расшифровывать HTTPS-трафик, снимая нагрузку с бэкенд-серверов, и передавать данные внутри сети уже в открытом виде (HTTP).

**Уровни работы (OSI Model).** HAProxy уникален тем, что может работать на разных уровнях сетевой модели OSI:

*   **Уровень 4 (TCP / Transport Layer):**
    *   Работает с IP-адресами и портами.
    *   Не анализирует содержимое запроса (не видит URL, куки, заголовки).
    *   **Плюсы:** Очень высокая производительность, низкая задержка.
    *   **Применение:** Базы данных (MySQL, PostgreSQL), SMTP, SSH, любые TCP-сервисы.
*   **Уровень 7 (HTTP / Application Layer):**
    *   Анализирует содержимое HTTP-запроса (URL, заголовки, куки, метод запроса).
    *   Позволяет принимать умные решения (например, запросы к `/api` отправлять на одни серверы, а к `/static` — на другие).
    *   **Плюсы:** Гибкость, возможность кэширования, сложная маршрутизация.
    *   **Применение:** Веб-сайты, API, микросервисы

**Ключевые возможности:**

*   **Алгоритмы балансировки:**
    *   `roundrobin`: по очереди (стандарт).
    *   `leastconn`: на сервер с наименьшим количеством активных соединений.
    *   `source`: привязка к IP клиента (хэш).
    *   `uri`: хэш от URL (полезно для кэширования).
*   **Health Checks (Проверка здоровья):** HAProxy постоянно «пингует» бэкенды. Если сервер не отвечает, он помечается как `DOWN` и исключается из ротации.
*   **Session Persistence (Липкие сессии):** Возможность направить одного и того же пользователя всегда на один и тот же сервер (через куки или IP), что важно для приложений, хранящих сессию в памяти.
*   **ACL (Access Control Lists):** Гибкие правила для блокировки IP, ограничения скорости (rate limiting) или маршрутизации по домену/пути.
*   **Панель статистики (Stats Page):** Встроенный веб-интерфейс, где видно состояние серверов, количество запросов, ошибки и возможность вручную включать/выключать узлы.
*   **Горячая перезагрузка:** Возможность обновлять конфигурацию без разрыва существующих соединений (в новых версиях).

### Настройка балансировки:

#### Базовая конфигурация:

```haproxy
#---------------------------------------------------------------------
# Глобальные настройки
#---------------------------------------------------------------------
global
    log /dev/log local0          # Отправка логов в syslog
    log /dev/log local1 notice   # Второй уровень логирования
    chroot /var/lib/haproxy      # Изоляция процесса (безопасность)
    stats socket /run/haproxy/admin.sock mode 660 level admin expose-fd listeners
                                 # Сокет для управления без перезагрузки (Runtime API)
    stats timeout 30s            # Таймаут для ответов через сокет
    user haproxy                 # Пользователь, от имени которого работает процесс
    group haproxy                # Группа
    daemon                       # Запуск в фоновом режиме
    maxconn 4096                 # Максимальное количество одновременных соединений

#---------------------------------------------------------------------
# Настройки по умолчанию
#---------------------------------------------------------------------
defaults
    log     global               # Использовать настройки логирования из global
    mode    http                 # Режим работы (http или tcp)
    option  httplog              # Логировать в формате HTTP
    option  dontlognull          # Не логировать пустые соединения (сканеры портов)
    option  forwardfor           # Передавать реальный IP клиента в заголовке X-Forwarded-For
    option  http-server-close    # Закрывать соединение с сервером после ответа (экономия ресурсов)
    timeout connect 5000         # Таймаут подключения к бэкенду (5 сек)
    timeout client  50000        # Таймаут активности клиента (50 сек)
    timeout server  50000        # Таймаут активности сервера (50 сек)
    retries 3                    # Количество попыток переподключения при ошибке

#---------------------------------------------------------------------
# Фронтенд (Входная точка для клиентов)
#---------------------------------------------------------------------
frontend http_front
    bind *:80                    # Слушать 80 порт на всех интерфейсах
    
    # ACL (Access Control Lists) - правила фильтрации
    acl is_api path_beg /api     # Если путь начинается с /api
    
    # Маршрутизация
    use_backend api_servers if is_api  # Если ACL выполнен, идти в api_servers
    default_backend web_servers        # Иначе идти в web_servers

#---------------------------------------------------------------------
# Бэкенд (Группа серверов для веб-трафика)
#---------------------------------------------------------------------
backend web_servers
    balance roundrobin           # Алгоритм балансировки (по кругу)
    option httpchk GET /health   # Проверка здоровья: запрос к пути /health
    http-check expect status 200 # Ожидать код ответа 200
    
    # Описание серверов
    # server <имя> <адрес>:<порт> [параметры]
    server web1 192.168.1.10:8080 check inter 5s fall 3 rise 2
    server web2 192.168.1.11:8080 check inter 5s fall 3 rise 2
    # check: включить проверку
    # inter 5s: проверять каждые 5 секунд
    # fall 3: считать сервер упавшим после 3 неудачных проверок
    # rise 2: считать сервер рабочим после 2 успешных проверок

#---------------------------------------------------------------------
# Бэкенд (Группа серверов для API)
#---------------------------------------------------------------------
backend api_servers
    balance leastconn            # Отправлять на сервер с наименьшим кол-вом соединений
    server api1 192.168.1.20:3000 check
    server api2 192.168.1.21:3000 check

#---------------------------------------------------------------------
# Статистика (Панель мониторинга)
#---------------------------------------------------------------------
listen stats
    bind *:8404                  # Статистика доступна на порту 8404
    stats enable                 # Включить страницу статистики
    stats uri /stats             # Путь к странице
    stats refresh 10s            # Автообновление каждые 10 секунд
    stats auth admin:password    # Логин:пароль для доступа (Basic Auth)
    stats admin if TRUE          # Разрешить управление серверами через веб-интерфейс
```

### Разбор ключевых секций

1.  **`global`**: Настройки, влияющие на весь процесс HAProxy (процессор, память, логи, права доступа). Изменение этих настроек обычно требует полной перезагрузки службы.
2.  **`defaults`**: Шаблоны настроек, которые наследуются всеми секциями `frontend`, `backend` и `listen`, если они не переопределены там явно. Сюда выносят таймауты и режим работы.
3.  **`frontend`**: Описывает, как HAProxy принимает соединения от клиентов. Здесь настраиваются порты (`bind`), SSL-сертификаты и правила маршрутизации (ACL).
4.  **`backend`**: Описывает пул серверов, куда пересылается трафик. Здесь выбирается алгоритм балансировки (`balance`) и настраиваются проверки здоровья (`check`).
5.  **`listen`**: Комбинирует функции frontend и backend. Часто используется для сервисов, работающих по TCP (например, базы данных) или для страницы статистики.
