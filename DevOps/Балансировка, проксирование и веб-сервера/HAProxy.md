**HAProxy** (High Availability Proxy) — это одно из самых популярных в мире решений с открытым исходным кодом для балансировки нагрузки (load balancing) и работы в качестве обратного прокси-сервера (reverse proxy).

Он написан на языке **C**, что делает его чрезвычайно быстрым, легковесным и надежным. HAProxy используется тысячами компаний (включая GitHub, Twitter, Instagram, Reddit) для распределения трафика между серверами.

Вот подробный обзор того, что нужно знать о HAProxy.

---

### 1. Основные задачи HAProxy

1.  **Балансировка нагрузки:** Распределение входящих запросов пользователей между несколькими серверами (бэкендами), чтобы ни один сервер не был перегружен.
2.  **Обратный прокси:** Клиенты обращаются к HAProxy, а он перенаправляет запросы на внутренние серверы. Клиент не знает реальных адресов бэкендов (это повышает безопасность).
3.  **Высокая доступность (High Availability):** Если один из серверов падает, HAProxy автоматически перестает отправлять на него трафик и направляет запросы на рабочие узлы.
4.  **SSL/TLS Терминация:** HAProxy может расшифровывать HTTPS-трафик, снимая нагрузку с бэкенд-серверов, и передавать данные внутри сети уже в открытом виде (HTTP).

---

### 2. Уровни работы (OSI Model)

HAProxy уникален тем, что может работать на разных уровнях сетевой модели OSI:

*   **Уровень 4 (TCP / Transport Layer):**
    *   Работает с IP-адресами и портами.
    *   Не анализирует содержимое запроса (не видит URL, куки, заголовки).
    *   **Плюсы:** Очень высокая производительность, низкая задержка.
    *   **Применение:** Базы данных (MySQL, PostgreSQL), SMTP, SSH, любые TCP-сервисы.
*   **Уровень 7 (HTTP / Application Layer):**
    *   Анализирует содержимое HTTP-запроса (URL, заголовки, куки, метод запроса).
    *   Позволяет принимать умные решения (например, запросы к `/api` отправлять на одни серверы, а к `/static` — на другие).
    *   **Плюсы:** Гибкость, возможность кэширования, сложная маршрутизация.
    *   **Применение:** Веб-сайты, API, микросервисы.

---

### 3. Ключевые возможности

*   **Алгоритмы балансировки:**
    *   `roundrobin`: по очереди (стандарт).
    *   `leastconn`: на сервер с наименьшим количеством активных соединений.
    *   `source`: привязка к IP клиента (хэш).
    *   `uri`: хэш от URL (полезно для кэширования).
*   **Health Checks (Проверка здоровья):** HAProxy постоянно «пингует» бэкенды. Если сервер не отвечает, он помечается как `DOWN` и исключается из ротации.
*   **Session Persistence (Липкие сессии):** Возможность направить одного и того же пользователя всегда на один и тот же сервер (через куки или IP), что важно для приложений, хранящих сессию в памяти.
*   **ACL (Access Control Lists):** Гибкие правила для блокировки IP, ограничения скорости (rate limiting) или маршрутизации по домену/пути.
*   **Панель статистики (Stats Page):** Встроенный веб-интерфейс, где видно состояние серверов, количество запросов, ошибки и возможность вручную включать/выключать узлы.
*   **Горячая перезагрузка:** Возможность обновлять конфигурацию без разрыва существующих соединений (в новых версиях).

### 4. Простой пример конфигурации

Конфигурационный файл обычно находится по пути `/etc/haproxy/haproxy.cfg`. Он делится на секции.

```haproxy
global
    log /dev/log local0
    maxconn 4096
    user haproxy
    group haproxy
    daemon

defaults
    log     global
    mode    http                # Работаем на уровне HTTP
    option  httplog
    option  dontlognull
    timeout connect 5000ms      # Время ожидания подключения к бэкенду
    timeout client  50000ms     # Время ожидания активности от клиента
    timeout server  50000ms     # Время ожидания активности от сервера

# Фронтенд (куда стучится клиент)
frontend http_front
    bind *:80
    default_backend web_servers

# Бэкенд (куда пересылаем трафик)
backend web_servers
    balance roundrobin          # Алгоритм балансировки
    server web1 192.168.1.10:80 check
    server web2 192.168.1.11:80 check
    server web3 192.168.1.12:80 check
```

В этом примере HAProxy слушает 80 порт и распределяет запросы по кругу между тремя серверами, периодически проверяя их доступность (`check`).

### 5. HAProxy vs Nginx

Часто возникает вопрос: что выбрать, HAProxy или Nginx?

| Характеристика | HAProxy | Nginx |
| :--- | :--- | :--- |
| **Основное назначение** | Специализированный балансировщик и прокси. | Веб-сервер + прокси + балансировщик. |
| **Производительность (TCP)** | Часто быстрее и стабильнее на чистом TCP. | Очень хорошо, но HAProxy имеет небольшое преимущество в специфичных сценариях. |
| **Конфигурация** | Строгая, специфичная. | Более гибкая, похожа на конфиги веб-сервера. |
| **Обработка статики** | Не умеет (только проксирует). | Отлично раздает статические файлы. |
| **Динамическая конфигурация** | Поддерживает Runtime API (изменение без перезагрузки). | Требует перезагрузки конфига (хотя есть модули). |

**Итог:**
*   Если вам нужен просто **веб-сервер** с балансировкой — берите **Nginx**.
*   Если вам нужна **сложная балансировка**, работа с базами данных (TCP), высокая надежность и продвинутые проверки здоровья — берите **HAProxy**.
*   **Частая связка:** HAProxy стоит на входе (как балансировщик), а за ним стоят серверы Nginx (как веб-серверы).

### 6. Версии: Community vs Enterprise

*   **HAProxy Community:** Полностью бесплатная, с открытым кодом. Покрывает 95% потребностей большинства компаний. Обновляется часто.
*   **HAProxy Enterprise:** Платная версия от компании HAProxy Technologies. Включает дополнительную модульность (WAF, гео-балансировка, более продвинутая аналитика), официальную поддержку 24/7 и гарантированную стабильность версий (LTS).

---

### 7. Плюсы и минусы

**Плюсы:**
*   ✅ Очень высокая производительность и низкое потребление ресурсов.
*   ✅ Отличная документация.
*   ✅ Надежность (работает годами без перезагрузки).
*   ✅ Глубокая работа с протоколами (HTTP/2, gRPC, WebSocket, TCP).
*   ✅ Мощный инструмент для отладки (логов и статистики).

**Минусы:**
*   ❌ Кривая обучения: синтаксис конфига может показаться сложным новичку.
*   ❌ Не умеет отдавать статический контент (картинки, html) сам по себе.
*   ❌ В старых версиях изменение конфига требовало перезагрузки (сейчас решено через Runtime API).

### Заключение

HAProxy — это «золотой стандарт» индустрии для балансировки нагрузки. Если вы строите отказоустойчивую архитектуру, где важно грамотно распределять трафик между множеством серверов, знание HAProxy является обязательным навыком для системного администратора или DevOps-инженера.
