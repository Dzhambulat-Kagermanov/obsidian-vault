### Основы:

**Apache HTTP Server** - это свободное кроссплатформенное программное обеспечение с открытым исходным кодом, используемое для создания и управления веб-серверами. Он обрабатывает запросы пользователей, обеспечивает соединение между браузером и сервером, а также передает файлы сайта (HTML, CSS, изображения), являясь одним из самых популярных решений для хостинга. Это свободное программное обеспечение с открытым исходным кодом, которое разрабатывается и поддерживается сообществом под эгидой **Apache Software Foundation (ASF)**.

> Apache работает от специального пользователя (обычно `www-data` в Debian/Ubuntu или `apache` в CentOS). Этот пользователь должен иметь право **читать** файлы необходимые в процессах.

#### Ключевые особенности:

*  **Кроссплатформенность:** Работает практически на любой ОС: Linux (Ubuntu, CentOS, Debian), Windows, macOS, Unix-системы;
*   **Модульная архитектура:** Функциональность сервера расширяется с помощью модулей. Вы можете включить только то, что нужно (например, `mod_ssl` для HTTPS, `mod_rewrite` для URL, `mod_proxy` для проксирования);
*   **Поддержка `.htaccess`:** Это уникальная фишка Apache. Файл `.htaccess` позволяет настраивать сервер **подробно для каждой директории** без перезагрузки основного сервиса. Это критически важно для **виртуального хостинга** (когда на одном сервере много разных пользователей), так как позволяет пользователям менять настройки своего сайта самостоятельно;
*   **Динамический контент:** Исторически очень тесно связан с PHP (через модуль `mod_php`), хотя сейчас чаще используется в связке с PHP-FPM.

#### Архитектура обработки запросов (MPM):

Apache использует **MPM (Multi-Processing Modules)** для обработки входящих соединений. Выбор MPM влияет на производительность:

1.  **Prefork:** Создает отдельный процесс для каждого запроса. Очень стабильный, совместим со старыми модулями (например, старым `mod_php`), но потребляет много оперативной памяти при высокой нагрузке;
2.  **Worker:** Использует многопоточность. Процессы порождают потоки. Экономнее по памяти, чем prefork, но сложнее в отладке;
3.  **Event (рекомендуемый):** Современный вариант. Оптимизирован для обработки множества длительных соединений (например, Keep-Alive). Работает асинхронно, похож на архитектуру Nginx.

#### Архитектура и Обработка запросов (MPM):

**Process (Процесс)** - это изолированный экземпляр программы в ОС. В Apache (в зависимости от MPM) может быть главный процесс (master) и дочерние (children).

**Thread (Поток)** - это легковесная единица выполнения внутри процесса. Используется в MPM Worker и Event.

**Listen** - это директива, указывающая, на каких IP-адресах и портах (обычно 80 и 443) сервер слушает входящие соединения.

**VirtualHost (Виртуальный хост)** - это механизм размещения нескольких сайтов на одном сервере (одном IP). Конфигурация оборачивается в теги `<VirtualHost *:80>`.

**Alias** - это директива, позволяющая сопоставить URL не с `DocumentRoot`, а с любой другой папкой на сервере.

#### Модули и Расширяемость:

**Module (Модуль)** - это компонент, добавляющий функциональность. Имена обычно начинаются с `mod_`.

**DSO (Dynamic Shared Object)** - это динамически загружаемые модули. Их можно включать и выключать без перекомпиляции сервера (команды `a2enmod` / `a2dismod` в Debian/Ubuntu).

#### Конвейер обработки запроса (Request Pipeline):

**Handler (Обработчик)** - определяет, _какой модуль_ должен обработать файл. Например, файл `.php` имеет обработчик `application/x-httpd-php`, который передает файл интерпретатору PHP. Файл `.html` обрабатывается самим Apache.

**Filter (Фильтр)** - это модули, которые обрабатывают контент _до_ отправки клиенту или _после_ получения от клиента.

**Hooks (Хуки)** - это точки внутри кода Apache, к которым могут подключаться модули, чтобы выполнить свой код на определенном этапе обработки запроса.

#### Плюсы и Минусы:

| Плюсы                                                            | Минусы                                                                                                                         |
| :--------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------- |
| **Гибкость:** Огромное количество модулей для любых задач.       | **Производительность:** При очень высокой нагрузке (тысячи одновременных соединений) уступает Nginx в скорости отдачи статики. |
| **`.htaccess`:** Удобно для разработчиков и хостинг-провайдеров. | **Потребление памяти:** Режим `prefork` может "съесть" много RAM при пиковых нагрузках.                                        |
| **Стабильность:** Проверен десятилетиями, очень надежен.         | **Сложность конфигурации:** Файлы конфига могут стать громоздкими и запутанными.                                               |
| **Сообщество:** Любую проблему можно загуглить, решений масса.   | **Динамическая перезагрузка:** Изменения в главном конфиге требуют перезагрузки сервиса (в отличие от `.htaccess`).            |

#### Актуальность сегодня:

Многие говорят, что "Apache умер" из-за роста Nginx, но **это не так**:

*   Согласно статистике (например, W3Techs), Apache до сих пор используется на **~30% всех веб-сайтов** мира;
*   Он остается стандартом де-факто для **дешевого виртуального хостинга** (Shared Hosting) именно из-за `.htaccess`;
*   В корпоративной среде и для специфических задач он по-прежнему незаменим.

#### Конфигурация:

*   **Основные файлы:**
    *   В Debian/Ubuntu: `/etc/apache2/apache2.conf`;
    *   В RHEL/CentOS: `/etc/httpd/conf/httpd.conf`
*   **Virtual Hosts (Виртуальные хосты):** Позволяет размещать несколько доменов на одном IP-адресе. Конфигурация обычно лежит в папках `sites-available` / `sites-enabled` (Debian) или `conf.d` (CentOS);
*   **Управление:** Команды зависят от ОС (например, `systemctl restart apache2` или `systemctl restart httpd`).

#### Логирование и Мониторинг:

- **Access Log (Журнал доступа):** Записывает каждый запрос к серверу (кто, когда, какой URL, статус ответа). Директива: `CustomLog`;
- **Error Log (Журнал ошибок):** Записывает проблемы в работе сервера, ошибки скриптов, проблемы с правами доступа. Директива: `ErrorLog`;
- **Log Level:** Уровень детализации логов (например, `warn`, `error`, `debug`, `info`);
- **Log Format:** Шаблон записи в лог (директива `LogFormat`), определяющий, какие данные записывать (IP, время, User-Agent и т.д.).

#### Безопасность и Доступ:

- **Authentication (Аутентификация):** Проверка личности пользователя (Логин/Пароль). Модули: `mod_auth_basic`, `mod_auth_digest`;
- **Authorization (Авторизация):** Проверка прав доступа (кому разрешено). Директивы: `Require`, `Allow`, `Deny` (в старых версиях);
- **SSL/TLS:** Настройки шифрования (сертификаты, ключи, протоколы). Настраиваются обычно в контексте `<VirtualHost *:443>`.

#### Управление состоянием:

- **Graceful Restart:** Перезагрузка конфигурации без разрыва текущих соединений (клиенты дослуживают, новые запросы идут по новой конфигурации). Команда: `apachectl graceful` или `systemctl reload apache2`;
- **Status Page:** Специальная страница (`/server-status`), доступная через модуль `mod_status`, которая показывает текущую нагрузку, активные запросы и состояние воркеров в реальном времени.


### Конфигурация Apache:

#### Шаг 1. Установка пакетов:

```bash
apt-get update && apt-get install apache2 -y
```

После установки сервер нужно запустить:

```bash
systemctl enable --now apache2
```

#### Структура директорий:

Важно понимать, где что лежит:

- `/etc/apache2/` — главная папка конфигурации:
    - `apache2.conf` — основной файл настроек;
    - `ports.conf` — настройки портов (80, 443);
    - `sites-available/` — здесь хранятся конфиги всех сайтов;
    - `sites-enabled/` — здесь лежат **ссылки** на активные сайты из `sites-available`;
    - `mods-available/` и `mods-enabled/` — аналогично для модулей;

#### Переменные в конфигурации:

Apache поддерживает **подстановку переменных окружения** через синтаксис `${VAR_NAME}`.

* `/etc/apache2/envvars`:

```bash
# Пример содержимого envvars:
export APACHE_LOG_DIR=/var/log/apache2
export APACHE_PID_FILE=/var/run/apache2/apache2.pid
export APACHE_RUN_USER=www-data
export APACHE_RUN_GROUP=www-data
```

* Системные переменные окружения, доступные процессу Apache;
* Встроенные переменные Apache:
	* - `${SERVER_NAME}`, `${SERVER_PORT}`;
	- `${DOCUMENT_ROOT}`, `${REQUEST_URI}`;



#### Создание виртуального хоста (Сайта):

##### Шаг 1. Создание директории для сайта:

```bash
mkdir -p /var/www/example.com/public_html
```

**Apache НЕ требует** структуры `/var/www/example.com/public_html`. Можно разместить файлы сайта **в любой папке**, к которой у Apache есть доступ. **Почему используется структуру `/var/www/example.com/public_html`?** Это конвенция (негласное правило), а не техническое требование.

| Часть пути     | Зачем это нужно?                                                                                                    | Можно ли изменить?                                                             |
| -------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `/var/www/`    | Стандартная директория для веб-контента в Linux (FHS — Filesystem Hierarchy Standard).                              | **Да.** Можно использовать `/srv/www/`, `/home/user/`, `/opt/sites/` и т.д.    |
| `example.com/` | Папка с именем домена. Помогает админу визуально понять, какой сайт где лежит, особенно если на сервере их десятки. | **Да.** Можно назвать папку `site1`, `myshop`, `abc123` — Apache всё равно.    |
| `public_html/` | Разделение: здесь лежат публичные файлы, а конфиги или приватные данные — на уровень выше.                          | **Да.** Часто используют просто корень сайта или папку `html`, `public`, `web` |

##### Шаг 2. Назначьте права доступа:

Apache работает от пользователя `www-data`. Нужно отдать ему права на папку:

```bash
chown -R www-data:www-data /var/www/example.com/public_html
chmod -R 755 /var/www/example.com
```

##### Шаг 3. Создание тестовую страницу:

```bash
nano /var/www/example.com/public_html/index.html
```

Вставьте туда простой HTML:

```html
<html>
    <head><title>Привет, Apache!</title></head>
    <body><h1>Сайт работает на Apache2</h1></body>
</html>
```

##### Шаг 4. Создание конфига виртуального хоста:

```bash
nano /etc/apache2/sites-available/example.com.conf
```

| Требование                               | Реальность                                                                                                                       |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **Имя должно совпадать с доменом?**      | ❌ **Нет.** Файл можно назвать `mysite.conf`, `site123.conf`, `aaa.conf` — Apache всё равно.                                      |
| **Расширение должно быть `.conf`?**      | ⚠️ **Рекомендуется.** Apache по умолчанию читает только `*.conf` в некоторых директориях, но это настраиваемо.                   |
| **Папка должна быть `sites-available`?** | ⚠️ **Зависит от дистрибутива.** В Debian/Ubuntu — да, это стандарт. В CentOS/RHEL — конфиги кладут сразу в `/etc/httpd/conf.d/`. |

В **Debian/Ubuntu** используется двухуровневая система:

```Structure
/etc/apache2/
├── sites-available/     # Здесь хранятся все конфиги сайтов (как библиотека)
│   ├── example.com.conf
│   ├── test-site.conf
│   └── 000-default.conf
│
└── sites-enabled/       # Здесь только символьные ссылки на активные сайты
    ├── example.com.conf -> ../sites-available/example.com.conf
    └── 000-default.conf -> ../sites-available/000-default.conf
```

**Как это работает:**

1. Создается файл в `sites-available` (имя — любое, но лучше понятное);
2. Команда `sudo a2ensite example.com.conf` создает **символическую ссылку** в `sites-enabled`;
3. В главном конфиге `apache2.conf` есть строка:

```vim
IncludeOptional sites-enabled/*.conf
# Она говорит Apache: "Загрузи все `.conf` файлы из папки `sites-enabled`".
```

```bash
<VirtualHost *:80>
    # Основной домен сайта (обязательно для виртуальных хостов)
    ServerName example.com
    
    # Дополнительные домены (www, старые домены, поддомены)
    ServerAlias www.example.com blog.example.com
    
    # Email администратора (отображается на страницах ошибок 500)
    ServerAdmin webmaster@example.com
    
    # ============================================================
    # ПУТИ К КОНТЕНТУ
    # ============================================================
    # Корневая директория сайта (файлы, которые будут отдаваться)
    DocumentRoot /var/www/example.com/public_html
    
    # Файлы, которые искать по умолчанию при заходе в директорию
    DirectoryIndex index.php index.html index.htm
    
    # ============================================================
    # НАСТРОЙКИ ДОСТУПА К ДИРЕКТОРИИ
    # ============================================================
    <Directory /var/www/example.com/public_html>
        # Опции обработки контента:
        # -Indexes: запретить листинг файлов (если нет index)
        # +FollowSymLinks: разрешить символические ссылки (нужно для ЧПУ)
        Options -Indexes +FollowSymLinks
        
        # Разрешить .htaccess переопределять настройки
        # All = всё, None = игнорировать, FileInfo = только редиректы/заголовки
        AllowOverride All
        
        # Контроль доступа (синтаксис Apache 2.4+)
        Require all granted
        
        # Если нужно ограничить доступ по IP:
        # Require ip 192.168.1.0/24
        # Require not ip 10.0.0.5
    </Directory>
    
    # Запретить доступ к служебным файлам (дополнительная защита)
    <FilesMatch "^\.">
        Require all denied
    </FilesMatch>
    
    # ============================================================
    # ЛОГИРОВАНИЕ
    # ============================================================
    # Файл для ошибок этого сайта
    ErrorLog ${APACHE_LOG_DIR}/example.com_error.log
    
    # Файл для логов доступа
    CustomLog ${APACHE_LOG_DIR}/example.com_access.log combined
    
    # Уровень детализации ошибок: emerg|alert|crit|error|warn|notice|info|debug
    LogLevel warn
    
    # ============================================================
    # ПЕРЕАДРЕСАЦИИ И ЧПУ (mod_rewrite)
    # ============================================================
    <IfModule mod_rewrite.c>
        RewriteEngine On
        
        # Принудительный редирект на www (опционально)
        RewriteCond %{HTTP_HOST} ^example\.com [NC]
        RewriteRule ^(.*)$ https://www.example.com/$1 [L,R=301]
        
        # Стандартные правила для ЧПУ (например, для WordPress)
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteCond %{REQUEST_FILENAME} !-d
        RewriteRule . /index.php [L]
    </IfModule>
    
    # Простой редирект (без mod_rewrite)
    # Redirect 301 /old-page.html https://example.com/new-page.html
    
    # ============================================================
    # ПРОКСИРОВАНИЕ (если сайт работает на бэкенде)
    # ============================================================
    # <IfModule mod_proxy.c>
    #     ProxyPreserveHost On
    #     ProxyPass / http://127.0.0.1:3000/
    #     ProxyPassReverse / http://127.0.0.1:3000/
    # </IfModule>
    
    # ============================================================
    # КЭШИРОВАНИЕ И СЖАТИЕ (на уровне хоста)
    # ============================================================
    # <IfModule mod_expires.c>
    #     ExpiresActive On
    #     ExpiresByType image/jpg "access plus 1 month"
    #     ExpiresByType text/css "access plus 1 week"
    # </IfModule>
    
    # <IfModule mod_deflate.c>
    #     AddOutputFilterByType DEFLATE text/html text/css application/javascript
    # </IfModule>
    
    # ============================================================
    # БЕЗОПАСНОСТЬ (заголовки)
    # ============================================================
    # <IfModule mod_headers.c>
    #     Header always set X-Frame-Options "SAMEORIGIN"
    #     Header always set X-Content-Type-Options "nosniff"
    #     Header always set X-XSS-Protection "1; mode=block"
    # </IfModule>
    
</VirtualHost>
```

**Идентификация виртуального хоста:**

* **ServerName** - Основной домен, на который реагирует этот блок. Обязательная директива. Пример: `example.com`;

* **ServerAlias** - Дополнительные имена (алиасы) для этого хоста. Необязательная директива. Пример: `www.example.com`;

* **ServerAdmin** - Контактный email, показывается на страницах ошибок.. Необязательная директива. Пример: `admin@example.com`

**Работа с файлами и путями:**

* **DocumentRoot** - Путь к корневой папке сайта. Все относительные пути в запросах считаются отсюда. Пример: `/var/www/site/html`.

* **DirectoryIndex** - Список файлов, которые искать по умолчанию при запросе директории (например, при заходе на `example.com/`). Пример: `index.php index.html`.

* **Alias** - Позволяет отдавать файлы из папки, не входящей в `DocumentRoot`. Запрос `/images/logo.png` отдаст файл из `/data/pics/logo.png`. Пример: `Alias /images /data/pics`

* **AliasMatch** - То же, что Alias, но с поддержкой регулярных выражений. Пример: `AliasMatch ^/backup/(.*) /srv/backup/$1`

**Контроль доступа (`<Directory>`, `<Files>`, `<Location>`):**

* **Options** - Это директива управляет тем, **что можно делать с файлами** в данной директории:
	* `Indexes` - С директивой apache показывает список всех файлов в папке (как в проводнике), без директивы показывается ошибка **403 Forbidden**;
	* `FollowSymLinks` - указывает на другой файл/папку;
	* `ExecCGI` - запрещает выполнение CGI-скриптов (безопасность);
	* `Includes` - разрешает Server Side Includes (SSI) — вставка контента на уровне Apache;
	* `MultiViews` - Включает согласование контента (например, авто-выбор языка);

* **AllowOverride** - Определяет, какие директивы можно переопределять через `.htaccess` в этой папке.

* **\<FilesMatch\>** - Блок для применения правил к файлам по регулярному выражению.

* **\<Location\>** - Применяет правила к URL-пути, а не к пути на диске. Пример:

```vim
# Защита админ-панели по IP

<Location "/admin">
    # Разрешить доступ только с рабочего офиса
    Require ip 203.0.113.50
    Require ip 192.168.1.0/24
    
    # Дополнительная аутентификация
    AuthType Basic
    AuthName "Admin Area"
    AuthUserFile /etc/apache2/.htpasswd
    Require valid-user
</Location>
```

* **Require** - (Apache 2.4) директива **авторизации**: кто имеет право получать доступ к ресурсу:
	* `Require all granted` - разрешить доступ **всем** (стандарт для публичных сайтов);
	* Запретить доступ **всем** (для блокировки)
	* `Require all denied` - запретить доступ **всем** (для блокировки);
	* `Require ip 192.168.1.10` - разрешить только с указанного IP;
	* `Require ip 192.168.1.0/24` - разрешить целой подсети;
	* `Require valid-user` - разрешить любому, кто прошел аутентификацию (логин/пароль)
	* `Require user admin dzhambulat` - разрешить только конкретным пользователям.

| Директива     | Пример                              | Описание                                                                                              |
| ------------- | ----------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **ErrorLog**  | `${APACHE_LOG_DIR}/error.log`       | Путь к файлу логов ошибок. Переменная `${APACHE_LOG_DIR}` обычно определена в `/etc/apache2/envvars`. |
| **CustomLog** | `.../access.log combined`           | Путь к логам доступа. `combined` — формат, включающий реферер и User-Agent.                           |
| **LogLevel**  | `warn`, `info`, `debug`             | Минимальный уровень сообщений для записи в `ErrorLog`.                                                |
| **LogFormat** | `"%v:%p %h %l %u %t \"%r\" %>s %O"` | Позволяет задать свой формат лога вместо стандартных `combined`, `common`.                            |

**Перенаправления и ЧПУ (`mod_rewrite`, `mod_alias`):**

| Директива         | Пример                      | Описание                                                                                                                    |
| ----------------- | --------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **RewriteEngine** | `On`                        | Включает/выключает движок переписывания URL.                                                                                |
| **RewriteCond**   | `%{HTTPS} off`              | Условие для следующего `RewriteRule`. Проверяет переменные (хост, протокол, путь и т.д.).                                   |
| **RewriteRule**   | `^old$ /new [R=301,L]`      | Правило замены: если путь совпадает с `^old$`, перенаправить на `/new`. Флаги: `R=301` (редирект), `L` (последнее правило). |
| **Redirect**      | `301 /old /new`             | Простой редирект без регулярных выражений. Код 301 (постоянный) или 302 (временный).                                        |
| **RedirectMatch** | `301 ^/blog/(.*)$ /news/$1` | Редирект с поддержкой регулярных выражений.                                                                                 |

##### Шаг 5. Включаем сайт:

Apache не читает файлы из `sites-available` напрямую, нужно создать ссылку в `sites-enabled`:

```bash
a2ensite example.com.conf
```

##### Шаг 6. Отключаем сайт по умолчанию (опционально):

```bash
a2dissite 000-default.conf
```

##### Шаг 7. Проверяем конфигурацию и перезагрузите:

```bash
apache2ctl configtest
```

Теперь по домену `example.com` (или IP, если домен еще не настроен в DNS) должен открываться ваш сайт.

#### Частые ошибки и решения:

1. **Ошибка 403 Forbidden:**
    - Проверьте права на папку: `chown -R www-data:www-data /var/www/...`.
    - Проверьте директиву `Require all granted` в конфиге.
2. **Ошибка 500 Internal Server Error:**
    - Смотрите `error.log`. Часто это ошибка в синтаксисе `.htaccess` или скрипта PHP.
3. **Изменения не применяются:**
    - Забыли сделать `reload` или `restart`?
    - Исправили файл в `sites-available`, но не сделали `a2ensite`?
    - Кэш браузера? Попробуйте Ctrl+F5.
4. **Syntax error on line X:**
    - Запустите `sudo apache2ctl configtest`, он укажет номер строки с ошибкой.
