### Основы:

**Apache HTTP Server** - это свободное кроссплатформенное программное обеспечение с открытым исходным кодом, используемое для создания и управления веб-серверами. Он обрабатывает запросы пользователей, обеспечивает соединение между браузером и сервером, а также передает файлы сайта (HTML, CSS, изображения), являясь одним из самых популярных решений для хостинга. Это свободное программное обеспечение с открытым исходным кодом, которое разрабатывается и поддерживается сообществом под эгидой **Apache Software Foundation (ASF)**.

> Apache работает от специального пользователя (обычно `www-data` в Debian/Ubuntu или `apache` в CentOS). Этот пользователь должен иметь право **читать** файлы необходимые в процессах.

#### Ключевые особенности:

*  **Кроссплатформенность:** Работает практически на любой ОС: Linux (Ubuntu, CentOS, Debian), Windows, macOS, Unix-системы;
*   **Модульная архитектура:** Функциональность сервера расширяется с помощью модулей. Вы можете включить только то, что нужно (например, `mod_ssl` для HTTPS, `mod_rewrite` для URL, `mod_proxy` для проксирования);
*   **Поддержка `.htaccess`:** Это уникальная фишка Apache. Файл `.htaccess` позволяет настраивать сервер **подробно для каждой директории** без перезагрузки основного сервиса. Это критически важно для **виртуального хостинга** (когда на одном сервере много разных пользователей), так как позволяет пользователям менять настройки своего сайта самостоятельно;
*   **Динамический контент:** Исторически очень тесно связан с PHP (через модуль `mod_php`), хотя сейчас чаще используется в связке с PHP-FPM.

#### Архитектура обработки запросов (MPM):

Apache использует **MPM (Multi-Processing Modules)** для обработки входящих соединений. Выбор MPM влияет на производительность:

1.  **Prefork:** Создает отдельный процесс для каждого запроса. Очень стабильный, совместим со старыми модулями (например, старым `mod_php`), но потребляет много оперативной памяти при высокой нагрузке;
2.  **Worker:** Использует многопоточность. Процессы порождают потоки. Экономнее по памяти, чем prefork, но сложнее в отладке;
3.  **Event (рекомендуемый):** Современный вариант. Оптимизирован для обработки множества длительных соединений (например, Keep-Alive). Работает асинхронно, похож на архитектуру Nginx.

#### Архитектура и Обработка запросов (MPM):

**Process (Процесс)** - это изолированный экземпляр программы в ОС. В Apache (в зависимости от MPM) может быть главный процесс (master) и дочерние (children).

**Thread (Поток)** - это легковесная единица выполнения внутри процесса. Используется в MPM Worker и Event.

**Listen** - это директива, указывающая, на каких IP-адресах и портах (обычно 80 и 443) сервер слушает входящие соединения.

**VirtualHost (Виртуальный хост)** - это механизм размещения нескольких сайтов на одном сервере (одном IP). Конфигурация оборачивается в теги `<VirtualHost *:80>`.

**Alias** - это директива, позволяющая сопоставить URL не с `DocumentRoot`, а с любой другой папкой на сервере.

#### Модули и Расширяемость:

**Module (Модуль)** - это компонент, добавляющий функциональность. Имена обычно начинаются с `mod_`.

**DSO (Dynamic Shared Object)** - это динамически загружаемые модули. Их можно включать и выключать без перекомпиляции сервера (команды `a2enmod` / `a2dismod` в Debian/Ubuntu).

#### Конвейер обработки запроса (Request Pipeline):

**Handler (Обработчик)** - определяет, _какой модуль_ должен обработать файл. Например, файл `.php` имеет обработчик `application/x-httpd-php`, который передает файл интерпретатору PHP. Файл `.html` обрабатывается самим Apache.

**Filter (Фильтр)** - это модули, которые обрабатывают контент _до_ отправки клиенту или _после_ получения от клиента.

**Hooks (Хуки)** - это точки внутри кода Apache, к которым могут подключаться модули, чтобы выполнить свой код на определенном этапе обработки запроса.

#### Плюсы и Минусы:

| Плюсы                                                            | Минусы                                                                                                                         |
| :--------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------- |
| **Гибкость:** Огромное количество модулей для любых задач.       | **Производительность:** При очень высокой нагрузке (тысячи одновременных соединений) уступает Nginx в скорости отдачи статики. |
| **`.htaccess`:** Удобно для разработчиков и хостинг-провайдеров. | **Потребление памяти:** Режим `prefork` может "съесть" много RAM при пиковых нагрузках.                                        |
| **Стабильность:** Проверен десятилетиями, очень надежен.         | **Сложность конфигурации:** Файлы конфига могут стать громоздкими и запутанными.                                               |
| **Сообщество:** Любую проблему можно загуглить, решений масса.   | **Динамическая перезагрузка:** Изменения в главном конфиге требуют перезагрузки сервиса (в отличие от `.htaccess`).            |

#### Актуальность сегодня:

Многие говорят, что "Apache умер" из-за роста Nginx, но **это не так**:

*   Согласно статистике (например, W3Techs), Apache до сих пор используется на **~30% всех веб-сайтов** мира;
*   Он остается стандартом де-факто для **дешевого виртуального хостинга** (Shared Hosting) именно из-за `.htaccess`;
*   В корпоративной среде и для специфических задач он по-прежнему незаменим.

#### Конфигурация:

*   **Основные файлы:**
    *   В Debian/Ubuntu: `/etc/apache2/apache2.conf`;
    *   В RHEL/CentOS: `/etc/httpd/conf/httpd.conf`
*   **Virtual Hosts (Виртуальные хосты):** Позволяет размещать несколько доменов на одном IP-адресе. Конфигурация обычно лежит в папках `sites-available` / `sites-enabled` (Debian) или `conf.d` (CentOS);
*   **Управление:** Команды зависят от ОС (например, `systemctl restart apache2` или `systemctl restart httpd`).

#### Логирование и Мониторинг:

- **Access Log (Журнал доступа):** Записывает каждый запрос к серверу (кто, когда, какой URL, статус ответа). Директива: `CustomLog`;
- **Error Log (Журнал ошибок):** Записывает проблемы в работе сервера, ошибки скриптов, проблемы с правами доступа. Директива: `ErrorLog`;
- **Log Level:** Уровень детализации логов (например, `warn`, `error`, `debug`, `info`);
- **Log Format:** Шаблон записи в лог (директива `LogFormat`), определяющий, какие данные записывать (IP, время, User-Agent и т.д.).

#### Безопасность и Доступ:

- **Authentication (Аутентификация):** Проверка личности пользователя (Логин/Пароль). Модули: `mod_auth_basic`, `mod_auth_digest`;
- **Authorization (Авторизация):** Проверка прав доступа (кому разрешено). Директивы: `Require`, `Allow`, `Deny` (в старых версиях);
- **SSL/TLS:** Настройки шифрования (сертификаты, ключи, протоколы). Настраиваются обычно в контексте `<VirtualHost *:443>`.

#### Управление состоянием:

- **Graceful Restart:** Перезагрузка конфигурации без разрыва текущих соединений (клиенты дослуживают, новые запросы идут по новой конфигурации). Команда: `apachectl graceful` или `systemctl reload apache2`;
- **Status Page:** Специальная страница (`/server-status`), доступная через модуль `mod_status`, которая показывает текущую нагрузку, активные запросы и состояние воркеров в реальном времени.


### Конфигурация Apache:

#### Установка пакетов:

Обновляем списки пакетов и устанавливаем веб-сервер:

```bash
apt-get update && apt-get install apache2 -y
```

После установки активируем сервис и проверим его статус:

```bash
systemctl enable --now apache2
systemctl status apache2
```

> **Важно!** После любых изменений в конфигурации всегда проверяйте синтаксис командой `apache2ctl configtest` перед перезагрузкой (`systemctl reload apache2`).

#### Структура директорий:

Логика работы Apache в Debian/Ubuntu построена на модульности и использовании символических ссылок:

- `/etc/apache2/` — главная папка конфигурации:
    - `apache2.conf` — основной файл, который через директивы `Include` подгружает остальные части;
    - `ports.conf` — определяет, какие порты слушает сервер (директива `Listen`);
    - `sites-available/` — шаблоны конфигураций виртуальных хостов (неактивны по умолчанию);
    - `sites-enabled/` — содержит **симлинки** на файлы из `sites-available`. Управление осуществляется утилитами `a2ensite` / `a2dissite`;
    - `mods-available/` и `mods-enabled/` — аналогично для модулей (`a2enmod` / `a2dismod`);
    - `conf-available/` и `conf-enabled/` — глобальные фрагменты конфигурации (например, настройки безопасности или кодировок), управляемые через `a2enconf`.

#### Переменные в конфигурации (Расширено):

В Apache существует четкое разделение между **переменными окружения процесса** и **переменными запроса**. Понимание этой разницы критично для правильной настройки.

##### 1. Переменные окружения процесса (Environment Variables)
Эти переменные задаются **до** запуска процесса Apache и доступны везде в конфигурации через синтаксис `${ИМЯ}`.

*   **Источник:** Файл `/etc/apache2/envvars`.
*   **Назначение:** Управление путями, правами доступа и поведением самого демона.
*   **Примеры использования в конфигах:**
    ```apache
    # Использование переменной из envvars для пути к логам
    ErrorLog ${APACHE_LOG_DIR}/error.log
    CustomLog ${APACHE_LOG_DIR}/access.log combined
    
    # Использование для определения пользователя/группы
    User ${APACHE_RUN_USER}
    Group ${APACHE_RUN_GROUP}
    ```
*   **Как добавить свою:** Отредактируйте `/etc/apache2/envvars`, добавив строку `export MY_CUSTOM_VAR=/path/to/data`, затем перезапустите Apache.

##### 2. Встроенные переменные сервера (Server Context)
Некоторые значения известны серверу в момент чтения конфигурации (контекст сервера или виртуального хоста), но могут зависеть от конкретного запроса.

*   `${SERVER_NAME}` — имя сервера из директивы `ServerName`.
*   `${SERVER_PORT}` — порт, на котором обрабатывается запрос.
*   `${DOCUMENT_ROOT}` — корневая директория сайта.
    *   *Нюанс:* Эта переменная корректно работает внутри блоков `<VirtualHost>`, так как там `DocumentRoot` уже определен.

##### 3. Динамические переменные запроса (Request Variables)
**Важное уточнение:** Синтаксис `${...}` **НЕ работает** для динамических данных каждого конкретного HTTP-запроса (например, URI, заголовки, метод запроса) внутри обычных директив конфигурации (как `ErrorLog` или `Redirect`).

Для работы с данными запроса используются другие механизмы:

*   **Модуль `mod_rewrite`:** Использует синтаксис `%{VAR}` внутри правил `RewriteCond` и `RewriteRule`.
    ```apache
    # Пример: редирект только если пользователь использует мобильное устройство
    RewriteCond %{HTTP_USER_AGENT} "Mobile"
    RewriteRule ^/old-page$ /new-mobile-page [R=302,L]
    ```
    *Доступные переменные:* `%{REQUEST_URI}`, `%{QUERY_STRING}`, `%{HTTP_HOST}`, `%{HTTPS}`, `%{REMOTE_ADDR}` и десятки других.

*   **Модуль `mod_log_config` (Логирование):** Использует синтаксис `%{VAR}type` для записи в логи.
    ```apache
    # Запись конкретного заголовка 'X-Forwarded-For' в лог
    LogFormat "%h %l %u %t \"%r\" %>s %b \"%{Referer}i\" \"%{User-Agent}i\" \"%{X-Forwarded-For}i\"" combined_forwarded
    ```

*   **Модуль `mod_setenvif` и `mod_headers`:** Позволяют устанавливать внутренние переменные окружения на основе характеристик запроса, которые потом можно использовать в других директивах.
    ```apache
    # Установить внутреннюю переменную "block_bad_bot", если User-Agent совпадает
    SetEnvIfNoCase User-Agent "BadBot" block_bad_bot
    
    # Использовать эту переменную для запрета доступа
    <RequireAll>
        Require all granted
        Require not env block_bad_bot
    </RequireAll>
    ```

##### Сводная таблица синтаксиса:

| Тип переменной | Синтаксис | Где работает | Пример |
| :--- | :--- | :--- | :--- |
| **Окружение процесса** | `${VAR_NAME}` | Глобально в `.conf` файлах | `${APACHE_LOG_DIR}` |
| **Контекст сервера** | `${SERVER_NAME}` | Виртуальные хосты, логи | `${SERVER_PORT}` |
| **Данные запроса** | `%{VAR_NAME}` | `mod_rewrite`, `mod_log_config` | `%{REQUEST_URI}` |
| **Заголовки (вход)** | `%{Name}i` | Логи, условия | `%{User-Agent}i` |
| **Заголовки (выход)** | `%{Name}o` | Логи | `%{Content-Type}o` |


### Алгоритм настройки веб-сервера для раздачи статики:

##### Шаг 1. Создание директории для сайта:

```bash
mkdir -p /var/www/example.com/public_html
```

**Apache НЕ требует** структуры `/var/www/example.com/public_html`. Можно разместить файлы сайта **в любой папке**, к которой у Apache есть доступ. **Почему используется структуру `/var/www/example.com/public_html`?** Это конвенция (негласное правило), а не техническое требование.

| Часть пути     | Зачем это нужно?                                                                                                    | Можно ли изменить?                                                             |
| -------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `/var/www/`    | Стандартная директория для веб-контента в Linux (FHS — Filesystem Hierarchy Standard).                              | **Да.** Можно использовать `/srv/www/`, `/home/user/`, `/opt/sites/` и т.д.    |
| `example.com/` | Папка с именем домена. Помогает админу визуально понять, какой сайт где лежит, особенно если на сервере их десятки. | **Да.** Можно назвать папку `site1`, `myshop`, `abc123` — Apache всё равно.    |
| `public_html/` | Разделение: здесь лежат публичные файлы, а конфиги или приватные данные — на уровень выше.                          | **Да.** Часто используют просто корень сайта или папку `html`, `public`, `web` |

##### Шаг 2. Назначение прав доступа:

Apache работает от пользователя `www-data`. Нужно отдать ему права на папку:

```bash
chown -R www-data:www-data /var/www/example.com/public_html
chmod -R 755 /var/www/example.com
```

##### Шаг 3. Создание тестовой страницы:

```bash
nano /var/www/example.com/public_html/index.html
```

Вставьте туда простой HTML:

```html
<html>
    <head><title>Привет, Apache!</title></head>
    <body><h1>Сайт работает на Apache2</h1></body>
</html>
```

##### Шаг 4. Настройка виртуальных хостов:

Смотреть ниже в модуле: **Создание конфига виртуального хоста**.

##### Шаг 5. Включение сайта:

Apache не читает файлы из `sites-available` напрямую, нужно создать ссылку в `sites-enabled`:

```bash
a2ensite example.com.conf
```

##### Шаг 6. Отключение сайта по умолчанию (опционально):

```bash
a2dissite 000-default.conf
```

##### Шаг 7. Проверка конфигурации и перезагрузка:

```bash
apache2ctl configtest
```

Теперь по домену `example.com` (или IP, если домен еще не настроен в DNS) должен открываться ваш сайт.



### Создание конфига виртуального хоста:

```bash
nano /etc/apache2/sites-available/example.com.conf
```

| Требование                               | Реальность                                                                                                                       |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **Имя должно совпадать с доменом?**      | ❌ **Нет.** Файл можно назвать `mysite.conf`, `site123.conf`, `aaa.conf` — Apache всё равно.                                      |
| **Расширение должно быть `.conf`?**      | ⚠️ **Рекомендуется.** Apache по умолчанию читает только `*.conf` в некоторых директориях, но это настраиваемо.                   |
| **Папка должна быть `sites-available`?** | ⚠️ **Зависит от дистрибутива.** В Debian/Ubuntu — да, это стандарт. В CentOS/RHEL — конфиги кладут сразу в `/etc/httpd/conf.d/`. |

В **Debian/Ubuntu** используется двухуровневая система:

```Structure
/etc/apache2/
├── sites-available/     # Здесь хранятся все конфиги сайтов (как библиотека)
│   ├── example.com.conf
│   ├── test-site.conf
│   └── 000-default.conf
│
└── sites-enabled/       # Здесь только символьные ссылки на активные сайты
    ├── example.com.conf -> ../sites-available/example.com.conf
    └── 000-default.conf -> ../sites-available/000-default.conf
```

**Как это работает:**

1. Создается файл в `sites-available` (имя — любое, но лучше понятное);
2. Команда `sudo a2ensite example.com.conf` создает **символическую ссылку** в `sites-enabled`;
3. В главном конфиге `apache2.conf` есть строка:

```Apache
IncludeOptional sites-enabled/*.conf
# Она говорит Apache: "Загрузи все `.conf` файлы из папки `sites-enabled`".
```

```Apache
<VirtualHost *:80>
    # Основной домен сайта (обязательно для виртуальных хостов)
    ServerName example.com
    
    # Дополнительные домены (www, старые домены, поддомены)
    ServerAlias www.example.com blog.example.com
    
    # Email администратора (отображается на страницах ошибок 500)
    ServerAdmin webmaster@example.com
    
    # ============================================================
    # ПУТИ К КОНТЕНТУ
    # ============================================================
    # Корневая директория сайта (файлы, которые будут отдаваться)
    DocumentRoot /var/www/example.com/public_html
    
    # Файлы, которые искать по умолчанию при заходе в директорию
    DirectoryIndex index.php index.html index.htm
    
    # ============================================================
    # НАСТРОЙКИ ДОСТУПА К ДИРЕКТОРИИ
    # ============================================================
    <Directory /var/www/example.com/public_html>
        # Опции обработки контента:
        # -Indexes: запретить листинг файлов (если нет index)
        # +FollowSymLinks: разрешить символические ссылки (нужно для ЧПУ)
        Options -Indexes +FollowSymLinks
        
        # Разрешить .htaccess переопределять настройки
        # All = всё, None = игнорировать, FileInfo = только редиректы/заголовки
        AllowOverride All
        
        # Контроль доступа (синтаксис Apache 2.4+)
        Require all granted
        
        # Если нужно ограничить доступ по IP:
        # Require ip 192.168.1.0/24
        # Require not ip 10.0.0.5
    </Directory>
    
    # Запретить доступ к служебным файлам (дополнительная защита)
    <FilesMatch "^\.">
        Require all denied
    </FilesMatch>
    
    # ============================================================
    # ЛОГИРОВАНИЕ
    # ============================================================
    # Файл для ошибок этого сайта
    ErrorLog ${APACHE_LOG_DIR}/example.com_error.log
    
    # Файл для логов доступа
    CustomLog ${APACHE_LOG_DIR}/example.com_access.log combined
    
    # Уровень детализации ошибок: emerg|alert|crit|error|warn|notice|info|debug
    LogLevel warn
    
    # ============================================================
    # ПЕРЕАДРЕСАЦИИ И ЧПУ (mod_rewrite)
    # ============================================================
    <IfModule mod_rewrite.c>
        RewriteEngine On
        
        # Принудительный редирект на www (опционально)
        RewriteCond %{HTTP_HOST} ^example\.com [NC]
        RewriteRule ^(.*)$ https://www.example.com/$1 [L,R=301]
        
        # Стандартные правила для ЧПУ (например, для WordPress)
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteCond %{REQUEST_FILENAME} !-d
        RewriteRule . /index.php [L]
    </IfModule>
    
    # Простой редирект (без mod_rewrite)
    # Redirect 301 /old-page.html https://example.com/new-page.html
    
    # ============================================================
    # ПРОКСИРОВАНИЕ (если сайт работает на бэкенде)
    # ============================================================
    # <IfModule mod_proxy.c>
    #     ProxyPreserveHost On
    #     ProxyPass / http://127.0.0.1:3000/
    #     ProxyPassReverse / http://127.0.0.1:3000/
    # </IfModule>
    
    # ============================================================
    # КЭШИРОВАНИЕ И СЖАТИЕ (на уровне хоста)
    # ============================================================
    # <IfModule mod_expires.c>
    #     ExpiresActive On
    #     ExpiresByType image/jpg "access plus 1 month"
    #     ExpiresByType text/css "access plus 1 week"
    # </IfModule>
    
    # <IfModule mod_deflate.c>
    #     AddOutputFilterByType DEFLATE text/html text/css application/javascript
    # </IfModule>
    
    # ============================================================
    # БЕЗОПАСНОСТЬ (заголовки)
    # ============================================================
    # <IfModule mod_headers.c>
    #     Header always set X-Frame-Options "SAMEORIGIN"
    #     Header always set X-Content-Type-Options "nosniff"
    #     Header always set X-XSS-Protection "1; mode=block"
    # </IfModule>
    
</VirtualHost>
```

#### Идентификация виртуального хоста:

Эти директивы определяют, какой блок `<VirtualHost>` будет выбран для обработки входящего запроса (на основе заголовка `Host`).

* **ServerName** — Основной канонический домен, на который реагирует этот блок.
    *   **Статус:** Обязательная директива (хотя Apache может запуститься без неё, используя IP или предупреждение, для корректной работы виртуальных хостов она необходима).
    *   **Пример:** `ServerName example.com`
    *   **Нюанс:** Если указано несколько блоков с одинаковым `IP:Port`, Apache выбирает первый в файле конфигурации как "дефолтный", если ни один `ServerName` не совпал с заголовком запроса.

* **ServerAlias** — Дополнительные имена (алиасы) для этого хоста. Позволяет одному блоку конфигурации обслуживать несколько доменов.
    *   **Статус:** Необязательная
    *   **Примеры:**
        ```apache
        ServerAlias www.example.com
        # Или несколько через пробел:
        ServerAlias *.example.com shop.example.com
        ```
    *   **Применение:** Часто используется для перенаправления `www` на основной домен или объединения поддоменов.

* **ServerAdmin** — Контактный email администратора сервера.
    *   **Статус:** Необязательная, но рекомендуемая;
    *   **Пример:** `ServerAdmin admin@example.com`;
    *   **Где видно:** Отображается на стандартных страницах ошибок Apache (например, ошибка 404 или 500), если включена директива `ServerSignature On`. Полезно для пользователей, чтобы сообщить о проблемах.

* **Дополнительные важные директивы идентификации:**

    *   **UseCanonicalName** — Определяет, как Apache формирует самоссылки (например, при редиректе `/dir` в `/dir/`):
        *   `On`: Использует значение `ServerName` и порт;
        *   `Off`: Использует имя и порт, указанные клиентом в заголовке `Host` (полезно для массового хостинга);
        *   `DNS`: Пытается сделать обратный DNS-запрос к IP клиента (не рекомендуется из-за нагрузки на DNS).

    *   **ServerSignature** — Добавляет подпись (строку с версией Apache и `ServerAdmin`) в нижний колонтитул страниц ошибок и листингов директорий.
        *   `ServerSignature Off` — **Рекомендуется для безопасности**, чтобы не раскрывать версию ПО и контакты злоумышленникам;
        *   `ServerSignature On` — Включает подпись.

**Пример полной конфигурации:**

```Apache
<VirtualHost *:80>
  ServerName example.com
  ServerAlias www.example.com blog.example.com
  ServerAdmin webmaster@example.com
   
  # Безопасность: скрываем версию сервера в ошибках
  ServerSignature Off
    
  # Используем имя из заголовка Host для редиректов
  UseCanonicalName Off

  DocumentRoot /var/www/example.com
  # ... остальные настройки
</VirtualHost>
```
#### Работа с файлами и путями:

Эти директивы управляют тем, как Apache сопоставляет URL-адреса запросов с физической файловой системой сервера.

* **DocumentRoot** — Абсолютный путь к корневой директории веб-сайта.
    *   **Логика:** Все относительные пути в запросах (например, `/css/style.css`) добавляются к этому пути. Запрос `GET /` будет искать файл, указанный в `DirectoryIndex`, внутри этой папки.
    *   **Пример:** `DocumentRoot /var/www/example.com/public_html`
    *   **Важно:** У процесса Apache (пользователь `www-data`) должны быть права на чтение этой директории. Также рекомендуется явно разрешать доступ к ней в блоке `<Directory>`.

* **DirectoryIndex** — Приоритетный список имен файлов, которые сервер будет искать при запросе директории.
    *   **Логика:** Проверка идет строго слева направо. Сервер отдает первый найденный файл и прекращает поиск.
    *   **Пример:** `DirectoryIndex index.php index.html index.htm`
    *   **Сценарий:** Если пользователь заходит на `example.com/blog/`, Apache сначала проверит наличие `index.php`, затем `index.html`. Если ни одного нет — поведение зависит от настройки `Options Indexes` (либо ошибка 403, либо листинг файлов).

* **Alias** — Создает псевдоним для директории, позволяя отдавать контент из мест, не входящих в `DocumentRoot`.
    *   **Логика:** Прямое сопоставление префикса пути. Порядок директив важен: более специфичные алиасы должны идти выше общих.
    *   **Пример:**
        ```Apache
        # Запрос /images/logo.png -> файл /srv/shared_assets/images/logo.png
        Alias /images /srv/shared_assets/images
        
        # Важно: Для доступа к алиасу часто нужен отдельный блок <Directory>
        <Directory /srv/shared_assets/images>
            Require all granted
        </Directory>
        ```
    *   **Применение:** Раздача статических файлов (картинки, скрипты) с другого диска или разделение кода приложения и медиа-контента.

* **AliasMatch** — Аналог `Alias`, но поддерживает регулярные выражения для гибкого маппинга путей.
    *   **Логика:** Позволяет перехватывать сложные паттерны и перенаправлять их, используя группы захвата (`$1`, `$2`).
    *   **Пример:**
        ```Apache
        # Запрос /backup/db_2023.sql -> файл /mnt/backups/sql/db_2023.sql
        AliasMatch ^/backup/(.*)$ /mnt/backups/sql/$1
        ```
    *   **Нюанс:** Регулярные выражения требуют экранирования специальных символов, если они нужны как литералы.

*   **\<Directory\>** — Блок конфигурации, применяющий настройки (права доступа, опции) к конкретной физической директории на диске.
	*   **Критично:** Без явного разрешения в этом блоке доступ к `DocumentRoot` или `Alias` будет запрещен по умолчанию (ошибка 403 Forbidden).
	*   **Пример:**
		```Apache
		<Directory /var/www/example.com/public_html>
			Options -Indexes +FollowSymLinks
			AllowOverride All
			Require all granted
		</Directory>
		```

*   **\<Location\>** — Блок конфигурации, применяющий настройки к URL-пути (виртуальному пути), а не к физической папке.
	*   **Отличие от Directory:** Работает с тем, что видит пользователь в браузере. Полезно для ограничения доступа к определенным разделам сайта или проксирования.
	*   **Предостережение:** Не используйте `<Location>` для ограничения доступа к файлам на диске, так как один и тот же файл может быть доступен через разные пути (например, через алиас), и защита не сработает. Для файлов всегда используйте `<Directory>` или `<Files>`.
	* Пример защиты админ панели:

		```Apache
		<Location "/admin">
		    # Разрешить доступ только с рабочего офиса
		    Require ip 203.0.113.50
		    Require ip 192.168.1.0/24
		    
		    # Дополнительная аутентификация
		    AuthType Basic
		    AuthName "Admin Area"
		    AuthUserFile /etc/apache2/.htpasswd
		    Require valid-user
		</Location>
		```

*   **\<Files\> и \<FilesMatch\>** — Применяют настройки к конкретным именам файлов (или паттернам имен) независимо от их расположения в дереве каталогов.
	*   **Пример (Безопасность):** Запрет чтения файлов `.htaccess` или `.git`:
		```Apache
		<FilesMatch "^\.">
			Require all denied
		</FilesMatch>
		```

*   **Options** — Управляет функциями внутри директории.
	* `Indexes` - С директивой apache показывает список всех файлов в папке (как в проводнике), без директивы показывается ошибка **403 Forbidden**;
	* `FollowSymLinks` - указывает на другой файл/папку;
	* `ExecCGI` - запрещает выполнение CGI-скриптов (безопасность);
	* `Includes` - разрешает Server Side Includes (SSI) — вставка контента на уровне Apache;
	* `MultiViews` - Включает согласование контента (например, авто-выбор языка);


*   **AllowOverride** — Определяет, какие директивы можно переопределять в файлах `.htaccess` внутри этой директории.
	*   `None`: Игнорировать `.htaccess` (максимальная производительность).
	*   `All`: Разрешить всё (требуется для работы ЧПУ в WordPress/Laravel и т.д.).
	*   `FileInfo`, `AuthConfig`: Разрешить только конкретные группы директив.

> **Порядок обработки путей:**
> 1. Проверяются `Alias` и `AliasMatch` (до проверки `DocumentRoot`).
> 2. Если алиас не сработал, путь ищется относительно `DocumentRoot`.
> 3. Применяются правила из соответствующих блоков `<Directory>`, `<Location>` или файлы `.htaccess`.



* **Require** - (Apache 2.4) директива **авторизации**: кто имеет право получать доступ к ресурсу:
	* `Require all granted` - разрешить доступ **всем** (стандарт для публичных сайтов);
	* Запретить доступ **всем** (для блокировки)
	* `Require all denied` - запретить доступ **всем** (для блокировки);
	* `Require ip 192.168.1.10` - разрешить только с указанного IP;
	* `Require ip 192.168.1.0/24` - разрешить целой подсети;
	* `Require valid-user` - разрешить любому, кто прошел аутентификацию (логин/пароль)
	* `Require user admin dzhambulat` - разрешить только конкретным пользователям.


#### Логирование:

Логи — главный инструмент диагностики проблем и анализа трафика. В Apache они гибко настраиваются для каждого виртуального хоста или глобально.

* **ErrorLog** — Путь к файлу, куда записываются ошибки сервера, сбои скриптов и предупреждения.
    *   **Переменные:** Поддерживает переменные окружения (например, `${APACHE_LOG_DIR}` из `envvars`) и, в некоторых версиях/контекстах, подстановку имени хоста (`%v`).
    *   **Пример:** `ErrorLog ${APACHE_LOG_DIR}/example.com-error.log`
    *   **Совет:** Для продакшена рекомендуется иметь отдельный файл ошибок для каждого виртуального хоста, чтобы быстрее находить проблемы конкретного сайта.

* **CustomLog** — Путь к файлу логов доступа (кто, когда и что запрашивал).
    *   **Синтаксис:** `CustomLog <путь> <формат>`
    *   **Стандартные форматы:** Определяются директивой `LogFormat` в `apache2.conf`.
        *   `combined`: Самый популярный. Включает IP, время, запрос, статус, размер, реферер и User-Agent.
        *   `common`: Базовый набор без реферера и User-Agent.
    *   **Пример:** `CustomLog ${APACHE_LOG_DIR}/example.com-access.log combined`
    *   **Ротация:** В Debian/Ubuntu ротацию обычно настраивает утилита `logrotate` (конфиг `/etc/logrotate.d/apache2`).

* **LogLevel** — Контролирует детализацию записей в `ErrorLog`.
    *   **Уровни (от менее детального к более подробному):**
        *   `emerg`, `alert`, `crit` — Критические ошибки (сервер может не работать).
        *   `error` — Обычные ошибки (рекомендуемый минимум для продакшена).
        *   `warn` — Предупреждения.
        *   `notice`, `info` — Информационные сообщения.
        *   `debug`, `trace1`...`trace8` — Детальная отладка (замедляет сервер, использовать только при поиске багов).
    *   **Гибкость:** Можно задать уровень глобально или переопределить для конкретного модуля.
        *   Пример: `LogLevel warn ssl:info` (глобально только предупреждения, но от модуля SSL писать информативные сообщения).

* **LogFormat** — Позволяет создать собственный именованный формат лога или использовать строку формата напрямую в `CustomLog`.
    *   **Спецификаторы (ключевые переменные):**
        *   `%h` — IP-адрес клиента (Remote Host).
        *   `%l` — Имя пользователя удаленного пользователя (обычно `-`, если не используется identd).
        *   `%u` — Имя пользователя (если использовалась HTTP-аутентификация).
        *   `%t` — Время и дата запроса в формате Common Log Format.
        *   `%r` — Первая строка запроса (метод, путь, протокол), например, `"GET /index.html HTTP/1.1"`.
        *   `%>s` — Код статуса ответа (после выполнения запроса). `%s` — код до выполнения (может быть неточным для редиректов).
        *   `%b` — Размер ответа в байтах (без заголовков). `%B` — то же самое, но `0` заменяется на `-`.
        *   `%O` — Количество отправленных байт (включая заголовки).
        *   `%{VAR}i` — Содержимое входного заголовка (например, `%{User-Agent}i`, `%{Referer}i`).
        *   `%{VAR}o` — Содержимое выходного заголовка.
        *   `%{VAR}c` — Содержимое куки.
        *   `%D` — Время обработки запроса в микросекундах (полезно для поиска медленных запросов).
        *   `%T` — Время обработки запроса в секундах.
        *   `%v` — Каноническое имя сервера (`ServerName`).
        *   `%p` — Порт сервера.
    *   **Пример кастомного формата для анализа производительности:**
        ```Apache
        # Формат: Время, Хост, Метод, Путь, Статус, Время_ответа(мкс), User-Agent
        LogFormat "%t %h \"%r\" %>s %D \"%{User-Agent}i\"" perf_debug
        CustomLog ${APACHE_LOG_DIR}/perf.log perf_debug
        ```

* **Дополнительные возможности логирования:**

    *   **Кондиционное логирование (Conditional Logging).** Позволяет записывать в лог только определенные запросы с помощью директивы `SetEnvIf` и оператора `env=` в `CustomLog`. *Сценарий:* Не логировать запросы на картинки или проверки работоспособности (health checks):
		```Apache
		# Не логировать запросы к файлам .png, .jpg, .gif
		SetEnvIf Request_URI "\.(gif|jpg|png|css|js)$" dontlog
		
		# Записать в лог только то, что НЕ помечено флагом dontlog
		CustomLog ${APACHE_LOG_DIR}/access.log combined env=!dontlog
		```

    *   **Логирование в пайп (Pipe):**
        Можно передавать логи не в файл, а во внешнюю программу для немедленной обработки (например, `rotatelogs` для ротации по времени или `cronolog`).
        *   Пример: `CustomLog "|/usr/bin/rotatelogs /var/log/apache2/access.%Y-%m-%d.log 86400" combined`

    *   **Виртуальные хосты:**
        Рекомендуется переопределять `ErrorLog` и `CustomLog` внутри каждого блока `<VirtualHost>`, чтобы изолировать логи разных сайтов. Если этого не сделать, все сайты будут писать в один общий файл (`access.log`), что усложнит анализ.

> **Практический совет по безопасности:**
> Убедитесь, что файлы логов доступны для записи только пользователю, от которого запущен Apache (`www-data`), и читаемы только администратором. Не храните логи в публично доступных директориях (`DocumentRoot`), чтобы злоумышленники не могли скачать их и проанализировать структуру вашего сайта или попытки взлома.

#### Перенаправления и ЧПУ:

Модули `mod_rewrite` и `mod_alias` отвечают за управление URL: изменение адресов, скрытие реальной структуры файлов и создание «красивых» ссылок (ЧПУ).

**1. Модуль `mod_alias` (Простые перенаправления)**
Идеален для базовых задач: переезд страниц, смена доменов или маппинг путей. Работает быстрее `mod_rewrite`, так как выполняется на ранней стадии обработки запроса.

*   **Включение:**
    ```bash
    a2enmod alias
    # Для RedirectMatch также нужен mod_alias (он включен по умолчанию в большинстве сборок)
    systemctl restart apache2
    ```

*   **Директива `Redirect`:** Выполняет простой маппинг префикса пути.
    ```apache
    # Синтаксис: Redirect [код] [старый-путь] [новый-URL]
    
    # Временный редирект (302) — браузер запомнит старый URL
    Redirect 302 /old-page.html https://example.com/new-page.html  
    
    # Постоянный редирект (301) — сообщает поисковикам о переезде (лучше для SEO)
    Redirect 301 /legacy https://example.com/modern  
    
    # Редирект всей папки (все, что начинается с /docs/, уйдет на новый домен)
    Redirect /docs/ https://newsite.com/documentation/
    ```
    > **Важно:** `Redirect` автоматически добавляет остаток пути. Если запрос `/docs/file.pdf`, он уйдет на `https://newsite.com/documentation/file.pdf`.

*   **Директива `RedirectMatch`:** Позволяет использовать регулярные выражения для гибкого управления.
    ```apache
    # Редирект всех .html файлов на .php (сохраняя имя файла)
    # $1 ссылается на первую группу захвата (.*)
    RedirectMatch 301 ^(.*)\.html$ $1.php

    # Редирект старых записей блога (2020/...) на новую структуру
    RedirectMatch 301 ^/blog/2020/(.*)$ /articles/$1

    # Редирект с удалением префикса (чистка URL от версии API)
    RedirectMatch 301 ^/site/v1/(.*)$ /$1
    
    # Принудительное закрытие доступа к пути (возврат 404 или 410)
    RedirectMatch 410 ^/temp-secret-folder/.*$
    ```

**2. Модуль `mod_rewrite` (Мощный движок правил)**
Используется для сложных сценариев: условия по заголовкам, переменным окружения, существованию файлов и создания полностью виртуальных URL (ЧПУ).

*   **Включение:**
    ```bash
    a2enmod rewrite
    systemctl restart apache2
    ```

*   **Базовые директивы:**
    *   `RewriteEngine On` — активирует движок переписывания в текущем контексте.
    *   `RewriteBase /` — задает базовый путь (важно при использовании в `.htaccess` внутри подпапок).
    *   `RewriteCond` — задает условие (проверка), которое должно выполниться перед правилом.
    *   `RewriteRule` — само правило замены.

*   **Синтаксис правила:**
    `RewriteRule ^Паттерн$ Подстановка [Флаги]`

*   **Примеры использования:**

    *   **ЧПУ (Friendly URLs) для CMS:**
        Превращает запрос `/article/my-first-post` в внутренний вызов `index.php?slug=my-first-post`.
        ```apache
        RewriteEngine On
        # Если файл не существует и директория не существует -> перенаправить на index.php
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteCond %{REQUEST_FILENAME} !-d
        RewriteRule ^(.*)$ index.php?q=$1 [L,QSA]
        ```
        *Флаги:* `L` (Last — остановить обработку правил), `QSA` (Query String Append — сохранить старые GET-параметры).

    *   **Принудительный HTTPS:**
        ```apache
        RewriteEngine On
        # Если соединение НЕ защищено (HTTPS != on)
        RewriteCond %{HTTPS} off
        # Перенаправить на тот же хост и URI, но с https://
        RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
        ```

    *   **Удаление `www` (или добавление):**
        ```apache
        RewriteEngine On
        # Если хост начинается с www.
        RewriteCond %{HTTP_HOST} ^www\.(.*)$ [NC]
        # Перенаправить на домен без www
        RewriteRule ^(.*)$ https://%1/$1 [R=301,L]
        ```

    *   **Блокировка по User-Agent (защита от ботов):**
        ```apache
        RewriteEngine On
        RewriteCond %{HTTP_USER_AGENT} (badbot|scraper) [NC]
        RewriteRule ^ - [F]
        ```
        *Флаг:* `F` (Forbidden — вернуть ошибку 403). `-` в подстановке означает "не менять URL".

*   **Где размещать правила:**
    1.  **В конфиге виртуального хоста (`/etc/apache2/sites-available/...`):**
        *   Внутри блока `<VirtualHost>` или `<Directory>`.
        *   **Плюс:** Высокая производительность (не нужно проверять наличие `.htaccess` при каждом запросе).
        *   **Требование:** `AllowOverride None` (или конкретные права) в соответствующем `<Directory>`.
    2.  **В файле `.htaccess` (в корне сайта):**
        *   **Плюс:** Не требует перезагрузки Apache после изменений; удобно на shared-хостинге.
        *   **Минус:** Снижает производительность (сервер ищет файл в каждой папке пути).
        *   **Требование:** `AllowOverride FileInfo` или `AllowOverride All` в конфиге сервера.

> **Сравнение `Redirect` vs `RewriteRule`:**
> *   Используйте `Redirect` (из `mod_alias`), если задача проста: "переместить страницу А в место Б". Это быстрее и проще читается.
> *   Используйте `RewriteRule` (из `mod_rewrite`), если нужны условия ("если файл не существует", "если мобильный пользователь") или сложная трансформация URL (ЧПУ).
> *   **Конфликт:** Не смешивайте их для одного и того же пути без понимания приоритетов. `mod_alias` выполняется *до* `mod_rewrite` на этапе трансляции URI, что иногда приводит к неожиданным результатам (например, `Redirect` может сработать раньше, чем вы успеете применить правило `Rewrite`). Если используете оба, лучше делать все через `RewriteRule`.

#### Проксирование:

Проксирование говорит Apache: _"Получи этот запрос, сходи на другой сервер, забери ответ и отдай клиенту от своего имени"._

**Включение необходимых модулей:**

```bash
a2enmod proxy proxy_http proxy_balancer proxy_wstunnel headers
```

| Модуль               | Зачем нужен                                        |
| -------------------- | -------------------------------------------------- |
| `mod_proxy`          | Базовый движок проксирования.                      |
| `mod_proxy_http`     | Проксирование HTTP/HTTPS запросов.                 |
| `mod_proxy_wstunnel` | Проксирование WebSocket (для чатов, игр).          |
| `mod_proxy_balancer` | Балансировка нагрузки между несколькими серверами. |
| `mod_headers`        | Корректировка заголовков (важно для прокси).       |
| `mod_ssl`            | Проксирование по HTTPS.                            |

##### Базовая настройка прокси:

```bash
<VirtualHost *:80>
    ServerName api.example.com
    
    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:3000/
    ProxyPassReverse / http://127.0.0.1:3000/
</VirtualHost>
```

| Директива             | Описание                                                                                         |
| --------------------- | ------------------------------------------------------------------------------------------------ |
| **ProxyPass**         | Перенаправляет запросы на бэкенд (Node.js, Python).                                              |
| **ProxyPassReverse**  | Переписывает заголовки `Location` и `Redirect` от бэкенда, чтобы клиент не увидел внутренний IP. |
| **ProxyPreserveHost** | Передает оригинальный заголовок `Host` на бэкенд. Нужно, если на бэкенде несколько сайтов.       |

##### Проксирование конкретных путей:

```vim
<VirtualHost *:80>
    ServerName example.com
    DocumentRoot /var/www/example.com/public_html
    
    # Статика обрабатывается Apache
    # ... настройки кэша ...
    
    # API уходит на Node.js бэкенд
    ProxyPass /api http://127.0.0.1:3000/
    ProxyPassReverse /api http://127.0.0.1:3000/
    
    # Админка на Python
    ProxyPass /admin http://127.0.0.1:8000/
    ProxyPassReverse /admin http://127.0.0.1:8000/
</VirtualHost>
```

##### Исключения из проксирования:

```vim
# Проксировать всё...
ProxyPass / http://127.0.0.1:3000/

# ...кроме статики и логов
ProxyPass /static/ !
ProxyPass /images/ !
ProxyPass /logs/ !
```

Правила `ProxyPass` обрабатываются **по порядку**. Исключения (`!`) должны идти после общего правила.

##### Проксирование WebSocket:

```vim
ProxyPass /ws ws://127.0.0.1:3000/ws
ProxyPassReverse /ws ws://127.0.0.1:3000/ws

# Для HTTPS:
ProxyPass /wss wss://127.0.0.1:3000/ws
```

##### Настройка таймаутов и буферов:

```vim
# Глобально или в VirtualHost
ProxyTimeout 60
ProxyBadHeader Ignore

# Для конкретного ProxyPass
ProxyPass /api http://127.0.0.1:3000/ timeout=120 connectiontimeout=5
```

| Параметр            | Описание                                                    |
| ------------------- | ----------------------------------------------------------- |
| `timeout`           | Максимальное время ожидания ответа от бэкенда.              |
| `connectiontimeout` | Время на установление соединения.                           |
| `retry`             | Время ожидания перед повторной попыткой (если бэкенд упал). |

##### Запрет проксирования произвольных URL:

Открытый прокси — это дыра в безопасности. `ProxyRequests Off` должен быть выключен, если вы не настраиваете явный forward-proxy.

```vim
# В глобальном конфиге (apache2.conf)
<IfModule mod_proxy.c>
    ProxyRequests Off  # Запрещает использование сервера как открытого прокси
</IfModule>
```

```vim
<Location "/proxy-status">

    Require ip 192.168.1.0/24  # Только из локальной сети
    Require ip 10.0.0.5        # Или с конкретного IP админа
</Location>
```

| Директива             | Пример                     | Описание                                                                                               |
| --------------------- | -------------------------- | ------------------------------------------------------------------------------------------------------ |
| **ProxyPass**         | `/ http://127.0.0.1:3000/` | Перенаправляет все запросы этого хоста на бэкенд (Node.js, Python, Docker).                            |
| **ProxyPassReverse**  | `/ http://127.0.0.1:3000/` | Корректирует заголовки `Location` от бэкенда, чтобы клиент не увидел внутренний адрес.                 |
| **ProxyPreserveHost** | `On`                       | Передает оригинальный заголовок `Host` от клиента на бэкенд (важно для виртуальных хостов на бэкенде). |
| **ProxyTimeout**      | `60`                       | Таймаут для прокси-соединений (сек).                                                                   |

#### Производительность: кэш и сжатие:

Для работы с сжатием включаем модуль:

```bash
a2enmod deflate
```

* **AddOutputFilterByType** - Включает gzip-сжатие для указанных MIME-типов перед отправкой клиенту. Для работы нужно включить модуль `deflate`;

* **DeflateCompressionLevel** - Настройка уровня сжатия (1-9, где 9 = максимум, но больше грузит CPU). Для работы нужно включить модуль `deflate`;

**Пример сжатия:**

```vim
<IfModule mod_deflate.c>
    # Сжимать только текстовые типы (картинки/видео уже сжаты)
    AddOutputFilterByType DEFLATE text/html text/plain text/xml text/css
    AddOutputFilterByType DEFLATE application/javascript application/json application/xml
    
    # Исключения: не сжимать уже сжатые файлы
    SetEnvIfNoCase Request_URI \.(?:gif|jpe?g|png|zip|gz|mp4)$ no-gzip dont-vary
    
    DeflateCompressionLevel 6
    
    # Добавлять заголовок Vary: Accept-Encoding (для корректной работы прокси/кэшей)
    Header append Vary Accept-Encoding
</IfModule>
```

Для работы с кешем включаем модуль:

```bash
a2enmod expires
```

* **ExpiresByType** - Указывает браузеру кэшировать файлы определенного типа указанный срок. Для работы нужно включить модуль `expires`;

* **ExpiresActive** - Включает модуль `mod_expires` для управления заголовками кэширования. Для работы нужно включить модуль `expires`;

**Пример кеширования:**

```vim
<IfModule mod_expires.c>
    ExpiresActive On
    
    # Изображения — кэш на 1 месяц
    ExpiresByType image/jpg "access plus 1 month"
    ExpiresByType image/jpeg "access plus 1 month"
    ExpiresByType image/png "access plus 1 month"
    ExpiresByType image/gif "access plus 1 month"
    ExpiresByType image/webp "access plus 1 month"
    ExpiresByType image/svg+xml "access plus 1 week"
    
    # CSS и JS — 1 неделя (чтобы быстро обновлять при изменениях)
    ExpiresByType text/css "access plus 1 week"
    ExpiresByType application/javascript "access plus 1 week"
    
    # Шрифты — 1 год (они редко меняются)
    ExpiresByType font/woff2 "access plus 1 year"
    ExpiresByType application/font-woff2 "access plus 1 year"
    
    # HTML — не кэшировать долго (чтобы видеть обновления контента)
    ExpiresByType text/html "access plus 0 seconds"
</IfModule>
```

**Пример кеширования через `mod_headers`:**

```vim
<IfModule mod_headers.c>
    # Статика: кэш на год + immutable (браузер не будет проверять актуальность)
    <FilesMatch "\.(jpg|jpeg|png|gif|webp|svg|woff2|ttf|eot)$">
        Header set Cache-Control "public, max-age=31536000, immutable"
    </FilesMatch>
    
    # CSS/JS: кэш на неделю, но без immutable (чтобы можно было обновить)
    <FilesMatch "\.(css|js)$">
        Header set Cache-Control "public, max-age=604800, must-revalidate"
    </FilesMatch>
    
    # HTML: не кэшировать
    <FilesMatch "\.(html|php)$">
        Header set Cache-Control "no-cache, no-store, must-revalidate"
    </FilesMatch>
</IfModule>
```

Директива `immutable` — это подсказка браузеру: **«Этот файл гарантированно не изменится в течение срока `max-age`»**.

| Ситуация                                               | Без `immutable`                                                                                   | С `immutable`                                           |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| Пользователь зашел на сайт впервые                     | Файл скачивается и кэшируется                                                                     | Файл скачивается и кэшируется                           |
| Пользователь нажал **обновить страницу (F5)**          | Браузер отправляет запрос с `If-None-Match` для проверки актуальности (статус `304 Not Modified`) | Браузер **не делает запрос вообще**, берет файл из кэша |
| Пользователь зашел через неделю (в пределах `max-age`) | Тот же запрос проверки                                                                            | Всё ещё берет из кэша без запроса                       |
| `max-age` истек                                        | Скачивает заново                                                                                  | Скачивает заново                                        |

Используйте `immutable` **ТОЛЬКО** для файлов с **хэшем в имени** (fingerprints):

```Text
✅ style.abc123.css  (при изменении стиля имя файла изменится на style.def456.css)
✅ app.v2.js
✅ logo.20240318.png

❌ style.css  (если вы обновите файл, пользователи не увидят изменений целый год!)
❌ common.js
```

Почему мы не кэшируем HTML, но кэшируем CSS/JS?

| Если кэшировать HTML агрессивно                              | Если НЕ кэшировать HTML                                             |
| ------------------------------------------------------------ | ------------------------------------------------------------------- |
| ❌ Вы обновили `style.abc123.css` на `style.ghi789.css`       | ✅ Пользователь всегда получает свежий `index.html`                  |
| ❌ Пользователь видит старый HTML со ссылками на старые файлы | ✅ В свежем HTML — ссылки на новые версии ассетов                    |
| ❌ Пользователь не увидит обновления контента/дизайна         | ✅ Браузер сам решит, кэшировать ли уже скачанные `style.abc123.css` |
HTML файлы являются точками входа в веб-сайт

#### Аутентификация и авторизация:

Apache предоставляет гибкую систему контроля доступа (Access Control), основанную на модулях `mod_auth*`. Процесс делится на два этапа:
1.  **Аутентификация (Authentication):** «Кто вы?» (Проверка логина/пароля, токена, сертификата).
2.  **Авторизация (Authorization):** «Что вам разрешено?» (Проверка прав доступа к ресурсу).

*   **AuthType** — Тип механизма аутентификации.
    *   `Basic`: Пароль передается в кодировке Base64 (фактически открытый текст без HTTPS). Самый совместимый метод.
    *   `Digest`: Пароль передается в виде хэша (MD5). Безопаснее по сети, но сложнее в настройке и менее поддерживаем современными браузерами.
    *   `Form`: (Требует сторонних модулей или прокси) Перенаправление на страницу входа.
    *   `None`: Отключает аутентификацию.

*   **AuthName** — «Realm» (Область). Текст, который браузер покажет в заголовке окна ввода пароля.
    *   **Важно:** Для `Digest` аутентификации это значение участвует в генерации хэша. Если изменить `AuthName` после создания файла паролей, старые пароли перестанут работать.

*   **AuthUserFile** — Абсолютный путь к файлу, содержащему список пользователей и хэши паролей. Файл должен быть недоступен для чтения из веба (лежать вне `DocumentRoot`).

*   **AuthGroupFile** — (Опционально) Путь к файлу групп. Позволяет управлять доступом по группам, а не по отдельным пользователям.

*   **Require** — Директива авторизации. Определяет, кто имеет право доступа после успешной аутентификации.
    *   `Require user admin bob`: Доступ только конкретным пользователям.
    *   `Require group admins`: Доступ членам группы (требуется `AuthGroupFile`).
    *   `Require valid-user`: Доступ любому успешно аутентифицированному пользователю.
    *   `Require ip 192.168.1.0/24`: Доступ только с определенных IP (без пароля).
    *   **Логические операторы:** Можно комбинировать условия через `<RequireAll>`, `<RequireAny>`, `<RequireNone>`.

##### Пример `Basic` защиты (Базовая):

Самый распространенный метод. **Обязательно используйте только поверх HTTPS**, иначе пароли будут перехвачены.

**1. Включение модулей:**
```bash
a2enmod auth_basic authn_file authz_user authz_groupfile
systemctl restart apache2
```

**2. Создание файла с паролями (`htpasswd`):**
Файл лучше хранить в `/etc/apache2/` или другой скрытой директории.

```bash
# Создание нового файла и первого пользователя (-c создает файл, перезаписывая старый!)
htpasswd -c /etc/apache2/.htpasswd admin

# Добавление второго пользователя (без -c, иначе удалите первого!)
htpasswd /etc/apache2/.htpasswd manager

# Использование более стойкого алгоритма шифрования (bcrypt) вместо default (apr1)
htpasswd -B /etc/apache2/.htpasswd developer
```

**3. Конфигурация виртуального хоста или `.htaccess`:**
```apache
<Directory "/var/www/site/secure_area">
    AuthType Basic
    AuthName "🔐 Admin Panel - Login Required"
    AuthUserFile /etc/apache2/.htpasswd
    
    # Разрешить только пользователю admin ИЛИ manager
    Require user admin manager
    
    # ИЛИ разрешить всем из файла:
    # Require valid-user
</Directory>
```

**Управление флагоми `htpasswd`:**

| Аргумент | Описание |
| :--- | :--- |
| `-c` | **Create**: Создать новый файл. **Внимание:** Перезапишет существующий файл! Используйте только для первого пользователя. |
| `-b` | **Batch**: Передать пароль в командной строке (`htpasswd -b ... user pass`). Небезопасно для истории bash. |
| `-n` | **No file**: Вывести результат в stdout (удобно для скриптов и Docker). |
| `-m` | Использовать алгоритм MD5 (устаревший, но совместимый). |
| `-B` | Использовать алгоритм **bcrypt** (рекомендуется для безопасности). |
| `-D` | Удалить пользователя из файла. |

##### Пример `Digest` защиты (Дайджест):

Пароль не передается в открытом виде, используется挑战-ответ (challenge-response). Менее популярно из-за проблем с совместимостью и сложностью управления хэшами.

**1. Включение модулей:**
```bash
a2enmod auth_digest authn_file authz_user
systemctl restart apache2
```

**2. Создание файла (`htdigest`):**
Синтаксис отличается от `htpasswd`: требуется указывать Realm (должен совпадать с `AuthName`).

```bash
# Синтаксис: htdigest -c <файл> <realm> <user>
htdigest -c /etc/apache2/.htdigest "My Secure Realm" admin
```

**3. Конфигурация:**
```apache
<Directory "/var/www/site/digest_area">
    AuthType Digest
    AuthName "My Secure Realm" 
    AuthDigestProvider file
    AuthUserFile /etc/apache2/.htdigest
    Require valid-user
</Directory>
```
> **Критично:** Если вы измените `AuthName` ("My Secure Realm") в конфиге, вам придется пересоздать файл `.htdigest` с новым именем области, иначе аутентификация не пройдет.

##### Пример аутентификации через LDAP (Корпоративная среда):

Позволяет использовать существующие учетные записи Active Directory или OpenLDAP без создания локальных файлов паролей.

**1. Включение модулей:**
```bash
a2enmod authnz_ldap auth_basic
systemctl restart apache2
```

**2. Конфигурация:**
```apache
<Directory "/var/www/internal">
    AuthType Basic
    AuthName "Company Intranet"
    
    # Указываем, что провайдер аутентификации - LDAP
    AuthBasicProvider ldap
    
    # URL подключения:
    # ldap://server:port/base_dn?attribute?scope?filter
    # uid - атрибут, по которому ищем пользователя (логин)
    AuthLDAPURL "ldap://dc1.company.local:389/ou=users,dc=company,dc=com?uid?sub?(objectClass=person)"
    
    # Данные для привязки (Service Account), если анонимный поиск запрещен
    AuthLDAPBindDN "cn=apache_svc,ou=services,dc=company,dc=com"
    AuthLDAPBindPassword "SuperSecretServicePassword"
    
    # Опционально: требовать членство в конкретной группе LDAP
    # AuthLDAPGroupAttribute memberUid
    # AuthLDAPGroupAttributeIsDN on
    # Require ldap-group cn=WebAdmins,ou=groups,dc=company,dc=com

    Require valid-user
</Directory>
```

##### Продвинутые сценарии авторизации:

**1. Доступ по группам (файловый метод):**
Создайте файл групп (например, `/etc/apache2/.htgroups`):
```text
admins: admin boss
editors: john jane
```
Конфиг Apache:
```apache
AuthGroupFile /etc/apache2/.htgroups
Require group admins
```

**2. Комбинированные правила (Логика):**
Разрешить доступ либо администраторам, либо с внутреннего IP, но запретить всем остальным.

```apache
<Directory "/var/www/sensitive">
    AuthType Basic
    AuthName "Restricted"
    AuthUserFile /etc/apache2/.htpasswd
    
    <RequireAny>
        # Вариант 1: Пользователь из группы admins
        <RequireAll>
            Require user admin
            # Дополнительные проверки можно добавить здесь
        </RequireAll>
        
        # Вариант 2: Доступ с внутренней сети без пароля
        Require ip 10.0.0.0/8 192.168.1.0/24
    </RequireAny>
</Directory>
```

**3. Защита конкретных файлов (например, `.git`, `.env`):**
Часто требуется не спрашивать пароль, а просто запрещать доступ.
```apache
<FilesMatch "^\.">
    Require all denied
</FilesMatch>

# Или конкретно для .env
<Files ".env">
    Require all denied
</Files>
```

> **Совет по безопасности:**
> При использовании `Basic` аутентификации всегда настраивайте редирект на HTTPS до блока авторизации, иначе браузер отправит логин и пароль в открытом виде.
> ```apache
> RewriteEngine On
> RewriteCond %{HTTPS} off
> RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
> ```


#### Настройка SSL/TLS в Apache (mod_ssl)

Безопасное соединение (HTTPS) обеспечивается модулем `mod_ssl`. В Debian/Ubuntu он устанавливается отдельно и требует наличия сертификата и приватного ключа.

##### 1. Установка и активация модуля

```bash
# Установка пакета (если не установлен)
sudo apt-get install libapache2-mod-ssl

# Включение модуля
sudo a2enmod ssl

# Включение стандартного конфига для безопасных настроек (рекомендуется)
sudo a2enconf ssl-params

# Перезагрузка сервиса
sudo systemctl restart apache2
```

##### 2. Создание сертификатов

Существует три основных способа получения сертификатов:

**А. Самоподписанный сертификат (Для тестов и внутренней сети)**
Браузеры будут показывать предупреждение о безопасности, но шифрование будет работать.

```bash
# Генерация ключа и сертификата одним файлом (срок действия 365 дней)
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/apache-selfsigned.key \
  -out /etc/ssl/certs/apache-selfsigned.crt

# Важно: Ключ должен быть доступен только root
sudo chmod 600 /etc/ssl/private/apache-selfsigned.key
```

**Б. Let's Encrypt (Бесплатный, доверенный браузерами)**
Используется утилита `certbot`. Автоматически обновляет сертификаты и настраивает Apache.

```bash
# Установка certbot и плагина для Apache
sudo apt-get install certbot python3-certbot-apache

# Получение сертификата (автоматическая настройка vhost)
sudo certbot --apache -d example.com -d www.example.com

# Проверка автообновления (сухой прогон)
sudo certbot renew --dry-run
```
*Certbot сам создаст файлы в `/etc/letsencrypt/live/example.com/` и пропишет пути в конфиге.*

**В. Покупной сертификат (Commercial CA)**
Вы генерируете CSR (Certificate Signing Request), отправляете его в центр сертификации (Comodo, GeoTrust и др.), получаете файл сертификата и цепочку промежуточных сертификатов (CA Bundle).

```bash
# Генерация приватного ключа и запроса CSR
sudo openssl req -new -newkey rsa:2048 -nodes \
  -keyout /etc/ssl/private/example.com.key \
  -out /etc/ssl/certs/example.com.csr
```
*(Далее содержимое `.csr` отправляется в ЦС)*

##### 3. Конфигурация VirtualHost для HTTPS

Обычно конфиг располагается в `/etc/apache2/sites-available/example.com-le-ssl.conf` (если использовался Certbot) или создается вручную.

```apache
<VirtualHost *:443>
    ServerName example.com
    ServerAdmin admin@example.com
    DocumentRoot /var/www/example.com/public_html

    # --- Основные директивы SSL ---
    SSLEngine on
    
    # Путь к сертификату сайта
    SSLCertificateFile      /etc/letsencrypt/live/example.com/fullchain.pem
    
    # Путь к приватному ключу
    SSLCertificateKeyFile   /etc/letsencrypt/live/example.com/privkey.pem
    
    # (Опционально) Путь к цепочке промежуточных сертификатов, если не используется fullchain
    # SSLCertificateChainFile /etc/ssl/certs/ca-bundle.crt

    # --- Безопасность и производительность ---
    # Принудительное использование современных протоколов (TLS 1.2 и 1.3)
    SSLProtocol             all -SSLv3 -TLSv1 -TLSv1.1
    
    # Приоритет шифров сервера над клиентом
    SSLCipherSuite          ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384
    SSLHonorCipherOrder     on
    
    # Сжатие (CRIME attack mitigation) - отключить сжатие для SSL
    SSLCompression          off
    
    # Session Cache (ускорение повторных подключений)
    SSLSessionCache         shmcb:/var/run/apache2/ssl_scache(512000)
    SSLSessionTimeout       300

    # HSTS (HTTP Strict Transport Security) - заставить браузер всегда использовать HTTPS
    <IfModule mod_headers.c>
        Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"
    </IfModule>

    # Логи
    ErrorLog ${APACHE_LOG_DIR}/error.log
    CustomLog ${APACHE_LOG_DIR}/access.log combined
</VirtualHost>
```

##### 4. Специальные директивы SSL (Подробно)

| Директива | Описание | Рекомендация |
| :--- | :--- | :--- |
| **SSLEngine** | Включает или выключает SSL/TLS для данного виртуального хоста. | `on` (обязательно для порта 443). |
| **SSLCertificateFile** | Путь к файлу сертификата (PEM формат). Если используете Let's Encrypt, указывайте `fullchain.pem` (сертификат + цепочка). | Абсолютный путь. |
| **SSLCertificateKeyFile** | Путь к файлу приватного ключа. | Должен быть защищен (chmod 600, owner root). |
| **SSLCertificateChainFile** | Путь к файлу промежуточных сертификатов (CA Bundle). | Не нужен, если в `SSLCertificateFile` указан `fullchain.pem`. |
| **SSLProtocol** | Разрешенные версии протокола. | `all -SSLv3 -TLSv1 -TLSv1.1` (оставляет только TLS 1.2 и 1.3). |
| **SSLCipherSuite** | Список разрешенных алгоритмов шифрования. | Используйте строки из [Mozilla SSL Config Generator](https://ssl-config.mozilla.org/). Избегайте слабых шифров (RC4, DES, 3DES). |
| **SSLHonorCipherOrder** | Заставляет сервер выбирать шифр из своего списка приоритетов, а не клиента. | `on` (для контроля безопасности). |
| **SSLCompression** | Включает/выключает сжатие данных перед шифрованием. | `off` (защита от атаки CRIME). Современный HTTP/2 обрабатывает сжатие иначе. |
| **SSLSessionCache** | Механизм кэширования сессий для ускорения повторных рукопожатий (Resume). | `shmcb:...` (разделяемая память) — самый быстрый вариант. |
| **SSLStaplingCache** | Кэш для OCSP Stapling (проверка отзыва сертификата без обращения клиента к ЦС). | Рекомендуется настроить для повышения скорости и приватности. |

##### 5. Принудительный редирект с HTTP на HTTPS

Лучшая практика — не обслуживать контент по порту 80, а сразу перенаправлять на 443.

**Вариант А: В отдельном VirtualHost *:80**
```apache
<VirtualHost *:80>
    ServerName example.com
    ServerAlias www.example.com

    # Перенаправление всего трафика на HTTPS
    RewriteEngine On
    RewriteCond %{HTTPS} off
    RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
    
    # Или проще через Redirect (если не нужны сложные условия):
    # Redirect permanent / https://example.com/
</VirtualHost>
```

**Вариант Б: Через .htaccess (если нет доступа к основному конфигу)**
```apache
RewriteEngine On
RewriteCond %{HTTPS} !=on
RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
```

##### 6. Продвинутые настройки безопасности

**OCSP Stapling**
Позволяет серверу самому запрашивать статус отзыва сертификата у центра сертификации и передавать его клиенту во время handshake. Это ускоряет соединение и скрывает IP клиента от ЦС.

```apache
# В глобальном конфиге или внутри VirtualHost
SSLUseStapling on
SSLStaplingCache "shmcb:logs/ssl_stapling(32768)"
```
*Требует, чтобы в сертификате был указан URL OCSP responder (Let's Encrypt поддерживает).*

**HTTP/2**
Современный протокол, работающий поверх TLS. Значительно ускоряет загрузку сайтов за счет мультиплексирования.

```bash
# Включение модуля
sudo a2enmod http2
```
Добавьте в глобальный конфиг или VirtualHost:
```apache
Protocols h2 http/1.1
# Или для конкретного хоста:
# <VirtualHost *:443>
#     Protocols h2 http/1.1
# </VirtualHost>
```

**Проверка конфигурации**
После настройки обязательно проверьте безопасность через онлайн-сервисы:
1.  **SSL Labs (Qualys):** [ssllabs.com/ssltest](https://www.ssllabs.com/ssltest/) — дает оценку от A+ до F.
2.  **Командная строка:**
    ```bash
    # Проверка поддерживаемых протоколов и шифров
    openssl s_client -connect example.com:443 -tls1_2
    openssl s_client -connect example.com:443 -tls1_3
    ```

### Утилиты и команды для администрирования Apache

В Debian/Ubuntu управление Apache разделено между системным менеджером (`systemctl`) и встроенными утилитами самого Apache (`apache2ctl`, `a2*`).

##### 1. Управление сервисом (Systemd)
Базовые команды для контроля состояния службы.

| Команда | Описание | Когда использовать |
| :--- | :--- | :--- |
| `sudo systemctl start apache2` | Запустить сервер. | После установки или ручной остановки. |
| `sudo systemctl stop apache2` | Остановить сервер. | Для проведения технических работ или замены бинарников. |
| `sudo systemctl restart apache2` | **Полная** перезагрузка. Останавливает и запускает заново. **Разрывает все активные соединения.** | После обновления пакетов (`apt upgrade`) или изменения глобальных параметров, требующих пересоздания процессов (например, смена пользователя запуска). |
| `sudo systemctl reload apache2` | **Мягкая** перезагрузка. Перечитывает конфиги без разрыва текущих соединений. Новые запросы обслуживаются по новым правилам, старые дорабатываются по старым. | **Основная команда** после изменения конфигов (`sites-available`, `apache2.conf`). Безопасна для продакшена. |
| `sudo systemctl status apache2` | Показать статус, версию, время работы и последние строки логов. | Быстрая проверка, запущен ли сервис и нет ли критических ошибок при старте. |
| `sudo systemctl enable apache2` | Добавить в автозагрузку при загрузке ОС. | Однократно после установки. |

##### 2. Встроенная утилита управления (`apache2ctl`)
Это обертка над бинарным файлом `apache2`, передающая环境变量 из `/etc/apache2/envvars`. Позволяет управлять самим процессом веб-сервера.

| Команда | Описание | Пример использования |
| :--- | :--- | :--- |
| `sudo apache2ctl configtest` | Проверка синтаксиса всех конфигурационных файлов. | **Обязательно** перед каждым `reload`. Возвращает `Syntax OK` или ошибку с номером строки. |
| `sudo apache2ctl graceful` | Аналог `reload`, но более "вежливый". Ждет завершения текущих запросов перед применением новых конфигов для старых воркеров. | Альтернатива `systemctl reload` при высокой нагрузке. |
| `sudo apache2ctl fullstatus` | Показывает детальную статистику сервера (требуется модуль `mod_status`). | Мониторинг нагрузки, количества свободных воркеров. |
| `sudo apache2ctl -M` | Показать список **загруженных** модулей. | Проверка, включен ли нужный модуль (например, `rewrite_module`). |
| `sudo apache2ctl -l` | Показать список **скомпилированных** в ядро модулей. | Редко используется, показывает статические модули. |
| `sudo apache2ctl -V` | Показать версии сервера, пути к конфигам и параметры компиляции. | Отладка путей установки. |
| `apache2 -v` | Показать только версию Apache (без путей). | Быстрая проверка версии. |

> **Важно:** Всегда используйте `apache2ctl` вместо прямого вызова `apache2`, так как `apache2ctl` подхватывает переменные окружения (пути к логам, пользователю), определенные в `/etc/apache2/envvars`. Прямой запуск `apache2` может привести к ошибкам прав доступа к логам.

##### 3. Утилиты управления конфигурацией (`a2*`)
Специфичные для Debian/Ubuntu скрипты, автоматизирующие создание символических ссылок. **Не редактируйте папки `enabled` вручную!**

| Команда | Описание | Пример |
| :--- | :--- | :--- |
| `a2enmod <module>` | Включить модуль (`mods-available` → `mods-enabled`). | `a2enmod ssl`, `a2enmod rewrite` |
| `a2dismod <module>` | Отключить модуль. | `a2dismod autoindex` (отключить листинг директорий) |
| `a2ensite <site>` | Включить виртуальный хост (`sites-available` → `sites-enabled`). | `a2ensite example.com.conf` |
| `a2dissite <site>` | Отключить виртуальный хост. | `a2dissite 000-default.conf` |
| `a2enconf <config>` | Включить фрагмент конфигурации (`conf-available` → `conf-enabled`). | `a2enconf security`, `a2enconf serve-cgi-bin` |
| `a2disconf <config>` | Отключить фрагмент конфигурации. | `a2disconf security` |

> **Алгоритм внесения изменений:**
> 1. Создать/изменить файл в `...-available`.
> 2. Включить его: `a2ensite mysite` / `a2enmod ssl`.
> 3. Проверить синтаксис: `apache2ctl configtest`.
> 4. Применить: `systemctl reload apache2`.

##### 4. Работа с пользователями и паролями

| Утилита | Описание | Пример |
| :--- | :--- | :--- |
| `htpasswd` | Создание и управление файлами пользователей для Basic/Digest аутентификации. | `htpasswd -c /etc/apache2/.htpasswd user1` |
| `htdigest` | Управление файлами пользователей для Digest аутентификации (требует указания Realm). | `htdigest -c /etc/apache2/.htdigest "Realm" user1` |

##### 5. Анализ логов и производительности

| Команда / Утилита | Описание |
| :--- | :--- |
| `tail -f /var/log/apache2/access.log` | Просмотр входящих запросов в реальном времени. |
| `tail -f /var/log/apache2/error.log` | Просмотр ошибок в реальном времени. |
| `grep "404" /var/log/apache2/access.log` | Поиск всех запросов, завершившихся ошибкой 404. |
| `awk '{print $1}' access.log \| sort \| uniq -c \| sort -nr` | Топ IP-адресов, делающих больше всего запросов (анализ атак или ботов). |
| `logrotate` | Системная утилита для ротации логов (автоматическое сжатие и удаление старых логов). Конфиг: `/etc/logrotate.d/apache2`. |

##### 6. Тестирование производительности (`ab`)
Утилита **Apache Benchmark** (пакет `apache2-utils`). Позволяет нагрузочное тестирование прямо из командной строки.

```bash
# Отправить 1000 запросов, одновременно выполняя 10 запросов (-c 10)
ab -n 1000 -c 10 http://localhost/

# Вывод покажет: Requests per second, Time per request, Failed requests.
```
> **Осторожно:** Не запускайте `ab` против продакшен-серверов с большими значениями `-n` и `-c`, это может положить сайт (DoS-эффект). Используйте только на тестовых стендах.
### Глобальные настройки сервера (Identity & Network):

Эти директивы обычно размещаются в `/etc/apache2/apache2.conf` или `/etc/apache2/ports.conf`. Они задают фундаментальное поведение демона до обработки конкретных виртуальных хостов.

| Директива                | Пример                        | Где использовать         | Объяснение и нюансы                                                                                                                                                                                                                                                                                                                                                                        |
| :----------------------- | :---------------------------- | :----------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ServerRoot**           | `"/etc/apache2"`              | Глобально                | Базовая директория установки. Все относительные пути в конфиге (например, `Include mods-enabled/*.load`) считаются отсюда. **Не меняйте** без веской причины.                                                                                                                                                                                                                              |
| **ServerName**           | `example.com:80`              | Глобально / VHost        | Каноническое имя сервера. <br>• **Глобально:** Используется в редиректах, если не указано в `<VirtualHost>`, и для генерации сообщений об ошибках.<br>• **Важно:** Без этой директивы Apache может ругаться при старте (`Could not reliably determine the server's fully qualified domain name`). Часто ставят `ServerName localhost` глобально, а реальные домены — в виртуальных хостах. |
| **ServerAdmin**          | `admin@example.com`           | Глобально / VHost        | Email администратора. Отображается на страницах ошибок (404, 500), если включена `ServerSignature On`. Полезно для обратной связи от пользователей.                                                                                                                                                                                                                                        |
| **Listen**               | `80`, `443`, `192.168.1.5:80` | Глобально (`ports.conf`) | Определяет, какие сетевые интерфейсы и порты слушать.<br>• `Listen 80`: Слушать все интерфейсы (0.0.0.0) на 80 порту.<br>• `Listen 192.168.1.10:80`: Слушать только конкретный IP (полезно для многодомных серверов).<br>• Для HTTPS требуется модуль `ssl` и порт `443`.                                                                                                                  |
| **Timeout**              | `300`                         | Глобально                | Время ожидания (сек) для различных этапов:<br>1. Чтение запроса от клиента.<br>2. Получение ответа от CGI-скрипта (важно для PHP-FPM через proxy).<br>3. Отправка ответа клиенту.<br>⚠️ **Безопасность:** Значение > 300 делает сервер уязвимым к атакам типа *Slowloris* (медленные запросы). Рекомендуемое значение: `60`–`120`.                                                         |
| **KeepAlive**            | `On`                          | Глобально                | Разрешает HTTP Keep-Alive (постоянные соединения). Клиент скачивает HTML, CSS, JS и картинки через один TCP-коннект, экономя время на рукопожатия (TCP handshake). **Всегда `On`** для современных сайтов.                                                                                                                                                                                 |
| **MaxKeepAliveRequests** | `100`                         | Глобально                | Максимальное количество запросов в одном соединении. После достижения лимита соединение закрывается. <br>• Слишком мало: частые переподключения.<br>• Слишком много (>500): процесс Apache может долго висеть в памяти, занимая слот. `100` — золотая середина.                                                                                                                            |
| **KeepAliveTimeout**     | `5`                           | Глобально                | Время (сек), которое сервер ждет *следующий* запрос от клиента в рамках открытого соединения. <br>• Если клиент медленный, процесс Apache будет занят это время.<br>• **Оптимизация:** Снижение до `2`–`5` сек освобождает воркеры быстрее, повышая общую пропускную способность под нагрузкой.                                                                                            |

#### Дополнительные важные глобальные настройки:

*   **ServerTokens** и **ServerSignature** (Безопасность):
    По умолчанию Apache раскрывает свою версию и ОС в заголовках и страницах ошибок. Это помогает злоумышленникам искать уязвимости конкретной версии.
    ```apache
    # Скрыть версию Apache и ОС, показывать только "Apache"
    ServerTokens Prod
    
    # Убрать подпись (версию) из страниц ошибок (404, 500 и т.д.)
    ServerSignature Off
    ```
    *Результат заголовка:* Было `Apache/2.4.52 (Ubuntu)`, стало `Apache`.

*   **HostnameLookups** (Производительность):
    По умолчанию выключено (`Off`). Если включить (`On`), Apache будет делать обратный DNS-запрос для каждого клиента, чтобы записать доменное имя вместо IP в логи.
    ```apache
    HostnameLookups Off
    ```
    ⚠️ **Важно:** Включение сильно замедляет обработку запросов при большом трафике. Используйте `Off`, а если нужно имя — используйте `%{HOSTNAME}c` в логах или пост-обработку.

*   **Mutex** (Блокировки):
    Управляет механизмом блокировок для многопроцессного доступа к ресурсам (например, к файлам логов или кэшу).
    ```apache
    # Использование файловых блокировок в конкретной директории (часто нужно для производительности на некоторых ФС)
    Mutex file:/var/lock/apache2 default
    ```

*   **PidFile**:
    Путь к файлу, где хранится PID главного процесса Apache. Нужен для работы `systemctl` и скриптов управления.
    ```apache
    PidFile ${APACHE_PID_FILE}
    ```

*   **User и Group**:
    Определяет, от имени какого пользователя и группы работают дочерние процессы (воркеры), обрабатывающие запросы.
    ```apache
    User ${APACHE_RUN_USER}   # Обычно www-data
    Group ${APACHE_RUN_GROUP} # Обычно www-data
    ```
    ⚠️ **Никогда не запускайте** рабочие процессы от `root`. Главный процесс работает от root (чтобы занять порты 80/443), но сразу после старта создает дочерние процессы от указанного пользователя.

> **Чек-лист оптимизации сети:**
> 1. `KeepAlive On` — обязательно.
> 2. `KeepAliveTimeout 2-5` — не держите соединения открытыми зря.
> 3. `ServerTokens Prod` + `ServerSignature Off` — скройте информацию о системе.
> 4. `Timeout 60-120` — защита от медленных атак.
> 5. `HostnameLookups Off` — экономия ресурсов CPU и сети.

### Частые ошибки и решения:

Диагностика проблем в Apache обычно сводится к чтению логов (`/var/log/apache2/error.log`) и проверке прав доступа. Вот подробный разбор типичных сценариев.

**1. Ошибка 403 Forbidden (Доступ запрещен)**
Сервер нашел файл, но у процесса Apache нет прав на его чтение или вход в директорию.
*   **Причина А: Права файловой системы (Linux permissions).**
    Пользователь `www-data` не имеет прав на чтение файлов или выполнение (`x`) для входа в папку.
    *   **Решение:**
        ```bash
        # Владелец должен быть www-data (или группа www-data)
        chown -R www-data:www-data /var/www/mysite
        
        # Права: Директории 755 (rwxr-xr-x), Файлы 644 (rw-r--r--)
        find /var/www/mysite -type d -exec chmod 755 {} \;
        find /var/www/mysite -type f -exec chmod 644 {} \;
        ```
    *   **Нюанс SELinux/AppArmor:** Если права верные, но ошибка есть, проверьте логи безопасности (`/var/log/audit/audit.log` или `dmesg`). Возможно, система безопасности блокирует доступ к новой папке.

*   **Причина Б: Конфигурация Apache (`Require` directive).**
    В блоке `<Directory>` явно запрещен доступ или отсутствует разрешение.
    *   **Решение:** Убедитесь, что внутри блока директории есть разрешение:
        ```apache
        <Directory /var/www/mysite/public_html>
            Options Indexes FollowSymLinks
            AllowOverride All
            Require all granted  # <-- Критично!
        </Directory>
        ```
    *   **Конфликт порядков:** Если используется старая директива `Order allow,deny`, она может конфликтовать с новой `Require`. Используйте только новый синтаксис (Apache 2.4+).

*   **Причина В: Отсутствие `DirectoryIndex`.**
    Вы зашли на папку (например, `example.com/images/`), там нет файла `index.html`, а опция `Options -Indexes` запрещает показ списка файлов.
    *   **Решение:** Создайте `index.html` или разрешите листинг (`Options +Indexes` — **не рекомендуется** для продакшена).

**2. Ошибка 500 Internal Server Error (Внутренняя ошибка сервера)**
Сервер столкнулся с проблемой при обработке запроса. Без логов причину найти невозможно.
*   **Шаг 1: Чтение логов.**
    ```bash
    tail -n 50 /var/log/apache2/error.log
    # Или для конкретного сайта:
    tail -n 50 /var/log/apache2/mysite-error.log
    ```
*   **Частые причины:**
    *   **Синтаксическая ошибка в `.htaccess`:** Самая частая причина. Например, опечатка в директиве `RewriteRule` или использование модуля, который не включен (`a2enmod rewrite`).
        *   *Проверка:* Временно переименуйте `.htaccess` (например, в `.htaccess.bak`). Если ошибка исчезла — проблема в нем.
    *   **Ошибка скрипта (PHP/Python):** Скрипт упал с фатальной ошибкой (нехватка памяти, синтаксис PHP). Ищите строки вида `PHP Fatal error` в логе ошибок Apache или в отдельном `php-error.log`.
    *   **Неверные права на `.htaccess`:** Файл должен быть читаем пользователем `www-data`.

**3. Изменения не применяются**
Вы отредактировали конфиг, но сайт ведет себя по-старому.
*   **Чек-лист:**
    1.  **Перезагрузка службы:** Изменения в конфигах не вступают в силу автоматически.
        ```bash
        systemctl reload apache2  # Мягкая перезагрузка (без разрыва соединений)
        # или
        systemctl restart apache2 # Полная перезагрузка
        ```
    2.  **Активация сайта:** Вы создали файл в `sites-available`, но забыли создать ссылку в `sites-enabled`.
        ```bash
        a2ensite mysite.conf
        systemctl reload apache2
        ```
    3.  **Кэш браузера:** Браузер мог закэшировать редирект (301) или страницу ошибки.
        *   *Решение:* Откройте в режиме инкогнито или нажмите `Ctrl + F5` (Hard Refresh).
    4.  **Кэш DNS:** Если меняли IP домена, возможно, ваш компьютер помнит старый IP.
        *   *Решение:* `ipconfig /flushdns` (Windows) или `sudo systemd-resolve --flush-caches` (Linux).
    5.  **Порядок включения конфигов:** Проверьте, не перебивает ли ваш конфиг другой файл, подключенный позже (например, глобальные настройки в `apache2.conf`).

**4. Syntax error on line X (Синтаксическая ошибка)**
Apache отказывается запускаться или перезагружаться из-за ошибки в конфигурации.
*   **Инструмент диагностики:**
    ```bash
    sudo apache2ctl configtest
    # или
    sudo apachectl configtest
    ```
    *Вывод:* `Syntax error on line 42 of /etc/apache2/sites-enabled/mysite.conf: Invalid command 'RewriteEngin'...`
*   **Типичные опечатки:**
    *   `RewriteEngin` вместо `RewriteEngine`.
    *   Забытая закрывающая скобка `</Directory>` или `</VirtualHost>`.
    *   Использование директивы модуля, который не установлен (например, `SSLEngine on` без `a2enmod ssl`).
    *   Лишние пробелы или кавычки в путях.

**5. Ошибка "Address already in use" (Порт занят)**
При запуске: `(98)Address already in use: AH00072: make_sock: could not bind to address [::]:80`.
*   **Причина:** Другой процесс (часто Nginx, другой экземпляр Apache или Skype) уже слушает порт 80/443.
*   **Диагностика:**
    ```bash
    sudo ss -tulpn | grep :80
    # или
    sudo netstat -tulpn | grep :80
    ```
*   **Решение:** Остановить конфликтующий сервис (`systemctl stop nginx`) или изменить порт в `ports.conf` у Apache.

**6. Ошибка "Could not reliably determine the server's fully qualified domain name"**
Предупреждение при старте или в логах.
*   **Причина:** Не задана глобальная директива `ServerName`.
*   **Решение:** Добавьте в `/etc/apache2/apache2.conf`:
    ```apache
    ServerName localhost
    # Или ваш основной домен
    # ServerName example.com
    ```

> **Совет профи:**
> При отладке сложных проблем временно увеличьте уровень логирования до `debug` в виртуальном хосте:
> ```apache
> LogLevel debug
> ```
> Это запишет в лог каждый шаг обработки запроса (какие модули сработали, какие правила Rewrite применились), но **не забудьте вернуть `warn` или `error` после отладки**, иначе диск быстро заполнится.
