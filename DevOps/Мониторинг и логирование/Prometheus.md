
**Prometheus** — это база данных временных рядов (Time Series Database — TSDB), которая собирает метрики (числовые данные с временной меткой) с ваших серверов, приложений и сетей, позволяет делать по ним запросы и настраивать алерты (уведомления), если что-то идет не так.

В отличие от многих старых систем (как Zabbix или Nagios), Prometheus работает по принципу **Pull (Вытягивание)**:

1. **Scraping (Скрейпинг):** Сервер Prometheus сам периодически опрашивает целевые объекты (targets) по HTTP, забирая у них метрики.
2. **Хранение:** Все данные хранятся локально на диске сервера Prometheus в виде временных рядов.
3. **Модель данных:** Данные — это не просто числа, а пары `Метрика + Метки (Labels)`.
    - Пример: `http_requests_total{method="POST", handler="/api", status="200"}`

#### Основные компоненты:

Prometheus — это не один монолит, а набор компонентов, хотя чаще всего под словом "Prometheus" подразумевают основной сервер.

1. **Prometheus Server (Ядро):**
    - **Retrieval (Сборщик):** Модуль, который ходит по адресам и забирает метрики.
    - **TSDB (База данных):** Хранит данные на диске. Данные сжимаются и разбиваются на блоки (chunks). Есть WAL (Write Ahead Log) для защиты от потери данных при краше.
    - **HTTP Server:** Принимает запросы от Grafana или UI.
    - **PromQL Engine:** Движок, который выполняет запросы к данным.
2. **Service Discovery (Обнаружение сервисов):**
    - Prometheus должен знать, кого опрашивать. В статическом конфиге вы пишете IP-адреса.
    - В динамической среде (Kubernetes, AWS EC2, Consul) Prometheus сам спрашивает у API оркестратора: "Какие поды сейчас живы?" и автоматически добавляет их в список целей (targets).
3. **Alertmanager (Менеджер оповещений):**
    - Это **отдельная программа**.
    - Prometheus оценивает правила (например, `if error_rate > 0.1`). Если условие истинно, он отправляет "алерт" в Alertmanager.
    - Alertmanager занимается группировкой, подавлением дублей (silencing) и отправкой в каналы (Slack, Email, Telegram, PagerDuty).
4. **Exporters (Экспортеры):**
    - Прокси-сервисы, которые превращают метрики сторонних систем в формат Prometheus.
5. **Pushgateway (Шлюз для пуша):**
    - Для задач, которые живут недолго (cron-скрипты). Они не могут ждать, пока Prometheus их опросит, поэтому сами отправляют данные в Pushgateway, а Prometheus забирает их оттуда.
#### Типы метрик:

1. Counter (Счетчик):
	- **Суть:** Метрика, которая может только **расти** (увеличиваться) или сбрасываться в ноль (при перезапуске процесса). Она никогда не уменьшается.
	- **Зачем:** Подсчет событий, которые произошли.
	- **Примеры:** Количество запросов, количество ошибок, количество проданных товаров.
	- **Пример данных:**

	```prometheus
	http_requests_total{method="POST", handler="/api"} 1024
	```

	- **Как запрашивать:** Сами по себе числа счетчика малоинформативны (что значит 1024 запроса всего?). Обычно используют функцию `rate()` или `increase()`, чтобы узнать скорость роста за промежуток времени.
	    - `rate(http_requests_total[5m])` — сколько запросов в секунду в среднем за последние 5 минут.

2. Gauge (Датчик / Измеритель):
	- **Суть:** Метрика, которая может **расти и убывать**. Это "моментальный снимок" состояния.
	- **Зачем:** Измерение текущего состояния системы.
	- **Примеры:** Использование оперативной памяти, температура процессора, количество активных пользователей, размер очереди.
	- **Пример данных:**
	
	```prometheus
	node_memory_MemAvailable_bytes 2147483648
	```

	- **Как запрашивать:** Берется текущее значение или функции `max()`, `min()`, `avg()`. Функция `rate()` здесь обычно не применяется.

3. Histogram (Гистограмма):
	- **Суть:** Разбивает наблюдения (например, время ответа) по "корзинам" (buckets). Позволяет понять распределение значений.
	- **Зачем:** Анализ латентности (времени ответа), размеров запросов.
	- **Как работает:** Создает несколько метрик автоматически:
	    1. Счетчики для каждой корзины (`_bucket`): сколько запросов было быстрее определенного порога.
	    2. Сумму всех наблюдений (`_sum`).
	    3. Количество всех наблюдений (`_count`).
	- **Пример данных:**
	
	```prometheus
	http_request_duration_seconds_bucket{le="0.1"} 24054
	http_request_duration_seconds_bucket{le="0.5"} 33444
	http_request_duration_seconds_bucket{le="1.0"} 100392
	http_request_duration_seconds_sum 53423
	http_request_duration_seconds_count 144340
	```

	- **Как запрашивать:** Позволяет вычислять перцентили (например, 95-й перцентиль времени ответа) на стороне сервера с помощью функции `histogram_quantile()`.
	    - `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))`

4. Summary (Сводка):
	- **Суть:** Похожа на гистограмму, но вычисляет квантили (перцентили) **на стороне клиента** (в приложении), а не на стороне Prometheus.
	- **Зачем:** То же, что и гистограмма, но с меньшими накладными расходами на хранение, однако менее гибкая при агрегации.
	- **Минус:** Нельзя агрегировать несколько Summary в одну (нельзя посчитать средний 95-й перцентиль по всем серверам).
	- **Рекомендация:** В 95% случаев лучше использовать **Histogram**, так как он гибче.

#### Принцип вытягивания метрик с клиента:

Это фундаментальное отличие от систем вроде Zabbix или StatsD (которые часто работают по Push-модели, когда агент сам отправляет данные).

**Процесс скрейпинга (Scraping):**

1. **Конфигурация:** В файле `prometheus.yml` указан `scrape_interval` (например, 15 секунд) и список целей (`targets`).
2. **HTTP Запрос:** Каждые 15 секунд сервер Prometheus отправляет обычный **HTTP GET запрос** по адресу цели (например, `http://192.168.1.5:9100/metrics`).
3. **Ответ:** Цель возвращает текст в специальном формате (Prometheus Exposition Format).
4. **Парсинг:** Prometheus разбирает текст, добавляет к метрикам системные лейблы (например, `instance="192.168.1.5:9100"`, `job="node"`) и текущую временную метку.
5. **Запись:** Данные записываются в TSDB.

**Почему Pull, а не Push?**

- **Контроль:** Вы точно знаете, когда и как часто собираются данные.
- **Надежность:** Если сеть между приложением и мониторингом пропадет, приложение продолжит работать. Когда сеть восстановится, Prometheus просто придет за новыми данными. (При Push-модели приложение могло бы потерять метрики, если некуда их отправить, так как клиент мог не отправлять повторно данные при возобновлении сети).
- **Безопасность:** Легче настроить фаервол, разрешив входящие соединения на сервер мониторинга, чем открывать доступ из всех приложений наружу.

Не все приложения "родные" для Prometheus. Старая база данных (PostgreSQL), железо сервера или сетевое оборудование не умеют отдавать метрики по HTTP в формате Prometheus. Для этого нужен **Экспортер**:

- **С одной стороны:** Он общается с системой на её языке (читает `/proc` в Linux, делает SQL-запрос к БД, опрашивает SNMP устройство).
- **С другой стороны:** Он открывает HTTP-порт (обычно 9100+) и отдает данные в формате Prometheus, когда приходит сервер мониторинга.

**Типы интеграции:**

1. **Через Экспортер (Sidecar / DaemonSet):**
    - _Пример:_ **Node Exporter**.
    - Вы ставите его на сервер. Он читает нагрузку CPU, диск, сеть.
    - Prometheus стучится в Node Exporter.
    - _Плюс:_ Не нужно менять код приложения.
    - _Минус:_ Нужно устанавливать и поддерживать дополнительное ПО.
2. **Прямая инструментация (Client Libraries):**
    - _Пример:_ Ваше приложение на Go/Python/Java.
    - Вы используете официальную библиотеку Prometheus в коде.
    - В коде вы создаете метрику: `requests_counter.inc()`.
    - Приложение само открывает порт (например, `/metrics`), куда Prometheus приходит за данными.
    - _Плюс:_ Максимальная детализация (бизнес-метрики).
    - _Минус:_ Нужно менять код приложения.
3. **Blackbox Exporter:**
    - Это особый случай. Он не стоит на целевом сервере.
    - Prometheus просит Blackbox: "Проверь, доступен ли google.com по HTTPS".
    - Blackbox сам делает запрос, замеряет время и возвращает метрику: `probe_success 1` или `0`.

### Основные сущности:

#### Sample:

Это атомарная точка данных, одна запись в базе данных. Пример структуры записи:

```
<метрика>{<лейблы>} <значение> <временная_метка>

node_cpu_seconds_total{cpu="0",mode="idle",instance="server-01:9100",job="node"} 12345.67 1712345678000
```

| Кто создает                                   | Как                                                                                  | Когда                                           |
| --------------------------------------------- | ------------------------------------------------------------------------------------ | ----------------------------------------------- |
| **Экспортер** (Node Exporter, MySQL Exporter) | Читает системные данные (`/proc`, SQL-запрос) → форматирует в текст → отдает по HTTP | При каждом запросе Prometheus к `/metrics`      |
| **Приложение с client library**               | Вызов кода: `counter.inc()` → библиотека хранит в памяти → отдает по запросу         | При выполнении кода приложения + при скрейпинге |
| **Pushgateway**                               | Принимает POST-запрос от batch-скрипта → хранит в памяти                             | Когда скрипт явно отправляет данные             |

> Sample создается **в момент скрейпинга** (когда Prometheus запрашивает `/metrics`). Экспортер генерирует текст "на лету" из текущих значений системы.


#### Label (Лейбл) — тег/измерение:

это метка которая позволяет:

1. **Фильтровать:** Показать метрики только для конкретного сервера (`instance="web-01"`).
2. **Группировать:** Посчитать сумму по всем серверам, но разбить по городам.
3. **Сравнивать:** Сравнить нагрузку на `env="prod"` и `env="test"`.

Структура:

```
key="value"
```

Пример создания label-ов в конфиге:
```yaml
scrape_configs:
  - job_name: 'api'
    static_configs:
      - targets: ['api-01:8080']
        labels:  # ← Создаем лейблы для всех метрик с этого таргета
          env: 'production'
          team: 'backend'
    metric_relabel_configs:
      - target_label: 'region'  # ← Создаем новый лейбл
        replacement: 'eu-west'
```

| Источник                                         | Как создается                                               | Пример                                        |
| ------------------------------------------------ | ----------------------------------------------------------- | --------------------------------------------- |
| **Код приложения**                               | При инкременте метрики: `labels(method="GET")`              | `http_requests_total{method="GET"}`           |
| **Экспортер**                                    | Жестко прописан в коде экспортера или извлечен из контекста | `node_cpu_seconds_total{cpu="0",mode="idle"}` |
| **Prometheus (автоматически)**                   | Добавляет `job`, `instance` при скрейпинге                  | `...{job="api", instance="10.0.0.1:8080"}`    |
| **Конфиг `external_labels`**                     | Добавляется глобально ко всем метрикам                      | `...{cluster="prod-eu"}`                      |
| **Конфиг `labels` в scrape_configs**             | Добавляется к целям в статическом конфиге                   | `...{env="production"}`                       |
| **`relabel_configs` / `metric_relabel_configs`** | Манипуляции в конфиге Prometheus                            | Переименование, добавление, удаление          |

**Главное правило:** Лейблы должны иметь **ограниченный набор значений**.

- ✅ Хорошо: `status="200"`, `method="GET"`, `env="prod"` (значений мало).
- ❌ Плохо: `user_id="12345"`, `email="user@example.com"` (значений миллионы — это "убьет" базу данных).

> **Важно:** Лейблы должны быть **объявлены заранее** в коде (для client library) или жестко заданы в экспортере. Нельзя добавить произвольный лейбл "на лету" без объявления — это вызовет ошибку.

#### Time Series (Временной ряд):

Журнал измерений одной конкретной метки: "Температура на складе №1, полка А, уровень 3" — записывается каждый час. **Временной ряд создается автоматически**, когда появляется **первый сэмпл** с уникальной комбинацией:

**Проблема, которую решает:**

Представьте, что у вас 3 сервера (`web-01`, `web-02`, `web-03`), каждый отдает метрику `http_requests_total`.

Структура:
```
<метрика> + <полный_набор_лейблов>
```

| Этап                   | Что происходит                                                                                                                          |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| 1️⃣ Первый сэмпл       | Prometheus видит: `http_requests_total{method="GET",status="200",instance="api-01"}` — такой комбинации еще нет → **создает новый ряд** |
| 2️⃣ Последующие сэмплы | Та же комбинация → **добавляет точку** в существующий ряд                                                                               |
| 3️⃣ Новый лейбл        | Появился `status="500"` → **создает НОВЫЙ ряд** (это уже другая серия)                                                                  |

**Пример:**
```tsdb
# Первый вызов — создается ряд #1
http_requests_total{method="GET",status="200"}  # ← Новый ряд создан!

# Второй вызов — добавляется точка в ряд #1
http_requests_total{method="GET",status="200"}  # ← Точка в существующий ряд

# Третий вызов с другим статусом — создается ряд #2
http_requests_total{method="GET",status="500"}  # ← Новый ряд создан!
```

> **Важно!** Каждый уникальный набор лейблов = новый временной ряд. 1 метрика × 1000 комбинаций лейблов = 1000 рядов. Это влияет на память и производительность!

#### Metric (Метрика):

Структура:

```
# HELP <name> <description>
# TYPE <name> <counter|gauge|histogram|summary>
<name>{...} ...
```

| Кто                    | Как                                    | Пример                                  |
| ---------------------- | -------------------------------------- | --------------------------------------- |
| **Разработчик в коде** | Объявляет метрику через client library | `Counter('http_requests_total', ...)`   |
| **Автор экспортера**   | Прописывает в коде экспортера          | `# TYPE node_cpu_seconds_total counter` |
| **Recording Rule**     | Создается автоматически по правилу     | `record: job:cpu:avg`                   |

После объявления метрика **еще не существует в Prometheus**. Она появится только когда:

1. Приложение отдаст `/metrics`
2. Prometheus скрейпнет этот эндпоинт
3. В ответе будет хотя бы один сэмпл с этим именем

#### Target (Цель) — адрес для сбора метрик:

Структура:

```
<host>:<port>
```

| Способ                                   | Как                                | Пример конфига                               |
| ---------------------------------------- | ---------------------------------- | -------------------------------------------- |
| **`static_configs`**                     | Явно прописан в `prometheus.yml`   | `targets: ['192.168.1.10:9100']`             |
| **Service Discovery (K8s)**              | Prometheus спрашивает API кластера | `kubernetes_sd_configs: {role: pod}`         |
| **Service Discovery (file)**             | Читает внешний JSON-файл           | `file_sd_configs: {files: ['targets.json']}` |
| **Service Discovery (consul, ec2, ...)** | Интеграция с внешними системами    | `consul_sd_configs: {server: 'consul:8500'}` |

#### Instance (Экземпляр) — уникальный идентификатор цели:

Метка `instance="<host>:<port>"`, добавляемая Prometheus автоматически.

Создается **автоматически Prometheus-ом** при скрейпинге каждой цели:

```
Вы указали в конфиге: targets: ['api-01:8080']

Prometheus при скрейпинге добавит:
→ Все метрики с этого таргета получат лейбл: instance="api-01:8080"
```

#### Job:

Лейбл, который добавляется ко всем метрикам с целей одной группы. Создается **автоматически Prometheus** на основе `job_name` в `scrape_configs`:

```yaml
scrape_configs:
  - job_name: 'api-servers'  # ← Значение будущего лейбла job="api-servers"
    static_configs:
      - targets:
          - 'api-01:8080'
          - 'api-02:8080'
```

#### Labelset (Набор лейблов) - полный контекст сэмпла:

Пример:

```tsdb
{job="api", instance="api-01:8080", env="prod", method="GET", status="200"}
```

**Собирается автоматически** из всех источников:

1. Из кода/экспортера: method="GET", status="200"
2. Prometheus добавляет: job="api", instance="api-01:8080"
3. external_labels из конфига: cluster="prod-eu"
4. labels из static_configs: env="production"
5. relabel_configs: может добавить/изменить/удалить любой лейбл


```yml
global:
  external_labels:
    cluster: 'prod-eu'  # ← Источник #3

scrape_configs:
  - job_name: 'api'     # ← Источник #2 (job)
    static_configs:
      - targets: ['api-01:8080']  # ← Источник #2 (instance)
        labels:
          env: 'production'       # ← Источник #4
    metric_relabel_configs:
      - target_label: 'region'    # ← Источник #5
        replacement: 'eu-west'
```

#### Rule (Правило) — логика обработки:

**Recording Rule (Правило записи)**. Предварительно вычисленная метрика.

**Как создается:**

1. Вы пишете YAML-файл с правилом
2. Указываете путь к файлу в `rule_files` в `prometheus.yml`
3. Prometheus загружает правило при старте (или по SIGHUP)
4. Каждые `evaluation_interval` правило вычисляется → создается новая метрика

Пример файла `rules/recording.yml`:

```yaml
groups:
  - name: api-recording-rules
    interval: 30s  # ← Как часто вычислять (переопределяет global)
    rules:
      - record: job:http_requests:rate5m  # ← Имя новой метрики!
        expr: sum(rate(http_requests_total[5m])) by (job)
```

Пример подключения в `prometheus.yml`:

```yml
rule_files:
  - "rules/recording/*.yml"  # ← Prometheus загрузит все файлы из этой папки
```

**Alerting Rule (Правило алерта).** Условие для генерации уведомления.

**Как создается:** Аналогично recording rule, но с дополнительной логикой.

Пример файла `rules/alerts.yml`:

```yml
groups:
  - name: api-alerts
    rules:
      - alert: HighErrorRate  # ← Имя алерта
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.1  # ← Условие
        for: 10m              # ← Должно держаться 10 минут, чтобы сработать
        labels:               # ← Лейблы для алерта (попадут в уведомление)
          severity: critical
          team: backend
        annotations:          # ← Человекочитаемое описание (поддерживает шаблонизацию)
          summary: "Высокий уровень ошибок на {{ $labels.instance }}"
          description: "Доля ошибок > 10% в течение 10 минут. Текущее значение: {{ $value }}"
```

#### Alert (Алерт) — сработавшее правило.

Экземпляр правила, который в данный момент истинен. Как создается:

| Этап            | Кто          | Что делает                                                                       |
| --------------- | ------------ | -------------------------------------------------------------------------------- |
| 1️⃣ Оценка      | Prometheus   | Проверяет `expr` правила каждые `evaluation_interval`                            |
| 2️⃣ PENDING     | Prometheus   | Если `expr = true` → создает алерт со статусом `PENDING`, запускает таймер `for` |
| 3️⃣ FIRING      | Prometheus   | Если таймер `for` истек → меняет статус на `FIRING`, отправляет в Alertmanager   |
| 4️⃣ Уведомление | Alertmanager | Получает алерт → применяет группировку/подавление → отправляет в каналы          |
| 5️⃣ RESOLVED    | Prometheus   | Когда `expr = false` → отправляет `RESOLVED` → Alertmanager шлет "всё починили"  |

Сценарий работы:

```yml
# Правило
- alert: HighCPU
  expr: avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) < 0.2
  for: 5m
```

```
Время 14:00: CPU idle = 25% → expr = false → ничего не происходит
Время 14:05: CPU idle = 15% → expr = true → алерт создан, статус: PENDING, таймер: 0/5m
Время 14:07: CPU idle = 10% → expr = true → таймер: 2/5m
Время 14:10: CPU idle = 8%  → expr = true → таймер истек! → статус: FIRING → отправка в Alertmanager
Время 14:12: CPU idle = 30% → expr = false → статус: RESOLVED → уведомление о восстановлении
```

#### TSDB (Time Series Database) — локальное хранилище:

Встроенная база данных Prometheus для хранения временных рядов на диске.

**Как создается и управляется:**

| Компонент                 | Как создается                                                                                | Назначение                                             |
| ------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------ |
| **WAL (Write Ahead Log)** | Автоматически при старте Prometheus в `path: /prometheus/data/wal`                           | Журнал последних данных для защиты от потери при краше |
| **Блоки (blocks)**        | Каждые `min_block_duration` (по умолчанию 2h) Prometheus "замораживает" данные в сжатый блок | Долгосрочное хранение, сжатие, эффективные запросы     |
| **Индекс**                | Создается при формировании блока                                                             | Быстрый поиск рядов по лейблам                         |
| **Meta-файлы**            | Автоматически                                                                                | Метаданные: время блока, статистика                    |

### Настройка prometheus:

```yml

  # Как часто опрашивать цели (targets). 
  # Можно переопределить для конкретного job в scrape_configs.
  scrape_interval: 15s

  # Обычно совпадает со scrape_interval или кратен ему.
  evaluation_interval: 15s

  # Нужно чтобы отличать источники при просмотре логов. Если есть 10 кластеров и все отправляется в одно хранилище (`remote_write`), эти метки помогут понять: "Эта метрика пришла из продакшена в Европе, а эта — из теста в Азии".
  external_labels:

    region: 'eu-west-1'

  scrape_timeout: 10s

# =============================================================================
# Говорит Prometheus, где лежат файлы с инструкциями: "Когда кричать" (алерты) и "Как заранее посчитать сложные цифры" (recording rules).

alerting:

        - targets:

            - alertmanager-2:9093

    retention:
      time: 15d
      # Или по размеру:
      size: 50GB
      
    # Как группировать данные на диске. Можно не менять, значения по умолчанию обычно хороши. Вы не кидаете каждую метрику в общую кучу. Вы собираете метрики за 2 часа в один большой контейнер (`блок`), запечатываете его и подписываете "12:00-14:00". Так легче искать и архивировать.
    # Минимальное время блока (advanced tuning)
    min_block_duration: 2h
    # Максимальное время блока
    max_block_duration: 2h

# =============================================================================

remote_write:

  - url: "http://thanos-receive:19291/api/v1/receive"
    # Сколько ждать, пока центральный офис подтвердит, что принял данные.
    remote_timeout: 30s

    queue_config:
		# Максимальное количество sample-ов, которые могут ждать в очереди, пока не будут отправлены. Если проблемы с сетью то sample переносятся в очередь ожидания
		capacity: 10000
		
		# Максимальное количество параллельных соединений/потоков для отправки данных.
	    max_shards: 50
	    
	    # Сколько сэмплов упаковывать в один HTTP-запрос. Слишком маленький размер → много мелких запросов, накладные расходы на HTTP. Слишком большой размер → один запрос может таймаутиться или "повесить" соединение.
	    max_samples_per_send: 5000
      

    # Фильтрует, ЧТО отправлять в удаленное хранилище.
    write_relabel_configs:

      - source_labels: [__name__]

  - url: "http://thanos-query:10902/api/v1/read"
    read_recent: true # Читать свежие данные из локального TSDB, старые отсюда

  - job_name: 'node-exporter'

          env: 'production'

  - job_name: 'kubernetes-pods'
```

#### Разбор ключевых параметров:

##### Что такое `remote_write` / `remote_read`:

Это специализированное хранилище, которое понимает Prometheus Remote API.

Prometheus использует **специальный протокол** для удаленной записи/чтения:
- **Формат:** HTTP POST/GET запросы.
- **Тело запроса:** Протокол Buffer (protobuf) + сжатие Snappy.
- **Эндпоинты:**
    - Запись: `POST /api/v1/receive`
    - Чтение: `POST /api/v1/read`

**Какие системы понимают этот протокол?**

| Система                    | Тип                          | Описание                                                                                  |
| -------------------------- | ---------------------------- | ----------------------------------------------------------------------------------------- |
| **Thanos Receive**         | Долгосрочное хранение        | Часть экосистемы Thanos, принимает `remote_write`, хранит в объектном хранилище (S3, GCS) |
| **Cortex**                 | Долгосрочное хранение        | Масштабируемое хранилище от Grafana Labs, тоже использует S3                              |
| **VictoriaMetrics**        | TSDB + долгосрочное хранение | Совместима с Prometheus, может быть как "приемником", так и заменой Prometheus            |
| **M3DB**                   | Долгосрочное хранение        | От Uber, распределенная TSDB                                                              |
| **InfluxDB** (с адаптером) | Универсальная TSDB           | Может принимать данные Prometheus через плагин                                            |

##### write_relabel_configs:
```yaml
write_relabel_configs:
  - source_labels: [__name__]
    regex: 'http_request_duration_seconds_bucket'
    action: drop
```

**Проблема:** Метрики с суффиксом `_bucket` (гистограммы) очень "дорогие":

- Одна гистограмма может создавать **10-20 временных рядов** (по одному на каждую "корзину" + `_sum`, `_count`).
- Если у вас 100 сервисов с гистограммами → это **тысячи дополнительных рядов**.
- Это раздувает объем данных, увеличивает стоимость хранения и замедляет запросы.

**Решение:** Если вам не нужны детальные перцентили в долгосрочном архиве — просто не отправляйте `_bucket` метрики туда. Оставьте только `_sum` и `_count` — их достаточно для расчета среднего.

**Удалить "шумные" лейблы (снижение кардинальности):**

```yaml
- source_labels: [user_id]
  action: labeldrop  # Удаляет лейбл user_id из всех метрик перед отправкой
```

**Отправлять только определенные метрики (белый список):**

```yaml
- source_labels: [__name__]
  regex: 'node_.*|http_.*'
  action: keep  # Отправлять только метрики, начинающиеся с node_ или http_
```

**Добавить метку перед отправкой:**

```yaml
- target_label: source_cluster
  replacement: 'prod-eu-1'  # Добавить метку "источник" ко всем отправляемым метрикам
```
