
Prometheus — это система мониторинга и алертинга с открытым исходным кодом. Главное отличие Prometheus от классических систем (как Zabbix или Nagios) — это **Pull-модель** сбора метрик и ориентация на **временные ряды (Time Series)**.

### Архитектура и основные компоненты:

#### Prometheus Server (Ядро):

Это главный демон, который делает три вещи:

1. **Scraping (Сбор данных):** Периодически опрашивает (делает HTTP GET запросы) целевые объекты (targets) по заданным интервалам.
2. **Storage (Хранение):** Сохраняет полученные данные в собственном формате на диске (TSDB — Time Series Database).
3. **HTTP Server:** Предоставляет API для запросов данных (PromQL) и интерфейс для управления.

> **Важно:** Prometheus хранит данные локально. Для долгосрочного хранения (долше нескольких недель/месяцев) обычно используют интеграцию с удаленными хранилищами (Thanos, Cortex, M3DB).

#### Client Libraries (Клиентские библиотеки):

Чтобы приложение могло отдавать метрики, в его код нужно внедрить библиотеку.

- Существуют официальные библиотеки для Go, Java/Scala, Python, Ruby. Для других языков есть сторонние.
- **Как это работает:** Библиотека создает HTTP-эндпоинт (обычно `/metrics`), на котором висит текст в определенном формате. Когда Prometheus делает `GET /metrics`, библиотека генерирует ответ "на лету".

#### Exporters (Экспортеры):

Если ты не можешь изменить код приложения (например, мониторим Nginx, PostgreSQL, сам сервер Linux, Cisco роутер), используются экспортеры. Это отдельные маленькие программы/агенты, которые собирают метрики со сторонней системы и отдают их в формате Prometheus. **Примеры:**
- `node_exporter`: собирает метрики ОС (CPU, RAM, Disk I/O, Network) с Linux/Unix.
- `blackbox_exporter`: проверяет доступность эндпоинтов по HTTP, HTTPS, DNS, TCP, ICMP (аналог ping/check_http).
- `snmp_exporter`: для сетевого оборудования.
- `postgres_exporter`: для БД PostgreSQL.

#### Pushgateway:

Используется для короткоживущих задач (batch jobs, cron scripts), которые не могут быть "опрошены" Prometheus, так как они живут слишком мало.

- Скрипт выполняет работу, отправляет метрики в Pushgateway через HTTP POST.
- Prometheus потом забирает эти метрики из Pushgateway.

> **Внимание:** Не используй Pushgateway для обычных сервисов. Это нарушает принцип Pull-модели и усложняет отказоустойчивость.

#### Alertmanager:

Prometheus сам по себе только собирает данные и вычисляет правила. Когда срабатывает правило алерта, Prometheus отправляет уведомление в **Alertmanager**. Alertmanager занимается:

- **Deduplication:** Если 10 серверов упали одновременно, он схлопнет это в одно уведомление.
- **Grouping:** Группирует алерты (например, все алерты по кластеру БД в одно сообщение).
- **Routing:** Маршрутизирует уведомления куда нужно: в Telegram, Slack, Email, PagerDuty, VictorOps.
- **Silencing/Inhibition:** Возможность заглушить алерты на время обслуживания или подавить вторичные алерты (если упал мастер-нода, не спамить алертами о том, что реплики недоступны).

#### Service Discovery (SD):

Так как targets может быть тысячи и они динамически меняются (особенно в Kubernetes или облаках), вручную прописывать IP в конфиге нельзя. Prometheus поддерживает SD из:
- Kubernetes (автоматически находит поды, сервисы, ноды).
- Consul, Etcd.
- DNS (SRV records).
- Файлов (`file_sd_configs`) — удобно для Ansible/Terraform: ты генерируешь JSON/YAML файл со списком хостов, а Prometheus его перечитывает.

#### Принцип работы (Data Flow):

1. **Конфигурация:** Ты описываешь в `prometheus.yml`, кого опрашивать (`scrape_configs`) и как часто (`scrape_interval`).
2. **Service Discovery:** Prometheus определяет список целей (targets).
3. **Scraping:** Каждый `scrape_interval` Prometheus делает HTTP запрос к каждому target'у по адресу `/metrics`.
4. **Parsing:** Полученный текст парсится. Каждая строка — это метрика.
5. **Storage:** Данные сохраняются в TSDB на диск. Старые данные удаляются или компактизируются согласно настройкам retention.
6. **Querying:** Пользователь (или Grafana) делает запрос через PromQL к API Prometheus.
7. **Alerting:** Rule Engine оценивает правила алертинга на основе данных. Если условие истинно -> отправка в Alertmanager.

### Модель данных (Data Model):

В Prometheus есть два уровня сущностей:

1. **Конфигурационные/Логические** (то, что ты описываешь в `prometheus.yml` и то, как Prometheus видит мир).
2. **Хранимые/Физические** (то, что лежит на диске в TSDB и с чем ты работаешь через PromQL).

#### Иерархия сущностей (От конфигурации к данным):

1. Уровень сбора (Scrape Context):
	1. **Job (Задача)**
	    - **Определение:** Логическая группа однотипных целей. Например, все веб-серверы nginx — это один job, все базы данных postgres — другой.
	    - **Зачем нужно:** Позволяет агрегировать метрики по типу сервиса. В метриках автоматически появляется метка `job="name"`.
	    - _Пример:_ `job_name: "node-exporters"`
	2. **Instance (Экземпляр / Target)**
	    - **Определение:** Конкретный источник метрик внутри Job. Обычно это `IP:Port` или `Hostname:Port`.
	    - **Зачем нужно:** Чтобы различать отдельные серверы. В метриках появляется лейбл `instance="192.168.1.10:9100"`.
	    - _Связь:_ Один Job содержит много Instances.

2. Уровень данных (Data Model):
	1.  **Metric Name (Имя метрики)**
	    *   **Определение:** Строковый идентификатор того, *что* мы измеряем.
	    *   **Правила:** Должно соответствовать regex `[a-zA-Z_:][a-zA-Z0-9_:]*`.
	    *   *Пример:* `http_requests_total`, `node_cpu_seconds_total`.
	
	2.  **Label Set (Набор лейблов)**
	    *   **Определение:** Набор пар `key="value"`, которые описывают измерения метрики.
	    *   **Важно:** Лейблы делают метрику многомерной.
	    *   *Пример:* `{method="POST", handler="/api/login", status="200"}`.
	    *   *Связь:* Имя метрики + Уникальный набор лейблов = **Time Series**.
	
	3.  **Time Series (Временной ряд)**
	    *   **Определение:** Поток данных, идентифицируемый уникальной комбинацией **Имени метрики** и **Набора лейблов**.
	    *   **Суть:** Это «труба», в которую текут данные. Если меняется хоть один лейбл — это уже *другой* временной ряд.
	    *   *Пример:* Ряд `http_requests_total{method="GET"}` и ряд `http_requests_total{method="POST"}` — это два разных Time Series.
	    * **Cardinality (Кардинальность)** — это количество уникальных Time Series. Если у тебя 100 серверов (`instance`) и 1000 методов API (`handler`), то метрика `http_requests_total` создаст $100 \times 1000 = 100,000$ временных рядов. Если добавишь лейбл `user_id` (1 млн пользователей), получишь $100,000 \times 1,000,000 = 10^{11}$ рядов. **Это убьет Prometheus.** Никогда не используй высококардинальные данные в лейблах
	
	4.  **Sample (Сэмпл / Точка данных)**
	    *   **Определение:** Конкретное значение в конкретный момент времени. Минимальная запись в TSDB.
	    *   **Структура:** `(Timestamp, Value)`.
	    *   **Value:** Float64 (число с плавающей точкой).
	    *   **Timestamp:** Unix timestamp в миллисекундах.
	    *   *Связь:* Time Series состоит из упорядоченной последовательности Sample.

3. **Уровень абстракции.** Когда ты сидишь в Grafana или пишешь алерты, ты работаешь с абстракциями более высокого уровня, чем просто Sample.

	1.  **Instant Vector (Мгновенный вектор)**
	    *   **Что это:** Набор Time Series, где у каждого ряда есть ровно одно значение (последний известный Sample) на выбранный момент времени.
	    *   **Где используется:** В большинстве графиков в Grafana, в условиях алертов.
	    *   *Пример запроса:* `up` (показывает статус всех таргетов прямо сейчас).
	
	2.  **Range Vector (Диапазонный вектор)**
	    *   **Что это:** Набор Time Series, где у каждого ряда есть массив значений (Sample) за определенный промежуток времени.
	    *   **Где используется:** Внутри функций агрегации (`rate`, `increase`, `avg_over_time`).
	    *   *Синтаксис:* `metric_name[5m]`.
	    *   *Пример:* `http_requests_total[5m]` вернет все точки за последние 5 минут для каждого ряда.
	
	3.  **Scalar (Скаляр)**
	    *   **Что это:** Простое числовое значение без временного ряда и лейблов.
	    *   **Где используется:** В математических операциях.
	    *   *Пример:* `5`, `pi()`.
	
	4.  **String (Строка)**
	    *   **Что это:** Строковое значение. Используется редко, в основном для отладки или сложных условий `label_replace`.

#### Как сущности создаются в TSDB (Внутренности хранения):

Когда Prometheus скрейпит target, происходит следующее:

1.  **Получение текста:** Prometheus получает текст от экспортера.
    ```text
    http_requests_total{method="GET", status="200"} 100
    http_requests_total{method="POST", status="200"} 50
    ```
2.  **Парсинг и идентификация:**
    *   Парсер разбирает строку.
    *   Проверяет, существует ли уже такой **Time Series** (по хешу имени + лейблов) в памяти.
    *   Если нет — создает новый ряд в памяти (Head Block).
3.  **Запись Sample:**
    *   Значение `100` и текущее время записываются как **Sample** в этот ряд.
4.  **Сброс на диск (Compaction):**
    *   Данные хранятся в оперативной памяти (WAL — Write Ahead Log) для скорости.
    *   Каждые 2 часа (по умолчанию) данные из памяти сбрасываются на диск в виде **Block**.
    *   Блок — это папка на диске, содержащая сжатые чанки (chunks) данных, индекс и метаданные.
    *   Старые блоки могут объединяться (compact) для экономии места.

**Важно про TSDB:**
TSDB оптимизирована для записи (append-only) и чтения по диапазонам времени. Она *неэффективна* для поиска по произвольным лейблам, если их слишком много (высокая кардинальность), потому что индекс растет в размере.
#### Типы метрик:

1. **Counter (Счетчик):**
    - Только растет (или сбрасывается в 0 при перезагрузке).
    - Используется для подсчета событий: количество запросов, количество ошибок, объем переданных байт.
    - _Пример:_ `http_requests_total`.
    - _Нельзя_ использовать для значений, которые могут уменьшаться (температура, свободное место).
2. **Gauge (Измеритель):**
    - Может расти и уменьшаться.
    - Используется для текущего состояния: использование CPU, свободная память, температура, размер очереди.
    - _Пример:_ `node_memory_MemAvailable_bytes`.
3. **Histogram (Гистограмма):**
    - Распределяет наблюдения (например, время ответа запроса) по бакетам (bucket).
    - Позволяет считать квантили (quantiles) и среднее.
    - Создает несколько временных рядов: `_count`, `_sum` и `_bucket{le="..."}`.
    - _Пример:_ `http_request_duration_seconds`.
4. **Summary (Сводка):**
    - Похож на Histogram, но рассчитывает квантили (0.5, 0.9, 0.99) на стороне клиента.
    - Менее гибкий для агрегации между инстансами, чем Histogram.

#### Хранение данных (TSDB)

- Prometheus хранит данные блоками (blocks) на диске.
- Каждый блок содержит данные за 2 часа (по умолчанию).
- Блоки сжимаются и объединяются со временем.
- Данные хранятся в формате, оптимизированном для записи и чтения временных рядов.
- **Retention:** По умолчанию данные хранятся 15 дней.

### PromQL (Prometheus Query Language):

Язык запросов, похожий на SQL, но для **Time Series**.

**Примеры:**

* Вернет все временные ряды с `http_requests_total` метрикой.

	```promql
	http_requests_total
	```

* Фильтрация по лейблам:

	```promql
	http_requests_total{job="api-server", status="500"}
	```


#### Агрегация данных (Aggregation):

В Prometheus данные приходят с каждого отдельного инстанса (сервера). Если у тебя 10 веб-серверов, запрос `http_requests_total` вернет **10 временных рядов** (по одному на каждый сервер):

```
http_requests_total{instance="server1"}  100
http_requests_total{instance="server2"}  200
http_requests_total{instance="server3"}  150
...
```

> **Важно!** Каждая запись это отдельный график.

Часто не важно, сколько запросов на каждом конкретном сервере, а важно **общее количество по всему кластеру** или среднее значение. Для этого используются **агрегирующие операторы**. Структура: 

```
<агрегатор>(выражение) by (<лейбл>)
<агрегатор>(выражение) without (<лейбл>)
```

- **без `by`**: собирает все Time Series для метрики в один единственный результат (скаляр или вектор из одного элемента).
- **`by (label)`**: группирует результаты по указанным меткам (через запятую) и возвращает выборку уникальную для каждой комбинации меток.
- **`without (label)`**: группирует результаты по всем возможным меткам кроме указанных (через запятую) и возвращает выборку уникальную для каждой комбинации меток.

Для разбора будут использоваться следующие входные данные для имитации TSDB записей:

| Время | cpu_usage{instance="server-A", core="1"} | cpu_usage{instance="server-A", core="2"} | cpu_usage{instance="server-B", core="1"} | cpu_usage{instance="server-B", core="2"} |
| ----- | ---------------------------------------- | ---------------------------------------- | ---------------------------------------- | ---------------------------------------- |
| 12:05 | 10                                       | 20                                       | 30                                       | 40                                       |
| 12:10 | 12                                       | 22                                       | 32                                       | 42                                       |
| 12:15 | 14                                       | 24                                       | 34                                       | Данные не пришли                         |

##### Синхронность данных при агрегации:

По умолчанию в Prometheus запись считается **устаревшей (stale)**, если с момента её последней записи прошло больше **5 минут**. Это значение по умолчанию переменной `promql.LookbackDelta`. Когда ты делаешь запрос (например, `sum(cpu_usage)`), Prometheus ищет последнюю доступную точку для каждого временного ряда:
- Если последняя точка была в 11:59:50 (10 сек назад) — **ОК**, используем её.
- Если последняя точка была в 11:58:00 (2 мин назад) — **ОК**, используем её (предполагаем, что значение не изменилось или сервер просто медленно присылает данные).
- Если последняя точка была в 11:54:00 (6 мин назад) — **Устарело!** Prometheus считает, что этот временной ряд «мертв» для данной точки времени, и **исключает его из агрегации**.

Если у тебя было 10 серверов, и один из них перестал слать метрики 6 минут назад, то при запросе `sum(...)` Prometheus сложит значения только **9 серверов**. Сумма станет меньше, хотя физически нагрузка могла не измениться. На графике это выглядит как резкий скачок вниз.

Prometheus спроектирован так, чтобы быть устойчивым к временным проблемам сети или перезагрузке экспортера. Логика такая: «Если я не получаю данные уже 5 минут (при стандартном интервале опроса 15-30 сек), значит, сервис скорее всего упал, а не просто лагает». Значение 5 минут является компромиссом между:

1. Желанием видеть данные даже при кратковременных потерях пакетов.
2. Необходимостью быстро понять, что источник данных исчез.

###### Параметр устаревания можно изменить:

1. **Глобально при запуске Prometheus:**

	```bash
	./prometheus --query.lookback-delta=2m
	```

2. **В самом запросе (функции с диапазоном).** Это более тонкий инструмент. Функции типа `rate()`, `increase()`, `avg_over_time()` сами по себе имеют окно lookback. Например: `rate(metric[5m])`. Здесь Prometheus ищет точки **строго внутри** последних 5 минут все что не попадает в диапазон считается устаревшим.

> Все операторы которые не используют диапазон времени для выборки, берут последние точки для каждого Time series. 

##### sum(metric):

Оператор для суммирования всех последних точек в Time Serie для указанной метрики.

* **Пример запроса без группировки с выводом:**
	
	```promql
	sum(cpu_usage)
	```
	
	```promql
	cpu_usage 84
	```

* **Пример запроса с группировкой по `instance` с выводом:**
	
	```promql
	sum(cpu_usage) by (instance)
	```
	
	```promql
	cpu_usage {instance="server-A"} 38
	cpu_usage {instance="server-B"} 34
	```

* **Пример запроса с группировкой по `core` с выводом:**
	
	```promql
	sum(cpu_usage) by (core)
	```
	
	```promql
	cpu_usage {core="1"} 48
	cpu_usage {core="2"} 24
	```

##### avg(metric):

Оператор для вычисления среднего арифметического всех последних точек в Time Serie для указанной метрики.

* **Пример запроса без группировки с выводом:**
	
	```promql
	avg(cpu_usage)
	```
	
	$(14 + 24 + 34) / 3 = 24$
	
	```promql	
	cpu_usage 24
	```

##### min(metric) и max(metric):

Операторы для вычисления наибольшей и наименьших метрик. Обычно используется для алертов.

##### rate(metric\[time\]):

Функция rate берет все точки (Samples) за указанный период (`[time]`), cтроит линейную регрессию (линию тренда), чтобы сгладить шум и учесть возможные пропуски скрейпинга. Автоматически обрабатывает сбросы (Если значение упало (перезагрузка), Prometheus понимает это и корректно считает скорость роста) и возвращает **среднюю скорость изменения в СЕКУНДУ**.

Функция применяется к каждому TimeSeries отдельно. Функция не «смешивает» данные разных серверов или лейблов между собой. Она работает как конвейер: берет один ряд, считает для него скорость, выдает результат, затем берет следующий ряд и делает то же самое.

* Объяснение принципа работы с аналогиями:
	Представь, у тебя есть счетчик воды в доме (`water_meter_total`). Он показывает **общее количество литров**, которое прошло через трубу с момента установки дома.
	
	- В 10:00 счетчик показывал: `1000` литров.
	- В 10:05 счетчик показывает: `1050` литров.
	
	Тебе не важно, что всего прошло 1050 литров. Тебе важно: **С какой скоростью сейчас течет вода?** (Литров в минуту).
	
	**Проблема:** Если просто вычесть 1050−1000=50, ты получишь объем за 5 минут. А если нужно в секунду? А если счетчик обнулился (заменили счетчик)?
	
	**Функция `rate()`** — это умный спидометр. Она смотрит на значения за последние 5 минут, видит рост на 50 литров, делит на время (300 секунд) и говорит тебе скорость.

* **Пример. Изменения потребления процессора за 5 минут с выводом:**
	
	```promql
	rate(cpu_usage(5m))
	```
	
	```promql
	cpu_usage{instance="server-A", core="1"} 4 // (14 - 10)
	cpu_usage{instance="server-A", core="2"} 4 // (24 - 20)
	cpu_usage{instance="server-B", core="1"} 4 // (34 - 30)
	```

##### increase(metric\[time\]):

Очень схожа с `rate()`, но возвращает не скорость в секунду, а **абсолютный прирост** за период.

- Используется для отчетов: «Сколько ошибок произошло за последние 24 часа?».

 * **Пример. Изменения потребления процессора за 5 минут с выводом:**
	
	```promql
	increase(cpu_usage(10m))
	```
	
	```promql
	cpu_usage{instance="server-A", core="1"} 2 // (14 - 12 / 300 (5 минут))
	cpu_usage{instance="server-A", core="2"} 0,0066 // (24 - 22 / 300 (5 минут))
	cpu_usage{instance="server-B", core="1"} 0,0066 // (34 - 32 / 300 (5 минут))
	```

#### `remote_write` / `remote_read`:

Это специализированное хранилище, которое понимает Prometheus Remote API.

Prometheus использует **специальный протокол** для удаленной записи/чтения:
- **Формат:** HTTP POST/GET запросы.
- **Тело запроса:** Протокол Buffer (protobuf) + сжатие Snappy.
- **Эндпоинты:**
    - Запись: `POST /api/v1/receive`
    - Чтение: `POST /api/v1/read

Системы которые понимают этот протокол:

| Система                    | Тип                          | Описание                                                                                  |
| -------------------------- | ---------------------------- | ----------------------------------------------------------------------------------------- |
| **Thanos Receive**         | Долгосрочное хранение        | Часть экосистемы Thanos, принимает `remote_write`, хранит в объектном хранилище (S3, GCS) |
| **Cortex**                 | Долгосрочное хранение        | Масштабируемое хранилище от Grafana Labs, тоже использует S3                              |
| **VictoriaMetrics**        | TSDB + долгосрочное хранение | Совместима с Prometheus, может быть как "приемником", так и заменой Prometheus            |
| **M3DB**                   | Долгосрочное хранение        | От Uber, распределенная TSDB                                                              |
| **InfluxDB** (с адаптером) | Универсальная TSDB           | Может принимать данные Prometheus через плагин                                            |
#### Relabeling:

В Prometheus есть два этапа, когда можно менять метки (labels).

* **Этап 1. До скрейпинга (Target Relabeling):**
	Происходит **до** того, как Prometheus делает HTTP-запрос к экспортеру. Здесь мы работаем с **мета-лейблами** (те, что начинаются с `__`).
	* `relabel_configs`:
		* **Цель:** Решить, **кого** опрашивать и **как** к нему подключаться.
		- **Что меняет:** Адрес подключения (`__address__`), путь (`__metrics_path__`), схему, или вообще выкидывает таргет из списка.
		- **Когда применять:**
		    - Если нужно изменить IP/порт перед подключением.
		    - Если нужно отфильтровать лишние таргеты из Service Discovery (например, в K8s оставить только поды с аннотацией `prometheus.io/scrape=true`).
		    - Чтобы добавить статические лейблы (например, `env=prod`) ко всем метрикам этого джоба.
- **Этап 2: После скрейпинга (Metric Relabeling):**
	Происходит **после** того, как Prometheus получил текст с метриками, распарсил его, но **перед** записью в базу данных (TSDB).
	* **`metric_relabel_configs`** (в блоке `scrape_configs`)
	    - **Цель:** Изменить, **что** сохранять в базу.
	    - **Что меняет:** Имена метрик, лейблы самих метрик, или выкидывает ненужные метрики.
	    - **Когда применять:**
	        - Чтобы удалить "мусорные" метрики, которые занимают место, но не нужны (снижение кардинальности).
	        - Чтобы переименовать конфликтующие метрики.
	        - Чтобы удалить чувствительные данные из лейблов (например, `user_id`).
	* **`write_relabel_configs`** (Глобальная настройка или в Remote Write)
	    - **Цель:** То же самое, что и `metric_relabel_configs`, но применяется **глобально** ко всем данным, которые уходят из этого экземпляра Prometheus.
	    - **Где используется:**
	        - Чаще всего при использовании **Remote Write** (отправка данных в Thanos, Cortex, VictoriaMetrics).
	        - Позволяет фильтровать данные _перед_ отправкой во внешнее хранилище, экономя трафик и место там.
	    - _Примечание:_ В обычном локальном Prometheus `write_relabel_configs` практически не используется, если ты не пишешь данные куда-то еще через `remote_write`. Для локальной TSDB используй `metric_relabel_configs`.

**Разбор действий (Actions).** В `relabel_configs` и `metric_relabel_configs` используются одни и те же действия:

- **`replace`** (по умолчанию): Заменяет значение целевого лейбла. Используется regex для поиска и `$1`, `$2` для вставки.
- **`keep`**: Оставляет таргет/метрику, **только если** regex совпадает с исходными лейблами. Если не совпал — удаляет. (Используется для фильтрации).
- **`drop`**: Удаляет таргет/метрику, **если** regex совпадает. (Обратное keep).
- **`labelmap`**: Копирует все лейблы, подходящие под regex, в новые лейблы. Удобно, чтобы забрать все мета-лейблы K8s.
- **`labeldrop`**: Удаляет лейблы, имена которых подходят под regex.
- **`labelkeep`**: Оставляет только те лейблы, имена которых подходят под regex. (Опасно, можно случайно удалить `__name__` или `instance`).

##### Meta-label-ы:

Мета-лейблы (meta labels) — это временные метки, которые Prometheus создает автоматически на этапе **Service Discovery (SD)**. Они содержат техническую информацию об источнике данных (IP, порт, имя пода, аннотации, теги облака и т.д.).

Главное правило:

1. Все мета-лейблы начинаются с двойного подчеркивания `__`.
2. Они **не сохраняются** в базу данных (TSDB).
3. Их единственная цель — помочь тебе в `relabel_configs` решить, что делать с таргетом или какие постоянные лейблы ему присвоить.
4. После этапа `relabel_configs` все лейблы, начинающиеся на `__`, удаляются (если ты явно не переименовал их, убрав подчеркивания).

Список мета-лейблов зависит от того, какой тип Service Discovery ты используешь. Вот самые популярные:

**Kubernetes SD (`kubernetes_sd_configs`).**

Это самый богатый источник мета-лейблов. Они зависят от `role` (pod, service, node, endpoint, endpointslice).

Для роли `pod` (самая частая):

- `__meta_kubernetes_namespace`: Неймспейс пода.
- `__meta_kubernetes_pod_name`: Имя пода.
- `__meta_kubernetes_pod_ip`: IP адрес пода.
- `__meta_kubernetes_pod_label_<labelname>`: Значение конкретного лейбла пода.
    - _Пример:_ Если у пода есть лейбл `app=nginx`, появится мета-лейбл `__meta_kubernetes_pod_label_app`.
- `__meta_kubernetes_pod_annotation_<annotationname>`: Значение аннотации.
    - _Пример:_ `__meta_kubernetes_pod_annotation_prometheus_io_scrape`.
- `__meta_kubernetes_pod_container_name`: Имя контейнера.
- `__meta_kubernetes_pod_node_name`: Имя ноды, где запущен под.
- `__address__`: Изначально равен IP пода и порту контейнера (если указан `container_port`).

Для роли `node`:

- `__meta_kubernetes_node_name`: Имя ноды.
- `__meta_kubernetes_node_label_<labelname>`: Лейблы ноды (например, `zone`, `instance-type`).
- `__meta_kubernetes_node_address_<address_type>`: Адреса ноды (InternalIP, ExternalIP, Hostname).

Для роли `service`:

- `__meta_kubernetes_namespace`: Неймспейс сервиса.
- `__meta_kubernetes_service_name`: Имя сервиса.
- `__meta_kubernetes_service_label_<labelname>`: Лейблы сервиса.
- `__meta_kubernetes_service_port_name`: Имя порта.
- `__meta_kubernetes_service_port_number`: Номер порта.

**File SD (`file_sd_configs`):**

Когда ты генерируешь JSON/YAML файлы через Ansible/Terraform.

- `__meta_filepath`: Путь к файлу, из которого был взят таргет.
- _(В самом JSON-файле ты можешь задать любые свои лейблы, они сразу станут обычными лейблами, а не мета, если не начинаются с `__`)_.

**Consul SD (`consul_sd_configs`):**

- `__meta_consul_address`: Адрес сервиса.
- `__meta_consul_dc`: Datacenter.
- `__meta_consul_node`: Имя ноды в Consul.
- `__meta_consul_service`: Имя сервиса.
- `__meta_consul_service_id`: ID сервиса.
- `__meta_consul_tag_<tagname>`: Теги сервиса.
    - _Пример:_ Если тег `production`, будет `__meta_consul_tag_production`.
- `__meta_consul_service_metadata_<key>`: Метаданные сервиса.

**DNS SD (`dns_sd_configs`):**

- `__meta_dns_name`: Имя хоста, полученное из DNS записи.

**EC2 SD (AWS):**

- `__meta_ec2_instance_id`: ID инстанса.
- `__meta_ec2_private_ip`: Приватный IP.
- `__meta_ec2_public_ip`: Публичный IP.
- `__meta_ec2_availability_zone`: AZ (например, `us-east-1a`).
- `__meta_ec2_tag_<tagname>`: Теги AWS.
    - _Пример:_ `__meta_ec2_tag_Name`.
- `__meta_ec2_instance_state`: Состояние (running, stopped).

**Общие мета-лейблы (есть всегда):**

Независимо от типа SD, Prometheus добавляет:

- `__address__`: **Самый важный!** Это целевой адрес (`host:port`), к которому Prometheus сделает HTTP запрос. По умолчанию берется из SD, но его часто меняют через `relabel_configs`.
- `__scheme__`: Протокол (`http` или `https`). По умолчанию `http`.
- `__metrics_path__`: Путь к метрикам. По умолчанию `/metrics`.
- `__param_<name>`: Параметры URL query string.


### Настройка alert-инга:

Настройка состоит из нескольких независимых частей:
1. **Rules (Правила):** Логика «Когда бить тревогу?» (находится в Prometheus).
2. **Alertmanager:** Логика «Кому и как отправлять?» (отдельный сервис).
3. **Интеграции:** Куда слать (Telegram, Slack, Email).

#### Создание правил срабатывания alert-а:

Правила хранятся в YAML-файлах. Обычно их кладут в папку `/etc/prometheus/rules/`.

```yaml
groups:
  - name: example_alerts
    rules:
      # 1. Простой алерт: Сервер недоступен
      - alert: InstanceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
          team: infrastructure
        annotations:
          summary: "Экземпляр {{ $labels.instance }} недоступен"
          description: "Сервер {{ $labels.instance }} не отвечает более 1 минуты. Job: {{ $labels.job }}"

      # 2. Алерт на высокую нагрузку CPU
      - alert: HighCpuUsage
        expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 5m
        labels:
          severity: warning
          team: devops
        annotations:
          summary: "Высокая нагрузка CPU на {{ $labels.instance }}"
          description: "Загрузка CPU превышает 80% уже 5 минут. Текущее значение: {{ $value | humanize }}%"
```

**Разбор полей:**

- **`alert`**: Имя алерта (уникальный идентификатор).
- **`expr`**: PromQL выражение. Если оно возвращает хоть один временной ряд (значение не пустое), условие считается истинным.
- **`for`**: Время ожидания (Pending state).
    - Если `up == 0` сработало только что, алерт переходит в статус **Pending**.
    - Он станет **Firing** (активным) только если условие держится истинным дольше `1m`.
    - _Зачем?_ Чтобы не спамить алертами при кратковременных сетевых лагах (flapping).
- **`labels`**: Метаданные, которые передадутся в Alertmanager.
    - `severity`: Критичность (critical, warning, info). Используется для маршрутизации.
    - `team`: Кто отвечает.
    - _Важно:_ Любые лейблы из метрики (например, `instance`) автоматически добавляются к алерту, если ты не перезапишешь их.
- **`annotations`**: Текст сообщения.
    - `{{ $labels.instance }}`: Подставляет значение лейбла из сработавшей метрики.
    - `{{ $value }}`: Подставляет численное значение выражения (например, `95.5`).
    - `| humanize`: Форматирует число (убирает лишние нули, делает читаемым).


#### Подключение правил к Prometheus:

В файле `prometheus.yml` нужно указать путь к файлам с правилами:

```yaml
global:
  scrape_interval: 15s

# Указываем, где лежат правила
rule_files:
  - "rules/*.yml"  # Можно использовать маски

# Указываем, куда слать сработавшие алерты
alerting:
  alertmanagers:
    - static_configs:
        - targets:
          - "alertmanager:9093" # Адрес твоего Alertmanager
```

> После изменения конфига нужно перезагрузить Prometheus или нажать кнопку "Reload Configuration" в веб-интерфейсе.

#### Настройка Alertmanager-а:

**Установка пакетов:**

```bash
apt-get update && apt-get install alertmanager
```

Alertmanager получает алерты от Prometheus и решает, что с ними делать. По умолчанию он слушает порт **9093**.

1. **Route (Маршрут):** Дерево решений. "Если severity=critical -> звони в телефон. Если warning -> пиши в Slack".
2. **Group By (Группировка):** Схлопывает похожие алерты в одно сообщение.
3. **Inhibit (Подавление):** "Если упал весь дата-центр, не шлли алерты о том, что не работает конкретный сервис внутри него".
4. **Receiver (Получатель):** Конфигурация канала связи (Webhook, Email, PagerDuty).

```yaml
global:
  resolve_timeout: 5m # Через сколько считать алерт решенным, если он пропал

# Маршрутизация по умолчанию
route:
  group_by: ['alertname', 'job'] # Группировать алерты по имени и джобе
  group_wait: 30s  # Ждать 30 сек перед первой отправкой группы (чтобы собрать пачку)
  group_interval: 5m # Ждать 5 мин перед отправкой следующей пачки по той же группе
  repeat_interval: 4h # Повторять уведомление каждые 4 часа, если алерт не закрыт
  
  receiver: 'default-receiver' # Куда слать всё, что не попало в другие правила

  # Дочерние маршруты
  routes:
    - match:
        severity: critical
      receiver: 'telegram-critical'
      continue: false # Не проверять следующие правила, если это совпало
      
    - match:
        severity: warning
      receiver: 'slack-warnings'

# Конфигурация получателей
receivers:
  - name: 'default-receiver'
    email_configs:
      - to: 'admin@example.com'
        from: 'alertmanager@example.com'
        smarthost: 'smtp.example.com:587'
        auth_username: 'alertmanager@example.com'
        auth_password: 'password'

  - name: 'telegram-critical'
    telegram_configs:
      - bot_token: 'token'
        chat_id: 'chat_id'
        parse_mode: 'HTML'
        # Можно кастомизировать сообщение
        message: |
          <b>🔥 CRITICAL ALERT</b>
          Alert: {{ .CommonAnnotations.summary }}
          Description: {{ .CommonAnnotations.description }}
          Instance: {{ .CommonLabels.instance }}

  - name: 'slack-warnings'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXXXXXX'
        channel: '#devops-warnings'
        text: "{{ .CommonAnnotations.summary }}"
```

**Запуск менеджера:**

```bash
./alertmanager --config.file=alertmanager.yml
```

#### Как это работает вместе (Жизненный цикл алерта):

1. **Evaluation:** Prometheus вычисляет `expr` каждые `evaluation_interval` (по умолчанию 15s или 1m).
2. **Pending:** Выражение стало истинным. Запускается таймер `for`.
3. **Firing:** Таймер `for` истек. Prometheus отправляет JSON-пакет с алертом в Alertmanager.
    - Статус алерта в пакете: `firing`.
4. **Grouping in AM:** Alertmanager получает алерт.
    - Он смотрит на `group_by`.
    - Ждет `group_wait` (30s), чтобы собрать другие алерты из этой же группы (например, если упали 3 сервера одновременно, они соберутся в одно сообщение).
5. **Sending:** Alertmanager отправляет сообщение в Receiver (Telegram/Slack).
6. **Resolved:**
    - Выражение в Prometheus стало ложным.
    - Prometheus ждет немного (чтобы убедиться, что это не флуктуация) и отправляет в Alertmanager пакет со статусом `resolved`.
    - Alertmanager помечает алерт как решенный и может отправить уведомление "All clear" (если настроено).

#### Нюансы при работе с alert-ами:

##### Cardinality в алертах:

Не рекомендуется группировать по меткам с высокой кардинальностью (например, `pod_name` или `container_id`).
- _Плохо:_ `group_by: ['pod_name']` (создаст тысячи отдельных уведомлений).
- _Хорошо:_ `group_by: ['job', 'namespace']` (сгруппирует все поды одного сервиса в одно сообщение: "В namespace X упало 5 подов сервиса Y").
##### Inhibition (Подавление):

Если у тебя упал хост (`InstanceDown`), нет смысла получать алерты `HighCpu` или `ServiceDown` для сервисов на этом хосте.

```yaml
inhibit_rules:
  - source_match:
      severity: 'critical'
      alertname: 'InstanceDown'
    target_match:
      severity: 'warning'
    equal: ['instance'] # Подавлять target, если у него такой же instance, как у source
```

Если есть критический алерт `InstanceDown` для `server-1`, то любые warning-алерты для `server-1` будут заглушены.

### Масштабирование и Федерация:

Prometheus масштабируется вертикально (более мощный сервер). Но есть пределы.

1. **Federation:** Один Prometheus может забирать агрегированные данные из других Prometheus. Используется для иерархической структуры (региональные -> центральный).
2. **Thanos / Cortex / M3DB:** Решения для горизонтального масштабирования и долгосрочного хранения. Например: **Thanos** Добавляет sidecar к каждому Prometheus, который выгружает блоки в S3. Позволяет делать глобальные запросы ко всем данным.
