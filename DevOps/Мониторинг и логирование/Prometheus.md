
Prometheus — это система мониторинга и алертинга с открытым исходным кодом. Главное отличие Prometheus от классических систем (как Zabbix или Nagios) — это **Pull-модель** сбора метрик и ориентация на **временные ряды (Time Series)**.

### Архитектура и основные компоненты:

#### Prometheus Server (Ядро):

Это главный демон, который делает три вещи:

1. **Scraping (Сбор данных):** Периодически опрашивает (делает HTTP GET запросы) целевые объекты (targets) по заданным интервалам.
2. **Storage (Хранение):** Сохраняет полученные данные в собственном формате на диске (TSDB — Time Series Database).
3. **HTTP Server:** Предоставляет API для запросов данных (PromQL) и интерфейс для управления.

> **Важно:** Prometheus хранит данные локально. Для долгосрочного хранения (долше нескольких недель/месяцев) обычно используют интеграцию с удаленными хранилищами (Thanos, Cortex, M3DB).

#### Client Libraries (Клиентские библиотеки):

Чтобы приложение могло отдавать метрики, в его код нужно внедрить библиотеку.

- Существуют официальные библиотеки для Go, Java/Scala, Python, Ruby. Для других языков есть сторонние.
- **Как это работает:** Библиотека создает HTTP-эндпоинт (обычно `/metrics`), на котором висит текст в определенном формате. Когда Prometheus делает `GET /metrics`, библиотека генерирует ответ "на лету".

#### Exporters (Экспортеры):

Если ты не можешь изменить код приложения (например, мониторим Nginx, PostgreSQL, сам сервер Linux, Cisco роутер), используются экспортеры. Это отдельные маленькие программы/агенты, которые собирают метрики со сторонней системы и отдают их в формате Prometheus. **Примеры:**
- `node_exporter`: собирает метрики ОС (CPU, RAM, Disk I/O, Network) с Linux/Unix.
- `blackbox_exporter`: проверяет доступность эндпоинтов по HTTP, HTTPS, DNS, TCP, ICMP (аналог ping/check_http).
- `snmp_exporter`: для сетевого оборудования.
- `postgres_exporter`: для БД PostgreSQL.

#### Pushgateway:

Используется для короткоживущих задач (batch jobs, cron scripts), которые не могут быть "опрошены" Prometheus, так как они живут слишком мало.

- Скрипт выполняет работу, отправляет метрики в Pushgateway через HTTP POST.
- Prometheus потом забирает эти метрики из Pushgateway.

> **Внимание:** Не используй Pushgateway для обычных сервисов. Это нарушает принцип Pull-модели и усложняет отказоустойчивость.

#### Alertmanager:

Prometheus сам по себе только собирает данные и вычисляет правила. Когда срабатывает правило алерта, Prometheus отправляет уведомление в **Alertmanager**. Alertmanager занимается:

- **Deduplication:** Если 10 серверов упали одновременно, он схлопнет это в одно уведомление.
- **Grouping:** Группирует алерты (например, все алерты по кластеру БД в одно сообщение).
- **Routing:** Маршрутизирует уведомления куда нужно: в Telegram, Slack, Email, PagerDuty, VictorOps.
- **Silencing/Inhibition:** Возможность заглушить алерты на время обслуживания или подавить вторичные алерты (если упал мастер-нода, не спамить алертами о том, что реплики недоступны).

#### Service Discovery (SD):

Так как targets может быть тысячи и они динамически меняются (особенно в Kubernetes или облаках), вручную прописывать IP в конфиге нельзя. Prometheus поддерживает SD из:
- Kubernetes (автоматически находит поды, сервисы, ноды).
- Consul, Etcd.
- DNS (SRV records).
- Файлов (`file_sd_configs`) — удобно для Ansible/Terraform: ты генерируешь JSON/YAML файл со списком хостов, а Prometheus его перечитывает.

#### Принцип работы (Data Flow):

1. **Конфигурация:** Ты описываешь в `prometheus.yml`, кого опрашивать (`scrape_configs`) и как часто (`scrape_interval`).
2. **Service Discovery:** Prometheus определяет список целей (targets).
3. **Scraping:** Каждый `scrape_interval` Prometheus делает HTTP запрос к каждому target'у по адресу `/metrics`.
4. **Parsing:** Полученный текст парсится. Каждая строка — это метрика.
5. **Storage:** Данные сохраняются в TSDB на диск. Старые данные удаляются или компактизируются согласно настройкам retention.
6. **Querying:** Пользователь (или Grafana) делает запрос через PromQL к API Prometheus.
7. **Alerting:** Rule Engine оценивает правила алертинга на основе данных. Если условие истинно -> отправка в Alertmanager.

### Модель данных (Data Model):

В Prometheus есть два уровня сущностей:

1. **Конфигурационные/Логические** (то, что ты описываешь в `prometheus.yml` и то, как Prometheus видит мир).
2. **Хранимые/Физические** (то, что лежит на диске в TSDB и с чем ты работаешь через PromQL).

#### Иерархия сущностей (От конфигурации к данным):

1. Уровень сбора (Scrape Context):
	1. **Job (Задача)**
	    - **Определение:** Логическая группа однотипных целей. Например, все веб-серверы nginx — это один job, все базы данных postgres — другой.
	    - **Зачем нужно:** Позволяет агрегировать метрики по типу сервиса. В метриках автоматически появляется метка `job="name"`.
	    - _Пример:_ `job_name: "node-exporters"`
	2. **Instance (Экземпляр / Target)**
	    - **Определение:** Конкретный источник метрик внутри Job. Обычно это `IP:Port` или `Hostname:Port`.
	    - **Зачем нужно:** Чтобы различать отдельные серверы. В метриках появляется лейбл `instance="192.168.1.10:9100"`.
	    - _Связь:_ Один Job содержит много Instances.

2. Уровень данных (Data Model):
	1.  **Metric Name (Имя метрики)**
	    *   **Определение:** Строковый идентификатор того, *что* мы измеряем.
	    *   **Правила:** Должно соответствовать regex `[a-zA-Z_:][a-zA-Z0-9_:]*`.
	    *   *Пример:* `http_requests_total`, `node_cpu_seconds_total`.
	
	2.  **Label Set (Набор лейблов)**
	    *   **Определение:** Набор пар `key="value"`, которые описывают измерения метрики.
	    *   **Важно:** Лейблы делают метрику многомерной.
	    *   *Пример:* `{method="POST", handler="/api/login", status="200"}`.
	    *   *Связь:* Имя метрики + Уникальный набор лейблов = **Time Series**.
	
	3.  **Time Series (Временной ряд)**
	    *   **Определение:** Поток данных, идентифицируемый уникальной комбинацией **Имени метрики** и **Набора лейблов**.
	    *   **Суть:** Это «труба», в которую текут данные. Если меняется хоть один лейбл — это уже *другой* временной ряд.
	    *   *Пример:* Ряд `http_requests_total{method="GET"}` и ряд `http_requests_total{method="POST"}` — это два разных Time Series.
	    * **Cardinality (Кардинальность)** — это количество уникальных Time Series. Если у тебя 100 серверов (`instance`) и 1000 методов API (`handler`), то метрика `http_requests_total` создаст $100 \times 1000 = 100,000$ временных рядов. Если добавишь лейбл `user_id` (1 млн пользователей), получишь $100,000 \times 1,000,000 = 10^{11}$ рядов. **Это убьет Prometheus.** Никогда не используй высококардинальные данные в лейблах
	
	4.  **Sample (Сэмпл / Точка данных)**
	    *   **Определение:** Конкретное значение в конкретный момент времени. Минимальная запись в TSDB.
	    *   **Структура:** `(Timestamp, Value)`.
	    *   **Value:** Float64 (число с плавающей точкой).
	    *   **Timestamp:** Unix timestamp в миллисекундах.
	    *   *Связь:* Time Series состоит из упорядоченной последовательности Sample.

3. **Уровень абстракции.** Когда ты сидишь в Grafana или пишешь алерты, ты работаешь с абстракциями более высокого уровня, чем просто Sample.

	1.  **Instant Vector (Мгновенный вектор)**
	    *   **Что это:** Набор Time Series, где у каждого ряда есть ровно одно значение (последний известный Sample) на выбранный момент времени.
	    *   **Где используется:** В большинстве графиков в Grafana, в условиях алертов.
	    *   *Пример запроса:* `up` (показывает статус всех таргетов прямо сейчас).
	
	2.  **Range Vector (Диапазонный вектор)**
	    *   **Что это:** Набор Time Series, где у каждого ряда есть массив значений (Sample) за определенный промежуток времени.
	    *   **Где используется:** Внутри функций агрегации (`rate`, `increase`, `avg_over_time`).
	    *   *Синтаксис:* `metric_name[5m]`.
	    *   *Пример:* `http_requests_total[5m]` вернет все точки за последние 5 минут для каждого ряда.
	
	3.  **Scalar (Скаляр)**
	    *   **Что это:** Простое числовое значение без временного ряда и лейблов.
	    *   **Где используется:** В математических операциях.
	    *   *Пример:* `5`, `pi()`.
	
	4.  **String (Строка)**
	    *   **Что это:** Строковое значение. Используется редко, в основном для отладки или сложных условий `label_replace`.

#### Как сущности создаются в TSDB (Внутренности хранения):

Когда Prometheus скрейпит target, происходит следующее:

1.  **Получение текста:** Prometheus получает текст от экспортера.
    ```text
    http_requests_total{method="GET", status="200"} 100
    http_requests_total{method="POST", status="200"} 50
    ```
2.  **Парсинг и идентификация:**
    *   Парсер разбирает строку.
    *   Проверяет, существует ли уже такой **Time Series** (по хешу имени + лейблов) в памяти.
    *   Если нет — создает новый ряд в памяти (Head Block).
3.  **Запись Sample:**
    *   Значение `100` и текущее время записываются как **Sample** в этот ряд.
4.  **Сброс на диск (Compaction):**
    *   Данные хранятся в оперативной памяти (WAL — Write Ahead Log) для скорости.
    *   Каждые 2 часа (по умолчанию) данные из памяти сбрасываются на диск в виде **Block**.
    *   Блок — это папка на диске, содержащая сжатые чанки (chunks) данных, индекс и метаданные.
    *   Старые блоки могут объединяться (compact) для экономии места.

**Важно про TSDB:**
TSDB оптимизирована для записи (append-only) и чтения по диапазонам времени. Она *неэффективна* для поиска по произвольным лейблам, если их слишком много (высокая кардинальность), потому что индекс растет в размере.
#### Типы метрик:

1. **Counter (Счетчик):**
    - Только растет (или сбрасывается в 0 при перезагрузке).
    - Используется для подсчета событий: количество запросов, количество ошибок, объем переданных байт.
    - _Пример:_ `http_requests_total`.
    - _Нельзя_ использовать для значений, которые могут уменьшаться (температура, свободное место).
2. **Gauge (Измеритель):**
    - Может расти и уменьшаться.
    - Используется для текущего состояния: использование CPU, свободная память, температура, размер очереди.
    - _Пример:_ `node_memory_MemAvailable_bytes`.
3. **Histogram (Гистограмма):**
    - Распределяет наблюдения (например, время ответа запроса) по бакетам (bucket).
    - Позволяет считать квантили (quantiles) и среднее.
    - Создает несколько временных рядов: `_count`, `_sum` и `_bucket{le="..."}`.
    - _Пример:_ `http_request_duration_seconds`.
4. **Summary (Сводка):**
    - Похож на Histogram, но рассчитывает квантили (0.5, 0.9, 0.99) на стороне клиента.
    - Менее гибкий для агрегации между инстансами, чем Histogram.

#### Хранение данных (TSDB)

- Prometheus хранит данные блоками (blocks) на диске.
- Каждый блок содержит данные за 2 часа (по умолчанию).
- Блоки сжимаются и объединяются со временем.
- Данные хранятся в формате, оптимизированном для записи и чтения временных рядов.
- **Retention:** По умолчанию данные хранятся 15 дней.

### PromQL (Prometheus Query Language):

Язык запросов, похожий на SQL, но для **Time Series**.

**Примеры:**

* Вернет все временные ряды с `http_requests_total` метрикой.

	```promql
	http_requests_total
	```

* Фильтрация по лейблам:

	```promql
	http_requests_total{job="api-server", status="500"}
	```


* **rate.** Функция `rate(vector[duration])` делает три вещи:
	1. Берет все точки (Samples) за указанный период (`[5m]`).
	2. Строит линейную регрессию (линию тренда), чтобы сгладить шум и учесть возможные пропуски скрейпинга.
	3. **Автоматически обрабатывает сбросы (resets).** Если значение упало (перезагрузка), Prometheus понимает это и корректно считает скорость роста.
	4. Возвращает **среднюю скорость изменения в секунду**.

	Пример: Допустим, за последние 5 минут счетчик вырос на 3000. `rate(http_requests_total{mode="idle", instance="server1"}[5m])` вернет: 3000/300 секунд=103000/300 секунд=10 запросов в секунду.


#### Агрегация данных (Aggregation):

В Prometheus данные приходят с каждого отдельного инстанса (сервера). Если у тебя 10 веб-серверов, запрос `http_requests_total` вернет **10 временных рядов** (по одному на каждый сервер):

```
http_requests_total{instance="server1"}  100
http_requests_total{instance="server2"}  200
http_requests_total{instance="server3"}  150
...
```

> **Важно!** Каждая запись это отдельный график.

Часто не важно, сколько запросов на каждом конкретном сервере, а важно **общее количество по всему кластеру** или среднее значение. Для этого используются **агрегирующие операторы**. 
##### sum():

Оператор для суммирования всех данных. 


Пример запроса:

```
sum(http_errors)
```

Структура: 

```
<агрегатор>(выражение) by (<лейбл>)
```

- **без `by`**: схлопывает все ряды в один единственный результат (скаляр или вектор из одного элемента).
- **`by (label)`**: группирует результаты по указанному лейблу.




### Масштабирование и Федерация:

Prometheus масштабируется вертикально (более мощный сервер). Но есть пределы.

1. **Federation:** Один Prometheus может забирать агрегированные данные из других Prometheus. Используется для иерархической структуры (региональные -> центральный).
2. **Thanos / Cortex / M3DB:** Решения для горизонтального масштабирования и долгосрочного хранения. Например: **Thanos** Добавляет sidecar к каждому Prometheus, который выгружает блоки в S3. Позволяет делать глобальные запросы ко всем данным.
