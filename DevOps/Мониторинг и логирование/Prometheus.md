
**Prometheus** — это база данных временных рядов (Time Series Database — TSDB), которая собирает метрики (числовые данные с временной меткой) с ваших серверов, приложений и сетей, позволяет делать по ним запросы и настраивать алерты (уведомления), если что-то идет не так.

В отличие от многих старых систем (как Zabbix или Nagios), Prometheus работает по принципу **Pull (Вытягивание)**:

1. **Scraping (Скрейпинг):** Сервер Prometheus сам периодически опрашивает целевые объекты (targets) по HTTP, забирая у них метрики.
2. **Хранение:** Все данные хранятся локально на диске сервера Prometheus в виде временных рядов.
3. **Модель данных:** Данные — это не просто числа, а пары `Метрика + Метки (Labels)`.
    - Пример: `http_requests_total{method="POST", handler="/api", status="200"}`

#### Основные компоненты:

Prometheus — это не один монолит, а набор компонентов, хотя чаще всего под словом "Prometheus" подразумевают основной сервер.

1. **Prometheus Server (Ядро):**
    - **Retrieval (Сборщик):** Модуль, который ходит по адресам и забирает метрики.
    - **TSDB (База данных):** Хранит данные на диске. Данные сжимаются и разбиваются на блоки (chunks). Есть WAL (Write Ahead Log) для защиты от потери данных при краше.
    - **HTTP Server:** Принимает запросы от Grafana или UI.
    - **PromQL Engine:** Движок, который выполняет запросы к данным.
2. **Service Discovery (Обнаружение сервисов):**
    - Prometheus должен знать, кого опрашивать. В статическом конфиге вы пишете IP-адреса.
    - В динамической среде (Kubernetes, AWS EC2, Consul) Prometheus сам спрашивает у API оркестратора: "Какие поды сейчас живы?" и автоматически добавляет их в список целей (targets).
3. **Alertmanager (Менеджер оповещений):**
    - Это **отдельная программа**.
    - Prometheus оценивает правила (например, `if error_rate > 0.1`). Если условие истинно, он отправляет "алерт" в Alertmanager.
    - Alertmanager занимается группировкой, подавлением дублей (silencing) и отправкой в каналы (Slack, Email, Telegram, PagerDuty).
4. **Exporters (Экспортеры):**
    - Прокси-сервисы, которые превращают метрики сторонних систем в формат Prometheus.
5. **Pushgateway (Шлюз для пуша):**
    - Для задач, которые живут недолго (cron-скрипты). Они не могут ждать, пока Prometheus их опросит, поэтому сами отправляют данные в Pushgateway, а Prometheus забирает их оттуда.
#### Типы метрик:

1. Counter (Счетчик):
	- **Суть:** Метрика, которая может только **расти** (увеличиваться) или сбрасываться в ноль (при перезапуске процесса). Она никогда не уменьшается.
	- **Зачем:** Подсчет событий, которые произошли.
	- **Примеры:** Количество запросов, количество ошибок, количество проданных товаров.
	- **Пример данных:**

	```prometheus
	http_requests_total{method="POST", handler="/api"} 1024
	```

	- **Как запрашивать:** Сами по себе числа счетчика малоинформативны (что значит 1024 запроса всего?). Обычно используют функцию `rate()` или `increase()`, чтобы узнать скорость роста за промежуток времени.
	    - `rate(http_requests_total[5m])` — сколько запросов в секунду в среднем за последние 5 минут.

2. Gauge (Датчик / Измеритель):
	- **Суть:** Метрика, которая может **расти и убывать**. Это "моментальный снимок" состояния.
	- **Зачем:** Измерение текущего состояния системы.
	- **Примеры:** Использование оперативной памяти, температура процессора, количество активных пользователей, размер очереди.
	- **Пример данных:**
	
	```prometheus
	node_memory_MemAvailable_bytes 2147483648
	```

	- **Как запрашивать:** Берется текущее значение или функции `max()`, `min()`, `avg()`. Функция `rate()` здесь обычно не применяется.

3. Histogram (Гистограмма):
	- **Суть:** Разбивает наблюдения (например, время ответа) по "корзинам" (buckets). Позволяет понять распределение значений.
	- **Зачем:** Анализ латентности (времени ответа), размеров запросов.
	- **Как работает:** Создает несколько метрик автоматически:
	    1. Счетчики для каждой корзины (`_bucket`): сколько запросов было быстрее определенного порога.
	    2. Сумму всех наблюдений (`_sum`).
	    3. Количество всех наблюдений (`_count`).
	- **Пример данных:**
	
	```prometheus
	http_request_duration_seconds_bucket{le="0.1"} 24054
	http_request_duration_seconds_bucket{le="0.5"} 33444
	http_request_duration_seconds_bucket{le="1.0"} 100392
	http_request_duration_seconds_sum 53423
	http_request_duration_seconds_count 144340
	```

	- **Как запрашивать:** Позволяет вычислять перцентили (например, 95-й перцентиль времени ответа) на стороне сервера с помощью функции `histogram_quantile()`.
	    - `histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))`

4. Summary (Сводка):
	- **Суть:** Похожа на гистограмму, но вычисляет квантили (перцентили) **на стороне клиента** (в приложении), а не на стороне Prometheus.
	- **Зачем:** То же, что и гистограмма, но с меньшими накладными расходами на хранение, однако менее гибкая при агрегации.
	- **Минус:** Нельзя агрегировать несколько Summary в одну (нельзя посчитать средний 95-й перцентиль по всем серверам).
	- **Рекомендация:** В 95% случаев лучше использовать **Histogram**, так как он гибче.

#### Принцип вытягивания метрик с клиента:

Это фундаментальное отличие от систем вроде Zabbix или StatsD (которые часто работают по Push-модели, когда агент сам отправляет данные).

**Процесс скрейпинга (Scraping):**

1. **Конфигурация:** В файле `prometheus.yml` указан `scrape_interval` (например, 15 секунд) и список целей (`targets`).
2. **HTTP Запрос:** Каждые 15 секунд сервер Prometheus отправляет обычный **HTTP GET запрос** по адресу цели (например, `http://192.168.1.5:9100/metrics`).
3. **Ответ:** Цель возвращает текст в специальном формате (Prometheus Exposition Format).
4. **Парсинг:** Prometheus разбирает текст, добавляет к метрикам системные лейблы (например, `instance="192.168.1.5:9100"`, `job="node"`) и текущую временную метку.
5. **Запись:** Данные записываются в TSDB.

**Почему Pull, а не Push?**

- **Контроль:** Вы точно знаете, когда и как часто собираются данные.
- **Надежность:** Если сеть между приложением и мониторингом пропадет, приложение продолжит работать. Когда сеть восстановится, Prometheus просто придет за новыми данными. (При Push-модели приложение могло бы потерять метрики, если некуда их отправить, так как клиент мог не отправлять повторно данные при возобновлении сети).
- **Безопасность:** Легче настроить фаервол, разрешив входящие соединения на сервер мониторинга, чем открывать доступ из всех приложений наружу.

Не все приложения "родные" для Prometheus. Старая база данных (PostgreSQL), железо сервера или сетевое оборудование не умеют отдавать метрики по HTTP в формате Prometheus. Для этого нужен **Экспортер**:

- **С одной стороны:** Он общается с системой на её языке (читает `/proc` в Linux, делает SQL-запрос к БД, опрашивает SNMP устройство).
- **С другой стороны:** Он открывает HTTP-порт (обычно 9100+) и отдает данные в формате Prometheus, когда приходит сервер мониторинга.

**Типы интеграции:**

1. **Через Экспортер (Sidecar / DaemonSet):**
    - _Пример:_ **Node Exporter**.
    - Вы ставите его на сервер. Он читает нагрузку CPU, диск, сеть.
    - Prometheus стучится в Node Exporter.
    - _Плюс:_ Не нужно менять код приложения.
    - _Минус:_ Нужно устанавливать и поддерживать дополнительное ПО.
2. **Прямая инструментация (Client Libraries):**
    - _Пример:_ Ваше приложение на Go/Python/Java.
    - Вы используете официальную библиотеку Prometheus в коде.
    - В коде вы создаете метрику: `requests_counter.inc()`.
    - Приложение само открывает порт (например, `/metrics`), куда Prometheus приходит за данными.
    - _Плюс:_ Максимальная детализация (бизнес-метрики).
    - _Минус:_ Нужно менять код приложения.
3. **Blackbox Exporter:**
    - Это особый случай. Он не стоит на целевом сервере.
    - Prometheus просит Blackbox: "Проверь, доступен ли google.com по HTTPS".
    - Blackbox сам делает запрос, замеряет время и возвращает метрику: `probe_success 1` или `0`.

### Основные сущности:

#### Label:

В Prometheus лейблы - это теги которые позволяют:

1. **Фильтровать:** Показать метрики только для конкретного сервера (`instance="web-01"`).
2. **Группировать:** Посчитать сумму по всем серверам, но разбить по городам.
3. **Сравнивать:** Сравнить нагрузку на `env="prod"` и `env="test"`.

**Главное правило:** Лейблы должны иметь **ограниченный набор значений**.

- ✅ Хорошо: `status="200"`, `method="GET"`, `env="prod"` (значений мало).
- ❌ Плохо: `user_id="12345"`, `email="user@example.com"` (значений миллионы — это "убьет" базу данных).


### Настройка prometheus:

```yml
# =============================================================================
# 1. ГЛОБАЛЬНЫЕ НАСТРОЙКИ (Global)
# =============================================================================
global:
  # Как часто опрашивать цели (targets). 
  # Можно переопределить для конкретного job в scrape_configs.
  scrape_interval: 15s

  # Как часто Prometheus проверяет правила: "А не пора ли сработать алерту?".
  # Обычно совпадает со scrape_interval или кратен ему.
  evaluation_interval: 15s

  # Метки, которые добавляются ко ВСЕМ метрикам и алертам данного prometheus экземпляра. 
  # Нужно чтобы отличать источники при просмотре логов. Если есть 10 кластеров и все отправляется в одно хранилище (`remote_write`), эти метки помогут понять: "Эта метрика пришла из продакшена в Европе, а эта — из теста в Азии".
  external_labels:
    monitor: 'prod-cluster-01'
    region: 'eu-west-1'

  # Сколько секунд ждать ответа от сервера, прежде чем сказать "Он не отвечает, идем дальше". Если цель не ответила за это время — ошибка. 
  # Важно! Должен быть меньше, чем `scrape_interval`. Иначе Prometheus "зависнет", ожидая одного медленного сервера.
  scrape_timeout: 10s

# =============================================================================
# 2. ФАЙЛЫ С ПРАВИЛАМИ (Rule Files)
# =============================================================================
# Говорит Prometheus, где лежат файлы с инструкциями: "Когда кричать" (алерты) и "Как заранее посчитать сложные цифры" (recording rules).
rule_files:
  - "rules/alerts/*.yml"
  - "rules/recording/*.yml"

# =============================================================================
# 3. НАСТРОЙКИ АЛЕРТИНГА (Alerting)
# =============================================================================
alerting:
  alertmanagers:
    - static_configs:
        # Говорит Prometheus, КОМУ отправлять сигнал тревоги, когда сработало правило. Можно указать несколько для HA.
        - targets:
            - alertmanager-1:9093
            - alertmanager-2:9093
      # Протокол и таймауты для обращения к alert-менеджеру
      scheme: http
      timeout: 10s
      # Версия HTTP (v2 используется чаще всего)
      api_version: v2

# =============================================================================
# 4. НАСТРОЙКИ ХРАНЕНИЯ (Storage)
# =============================================================================
storage:
  tsdb:
    # Путь к данным (по умолчанию ./data)
    path: /prometheus/data
    # Сколько хранить данные, прежде чем удалить старые.
    # Для долгосрочного хранения используют remote_write + Thanos/Cortex.
    retention:
      time: 15d
      # Или по размеру:
      size: 50GB
      
    # Как группировать данные на диске. Можно не менять, значения по умолчанию обычно хороши. Вы не кидаете каждую метрику в общую кучу. Вы собираете метрики за 2 часа в один большой контейнер (`блок`), запечатываете его и подписываете "12:00-14:00". Так легче искать и архивировать.
    # Минимальное время блока (advanced tuning)
    min_block_duration: 2h
    # Максимальное время блока
    max_block_duration: 2h

# =============================================================================
# 5. УДАЛЕННОЕ ЧТЕНИЕ/ЗАПИСЬ (Remote Read/Write)
# =============================================================================
# Используется для долгосрочного хранения или глобального просмотра.
remote_write:
	# Отправляй копию всех метрик вот по этому адресу
  - url: "http://thanos-receive:19291/api/v1/receive"
    # Сколько ждать, пока центральный офис подтвердит, что принял данные.
    remote_timeout: 30s
    
    # Очередь записи (tuning performance)
    queue_config:
      capacity: 10000
      max_shards: 50
      max_samples_per_send: 5000
      

    # Фильтрует, ЧТО отправлять в удаленное хранилище.
    write_relabel_configs:
      # Пример: не отправлять метрики с высоким кардиналом в удаленку
      - source_labels: [__name__]
        regex: 'http_request_duration_seconds_bucket'
        action: drop

remote_read:
  - url: "http://thanos-query:10902/api/v1/read"
    read_recent: true # Читать свежие данные из локального TSDB, старые отсюда

# =============================================================================
# 6. КОНФИГУРАЦИЯ СБОРА ДАННЫХ (Scrape Configs)
# =============================================================================
scrape_configs:

  # --- 6.1. Мониторинг самого Prometheus ---
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']
    # Добавляем метку, кто это
    metric_relabel_configs:
      - target_label: 'service'
        replacement: 'prometheus'

  # --- 6.2. Node Exporter (Статический список) ---
  - job_name: 'node-exporter'
    scrape_interval: 30s # Переопределяем глобальный интервал
    static_configs:
      - targets: 
          - '192.168.1.10:9100'
          - '192.168.1.11:9100'
        # Лейблы для этой группы хостов
        labels:
          env: 'production'
          team: 'infra'
    # Relabeling до скрейпинга (манипуляция целями)
    relabel_configs:
      # Пример: если метка env != production, не скрейпить
      - source_labels: [env]
        regex: 'production'
        action: keep

  # --- 6.3. Kubernetes Service Discovery (Динамический) ---
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod
        # В каком кластере искать (если конфиг внутри кластера, не нужно)
        # api_server: 'https://k8s-api:6443'
        # tls_config: ...
    
    # Фильтруем только поды с аннотацией prometheus.io/scrape: "true"
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true
      
      # Берем порт из аннотации, если есть
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        target_label: __address__
        regex: (.+)
        replacement: ${1}
      
      # Добавляем имя пода как лейбл
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
      
      # Добавляем неймспейс
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace

  # --- 6.4. File Service Discovery (Для динамических IP вне K8s) ---
  # Скрипт снаружи генерирует файл targets.json, Prometheus его подхватывает
  - job_name: 'file-sd-targets'
    file_sd_configs:
      - files:
          - '/etc/prometheus/targets/*.json'
        refresh_interval: 5s
    metric_relabel_configs:
      # Пример: переименовать метрику на лету
      - source_labels: [__name__]
        regex: 'old_metric_name'
        target_label: __name__
        replacement: 'new_metric_name'

  # --- 6.5. Blackbox Exporter (Проверка доступности) ---
  - job_name: 'blackbox-http'
    metrics_path: /probe
    params:
      module: [http_2xx] # Какой модуль blackbox использовать
    static_configs:
      - targets:
          - https://google.com
          - https://mysite.com
    relabel_configs:
      # Адрес цели передаем в параметр target для blackbox
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      # Реальный адрес blackbox exporter
      - target_label: __address__
        replacement: blackbox-exporter:9115
```

#### Разбор ключевых параметров:

**Глобальные (`global`)**

- `scrape_interval`: Базовая частота опроса. **Важно:** Не ставьте слишком часто (например, 1с) без нужды. Это создает нагрузку на сеть и целевые приложения. 15с–30с — золотой стандарт.
- `external_labels`: Добавляются ко всем метрикам **перед** сохранением. Если вы используете федерацию или `remote_write` в центральное хранилище, эти метки позволяют понять, от какого кластера пришли данные.
