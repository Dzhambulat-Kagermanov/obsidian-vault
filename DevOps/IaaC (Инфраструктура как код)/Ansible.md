### Основы:

**Ansible** — это инструмент автоматизации IT-инфраструктуры с открытым исходным кодом, предназначенный для управления конфигурациями, развертывания приложений (деплоя) и оркестрации задач. Он позволяет управлять сотнями серверов через единый сценарий, используя SSH для подключения без установки агентов на целевые машины

#### Ключевые особенности:

1.  **Безагентная архитектура (Agentless).** Вам не нужно устанавливать специальное ПО (агенты) на серверы, которыми вы управляете. Ansible подключается к ним по стандартным протоколам: **SSH** (для Linux/Unix) или **WinRM** (для Windows);
2.  **Идемпотентность (Idempotency).** Это значит, что вы можете запускать один и тот же сценарий много раз, и результат будет одинаковым. *Пример:* Если в плейбуке написано "установить Nginx", Ansible проверит: если он уже установлен и настроен верно, ничего делать не будет. Если нет — установит. Это предотвращает поломки при повторных запусках;
3.  **Push-модель.** Управляющий сервер сам "толкает" конфигурацию на целевые узлы (в отличие от Puppet/Chef, где узлы сами "тянут" конфигурацию);
4. **Простота (YAML):** Сценарии (плейбуки) пишутся на понятном языке YAML, описывая желаемое состояние системы.

#### Основные понятия: 

*   **Control Node (Управляющий узел)** — машина, на которой установлен Ansible и с которой запускаются команды;
*   **Managed Nodes (Управляемые узлы)** — серверы, которыми управляет Ansible;
*   **Inventory (Инвентарь)** — файл (обычно `hosts`), в котором перечислены IP-адреса или доменные имена управляемых серверов, сгруппированные по ролям (например, `webservers`, `dbservers`);
*   **Playbook (Плейбук)** — YAML-файл, описывающий сценарий автоматизации (что и в каком порядке делать);
*   **Task (Задача)** — отдельное действие внутри плейбука (например, "скопировать файл", "перезапустить сервис");
*   **Module (Модуль)** — готовый скрипт, который выполняет задачу (например, `apt`, `yum`, `copy`, `service`, `docker`). В Ansible сотни встроенных модулей;
*   **Role (Роль)** — структурированный набор плейбуков, переменных и файлов для переиспользования. Позволяет не писать всё с нуля, а брать готовые роли;
*   **Ansible Galaxy** — публичный репозиторий, где можно скачивать готовые роли, написанные сообществом;
* **Факты (facts)** — это информация о целевых узлах (серверах), автоматически собираемая модулем `setup` перед выполнением задач плейбука. Они включают данные об ОС, IP-адресах, дисках, памяти и процессоре. Собранные факты доступны в плейбуках как переменные `ansible_facts`, позволяя адаптировать настройки под конкретный хост. Для ускорения работы плейбуков, если факты не нужны, сбор можно отключить с помощью.

#### Плюсы и Минусы

**Плюсы:**
*   **Низкий порог входа:** YAML понятен даже новичкам, не нужно учить сложный код.
*   **Не требует агентов:** Экономит ресурсы на серверах и упрощает безопасность.
*   **Огромное сообщество:** Тысячи готовых модулей и ролей.
*   **Гибкость:** Можно управлять не только серверами, но и сетевым оборудованием (Cisco, Juniper), облаками (AWS, Azure) и контейнерами.

**Минусы:**
*   **Скорость:** Так как используется SSH, при управлении тысячами серверов одновременно это может быть медленнее, чем у агентских решений.
*   **Отсутствие состояния (Stateless):** Ansible не хранит базу данных о том, в каком состоянии находятся серверы (в отличие от Puppet). Он полагается на факты, собранные в момент запуска.
*   **Windows:** Поддержка есть, но исторически она была менее стабильной и удобной, чем для Linux (хотя сейчас ситуация намного лучше).

#### Для чего используют Ansible:

1.  **Configuration Management:** Настройка ОС, установка пакетов, управление пользователями, правками конфигов;
2.  **Application Deployment:** Развертывание кода приложений на продакшн;
3.  **Orchestration:** Координация сложных процессов (например, сначала обновить базу данных, потом бэкап, потом приложение);
4.  **Provisioning:** Создание серверов в облаке (хотя для этого чаще используют Terraform, Ansible тоже умеет);
5.  **Security & Compliance:** Проверка серверов на соответствие стандартам безопасности (например, все ли закрыты порты, обновлен ли SSL).

#### Ansible Core vs Ansible Automation Platform:

*   **Ansible Core:** Бесплатная версия с открытым кодом. Это то, что используют 90% инженеров;
*   **Ansible Automation Platform (ранее Ansible Tower):** Платная корпоративная версия от Red Hat. Добавляет веб-интерфейс, управление правами доступа, расписание задач, логи и поддержку.



### Настройка глобального файла конфигурации:

Ansible управляется через конфигурационный файл. В нем задаются параметры по умолчанию, которые действуют для всех запусков, если не переопределены иначе.

Ansible ищет конфигурационный файл в нескольких местах. **Первый найденный файл используется, остальные игнорируются.**

Порядок приоритета (от высшего к низшему):

1. **Переменная окружения `ANSIBLE_CONFIG`**: Если вы задали `export ANSIBLE_CONFIG=/my/custom.cfg`, Ansible будет использовать именно этот файл, независимо от того, где вы находитесь;
2. **Локальный файл `./ansible.cfg`**: Файл в текущей директории, откуда запущена команда. Это **лучшая практика** для проектов (хранить конфиг в репозитории с кодом);
3. **Домашний файл `~/.ansible.cfg`**: Личные настройки пользователя;
4. **Глобальный файл `/etc/ansible/ansible.cfg`**: Системные настройки по умолчанию (устанавливаются при установке пакета). **Редактировать его не рекомендуется.*

#### Конфигурация ansible.cfg:

Файл `ansible.cfg` имеет формат **INI** и состоит из секций (в квадратных скобках) и параметров внутри них. Хотя в Ansible сотни настроек, на практике активно используется около 20-30.

Почти любую настройку из `ansible.cfg` можно переопределить переменной окружения без редактирования файлов. Имя переменной обычно совпадает с параметром, но в верхнем регистре с префиксом `ANSIBLE_`. _Примеры:_

- В конфиге: `remote_user = admin`
- Переменная: `export ANSIBLE_REMOTE_USER=root`
- В конфиге: `inventory = ./hosts`
- Переменная: `export ANSIBLE_INVENTORY=/etc/ansible/prod_hosts`

Чтобы увидеть все активные настройки и их источники, используйте:

```bash
ansible-config view
# Или более подробно:
ansible-config dump
```

##### Секция `[defaults]`

Это основная секция, где настраивается поведение Ansible по умолчанию.

###### Пулы и файлы

- **`inventory`**: Путь к файлу инвентаря (хостов).
    - _Пример:_ `inventory = ./hosts` или `inventory = /etc/ansible/hosts`
    - _Зачем:_ Чтобы не указывать `-i` каждый раз при запуске.
- **`log_path`**: Путь к файлу логирования.
    - _Пример:_ `log_path = /var/log/ansible.log`
    - _Зачем:_ Для аудита и отладки (кто, когда и что запускал).
- **`roles_path`**: Путь к поиску ролей.
    - _Пример:_ `roles_path = ./roles:/usr/share/ansible/roles`
    - _Зачем:_ Если вы храните роли не в стандартном месте.
- **`collections_paths`**: Путь к коллекциям (актуально для Ansible 2.9+).
    - _Пример:_ `collections_paths = ./collections`
- **`vault_password_file`**: Путь к файлу с паролем для Ansible Vault.
    - _Пример:_ `vault_password_file = ~/.vault_pass.txt`
    - _Зачем:_ Автоматическая расшифровка зашифрованных файлов без ввода пароля вручную.
- **`retry_files_enabled`**: Включить создание `.retry` файлов при ошибке плейбука.
    - _Значение:_ `True` / `False`
    - _Рекомендация:_ Лучше `False`, чтобы не засорять директорию (сейчас считается устаревшей практикой).

###### Производительность и соединение:

- **`forks`**: Количество параллельных процессов.
    - _Значение:_ `5` (по умолчанию), `20`, `50`...
    - _Зачем:_ Увеличивает скорость работы на большом количестве хостов. Не ставьте слишком много, чтобы не перегрузить контроллер.
- **`timeout`**: Таймаут соединения с хостом (в секундах).
    - _Значение:_ `10` (по умолчанию).
    - _Зачем:_ Если сеть медленная, увеличьте до 30-60.
- **`poll_interval`**: Интервал опроса асинхронных задач.
    - _Значение:_ `15`
- **`gathering`**: Политика сбора фактов.
    - _Значение:_ `implicit` (стандарт), `explicit` (только если вызван модуль setup), `smart` (кэширование).
- **`fact_caching`**: Тип кэширования фактов (например, `jsonfile`, `redis`, `memcached`).
    - _Зачем:_ Ускоряет работу, если не нужно собирать факты каждый раз.
- **`fact_caching_connection`**: Путь или строка подключения для кэша (например, путь к папке для `jsonfile`).
- **`fact_caching_timeout`**: Время жизни кэша в секундах.

###### Вывод и интерфейс:

- **`stdout_callback`**: Формат вывода результатов в консоль.
    - _Значения:_ `default`, `yaml`, `json`, `debug`, `unixy`.
    - _Зачем:_ `yaml` делает вывод более читаемым. `json` удобен для парсинга скриптами.
- **`bin_ansible_callbacks`**: Включать ли кастомные плагины callback из директории `callback_plugins`.
    - _Значение:_ `True` / `False`
- **`display_skipped_hosts`**: Показывать ли хосты, где задачи были пропущены.
    - _Значение:_ `True` / `False`
- **`display_args_to_stdout`**: Показывать аргументы задач в выводе (может раскрыть секреты!).
    - _Значение:_ `False` (рекомендуется для безопасности).
- **`deprecation_warnings`**: Показывать предупреждения об устаревших функциях.
    - _Значение:_ `True` / `False`

###### Безопасность:

- **`host_key_checking`**: Проверка ключей SSH сервера.
    - _Значение:_ `True` / `False`
    - _Внимание:_ **Никогда не ставьте `False` в продакшене.** Это открывает путь для MITM-атак. Отключайте только во временных тестовых стендах.
- **`remote_user`**: Пользователь по умолчанию для подключения.
    - _Пример:_ `remote_user = ansible`
- **`private_key_file`**: Путь к приватному SSH ключу.
    - _Пример:_ `private_key_file = ~/.ssh/id_rsa_ansible`

##### Секция `[privilege_escalation]`:

Настройки для повышения прав (sudo, su и т.д.).

- **`become`**: Включить повышение прав по умолчанию.
    - _Значение:_ `True` / `False`
    - _Зачем:_ Чтобы не писать `become: yes` в каждом плейбуке.
- **`become_method`**: Метод повышения прав.
    - _Значения:_ `sudo` (стандарт), `su`, `pbrun`, `pfexec`.
- **`become_user`**: Пользователь, от имени которого выполняются задачи.
    - _Значение:_ `root` (стандарт).
- **`become_ask_pass`**: Запрашивать ли пароль для sudo.
    - _Значение:_ `True` / `False`
    - _Зачем:_ Если у пользователя настроено `sudo` без пароля (NOPASSWD), ставьте `False`.

##### Секция `[ssh_connection]`:

Настройки конкретного транспортного уровня (SSH):

- **`pipelining`**: Включение канализации (pipelining) для SSH.
    - _Значение:_ `True` / `False`
    - _Зачем:_ **Сильно ускоряет работу.** Вместо открытия нового SSH-соединения для каждого модуля, команды передаются через одно соединение.
    - _Требование:_ На удаленных серверах в `/etc/ssh/sshd_config` должно быть `Requiretty no` (обычно по умолчанию выключено).
- **`control_path`**: Путь к сокету управления соединением (Connection Multiplexing).
    - _Пример:_ `control_path = %(directory)s/%%h-%%p-%%r`
    - _Зачем:_ Позволяет переиспользовать одно SSH-соединение для нескольких задач, ускоряя работу.
- **`ssh_args`**: Дополнительные аргументы для команды ssh.
    - _Пример:_ `ssh_args = -o ControlMaster=auto -o ControlPersist=60s`
- **`retries`**: Количество попыток переподключения при ошибке SSH.
    - _Значение:_ `3` (по умолчанию).

##### Секция `[galaxy]`:

Настройки для работы с `ansible-galaxy` (скачивание ролей и коллекций).

- **`server_list`**: Список серверов Galaxy (например, `release_galaxy`, `my_private_gallery`).
- **`ignore_certs`**: Игнорировать проверку SSL сертификатов при скачивании.
    - _Значение:_ `True` / `False`
    - _Внимание:_ Опасно для безопасности, используйте только во внутренней сети с самоподписанными сертификатами.
- **`role_skeleton`**: Шаблон для создания новой роли через `ansible-galaxy init`.

##### Секция `[diff]`:

Настройки отображения различий (что изменилось в файлах).

- **`always`**: Показывать дифф всегда, даже без флага `--diff`.
    - _Значение:_ `True` / `False`
    - _Зачем:_ Удобно при отладке, но может засорить вывод логами.
- **`context`**: Количество строк контекста вокруг изменения.
    - _Значение:_ `3` (по умолчанию).

##### Секция `[colors]`:

Кастомизация цветов вывода в терминале.

- **`highlight`**: Цвет для подсветки изменений.
- **`verbose`**: Цвет для режима подробного вывода.
- **`warn`**: Цвет для предупреждений.
    - _Пример:_ `warn = bright purple`

### Настройка Inventory файла:

Инвентарь (Inventory) в Ansible — это файл или набор файлов, в которых описываются управляемые хосты, их группировка и переменные. Ansible поддерживает два основных формата: **INI** (традиционный) и **YAML** (современный).

#### INI Format (Традиционный):

Это классический формат, похожий на конфигурационные файлы Windows. Он компактен и привычен для многих администраторов.

```ini
# =============================================================================
# Ansible Inventory - INI Format
# =============================================================================

# -----------------------------------------------------------------------------
# Глобальные переменные (применяются ко всем хостам)
# -----------------------------------------------------------------------------
[all:vars]
ansible_user = ansible
ansible_ssh_private_key_file = ~/.ssh/id_rsa_ansible
ansible_python_interpreter = /usr/bin/python3
ansible_become = yes
ansible_become_method = sudo

# -----------------------------------------------------------------------------
# Группы серверов
# -----------------------------------------------------------------------------

# Веб-серверы (Production)
[webservers]
web01.example.com ansible_host=192.168.1.10
web02.example.com ansible_host=192.168.1.11
web03.example.com ansible_host=192.168.1.12

  
[racks]
# Буквенный диапазон (например, для стоек)
rack[a:c].datacenter.local

[redis]
# Редис сервервы от 1 до 3
192.168.176.[1:3]

# Веб-серверы от 01 до 05 (с ведущим нулем)

[backend]
# Важно: ansible_host нельзя задать диапазоном для каждого хоста индивидуально
# таким образом. Если IP разные, лучше перечислить явно или использовать
# групповые переменные + скрипт.
backend[01:05].example.com ansible_host=192.168.1.10


# Веб-серверы (Staging)
[webservers_staging]
web-stage01.example.com ansible_host=10.0.1.10
web-stage02.example.com ansible_host=10.0.1.11

# Базы данных
[dbservers]
db01.example.com ansible_host=192.168.1.20 db_role=master
db02.example.com ansible_host=192.168.1.21 db_role=replica
db03.example.com ansible_host=192.168.1.22 db_role=replica

# Балансировщики нагрузки
[lbservers]
lb01.example.com ansible_host=192.168.1.5
lb02.example.com ansible_host=192.168.1.6

# Мониторинг
[monitoring]
monitor01.example.com ansible_host=192.168.1.30

# -----------------------------------------------------------------------------
# Вложенные группы (Группы групп)
# -----------------------------------------------------------------------------

# Все производственные серверы
[production:children]
webservers
dbservers
lbservers
monitoring

# Все веб-серверы (prod + staging)
[all_webservers:children]
webservers
webservers_staging

# -----------------------------------------------------------------------------
# Переменные для групп
# -----------------------------------------------------------------------------

[webservers:vars]
http_port = 80
https_port = 443
nginx_version = 1.24.0
max_connections = 1000

[dbservers:vars]
db_port = 5432
db_engine = postgresql
db_backup_enabled = true

[production:vars]
environment = production
monitoring_enabled = true
log_level = warning

[webservers_staging:vars]
environment = staging
monitoring_enabled = false
log_level = debug

# -----------------------------------------------------------------------------
# Хосты с индивидуальными переменными
# -----------------------------------------------------------------------------

[special:vars]
# Хост с нестандартным портом SSH
bastion.example.com ansible_host=203.0.113.1 ansible_port=2222 ansible_user=admin

# Хост с другим интерпретатором Python (например, старый CentOS)
legacy01.example.com ansible_host=192.168.1.100 ansible_python_interpreter=/usr/bin/python2
```

**Ключевые элементы INI формата:**

| Часть строки                | Значение                        | Тип           |
| --------------------------- | ------------------------------- | ------------- |
| `db03.example.com`          | **Имя хоста в инвентаре**       | Идентификатор |
| `ansible_host=192.168.1.22` | **Переменная подключения**      | Переменная    |
| `db_role=replica`           | **Пользовательская переменная** | Переменная    |
1. **`db03.example.com`**
    - Это **логическое имя** хоста, которое Ansible будет использовать внутри плейбуков.
    - По этому имени вы будете обращаться к серверу в командах: `ansible db03.example.com -m ping`.
    - Оно также используется для сопоставления с файлами в папке `host_vars/db03.example.com.yml`.
2. **`ansible_host=192.168.1.22`**
    - Это специальная зарезервированная переменная.
    - Она говорит Ansible: _"Когда будешь подключаться к хосту `db03.example.com`, реально стучись по IP-адресу `192.168.1.22`"_.
    - **Зачем это нужно?** Часто DNS-имя (`db03.example.com`) может не резолвиться (не превращаться в IP) на машине, где запущен Ansible, или же вы хотите управлять сервером по внутреннему IP, пока в DNS прописан внешний. Если этой переменной нет, Ansible попытается найти IP сам через DNS, используя имя `db03.example.com`.
3. **`db_role=replica`**
    - Это **пользовательская переменная**, которую вы придумали сами.
    - Ansible сохранит её в память для этого хоста со значением `replica`.
    - Вы сможете использовать её в плейбуках для логики.

|Синтаксис|Описание|
|---|---|
|`[groupname]`|Объявление группы хостов|
|`[groupname:children]`|Группа, содержащая другие группы (вложенность)|
|`[groupname:vars]`|Переменные, применяемые ко всей группе|
|`[all:vars]`|Глобальные переменные для всех хостов|
|`key=value`|Переменные хоста, указанные прямо в строке|

#### YAML Format (Современный):

Формат YAML более структурирован, удобен для версионирования и позволяет использовать сложные конструкции (списки, словари). Файл обычно называется `hosts.yml` или `inventory.yml`. В YAML формате нет возможности указывания диапазонов серверов как в INI (db\[01:02\].example.local)

```yaml
---
# =============================================================================
# Ansible Inventory - YAML Format
# =============================================================================

all:
  # Глобальные переменные для всех хостов
  vars:
    ansible_user: ansible
    ansible_ssh_private_key_file: ~/.ssh/id_rsa_ansible
    ansible_python_interpreter: /usr/bin/python3
    ansible_become: true
    ansible_become_method: sudo
    ntp_servers:
      - 0.pool.ntp.org
      - 1.pool.ntp.org

  # Дочерние группы
  children:
    
    # -----------------------------------------------------------------------
    # Веб-серверы (Production)
    # -----------------------------------------------------------------------
    webservers:
      hosts:
        web01.example.com:
          ansible_host: 192.168.1.10
          server_id: 1
          datacenter: dc1
        web02.example.com:
          ansible_host: 192.168.1.11
          server_id: 2
          datacenter: dc1
        web03.example.com:
          ansible_host: 192.168.1.12
          server_id: 3
          datacenter: dc2
      vars:
        http_port: 80
        https_port: 443
        nginx_version: 1.24.0
        max_connections: 1000
        environment: production

    # -----------------------------------------------------------------------
    # Веб-серверы (Staging)
    # -----------------------------------------------------------------------
    webservers_staging:
      hosts:
        web-stage01.example.com:
          ansible_host: 10.0.1.10
          server_id: 101
        web-stage02.example.com:
          ansible_host: 10.0.1.11
          server_id: 102
      vars:
        environment: staging
        monitoring_enabled: false
        log_level: debug

    # -----------------------------------------------------------------------
    # Базы данных
    # -----------------------------------------------------------------------
    dbservers:
      hosts:
        db01.example.com:
          ansible_host: 192.168.1.20
          db_role: master
          db_priority: 1
        db02.example.com:
          ansible_host: 192.168.1.21
          db_role: replica
          db_priority: 2
        db03.example.com:
          ansible_host: 192.168.1.22
          db_role: replica
          db_priority: 3
      vars:
        db_port: 5432
        db_engine: postgresql
        db_backup_enabled: true
        db_backup_time: "02:00"

    # -----------------------------------------------------------------------
    # Балансировщики нагрузки
    # -----------------------------------------------------------------------
    lbservers:
      hosts:
        lb01.example.com:
          ansible_host: 192.168.1.5
          lb_algorithm: round_robin
        lb02.example.com:
          ansible_host: 192.168.1.6
          lb_algorithm: least_conn
      vars:
        haproxy_version: 2.8

    # -----------------------------------------------------------------------
    # Мониторинг
    # -----------------------------------------------------------------------
    monitoring:
      hosts:
        monitor01.example.com:
          ansible_host: 192.168.1.30
          prometheus_port: 9090
          grafana_port: 3000
      vars:
        monitoring_enabled: true
        alert_email: admin@example.com

    # -----------------------------------------------------------------------
    # Специальные хосты
    # -----------------------------------------------------------------------
    special:
      hosts:
        bastion.example.com:
          ansible_host: 203.0.113.1
          ansible_port: 2222
          ansible_user: admin
          is_bastion: true
        legacy01.example.com:
          ansible_host: 192.168.1.100
          ansible_python_interpreter: /usr/bin/python2
          is_legacy: true

    # -----------------------------------------------------------------------
    # Вложенные группы (Группы групп)
    # -----------------------------------------------------------------------
    production:
      children:
        webservers:
        dbservers:
        lbservers:
        monitoring:
      vars:
        environment: production
        monitoring_enabled: true
        log_level: warning
        compliance_required: true

    all_webservers:
      children:
        webservers:
        webservers_staging:
```

**Ключевые элементы YAML формата:**

| Ключ                       | Описание                     |
| -------------------------- | ---------------------------- |
| `all`                      | Корневой элемент инвентаря   |
| `all.vars`                 | Глобальные переменные        |
| `all.children`             | Секция для определения групп |
| `hosts`                    | Список хостов внутри группы  |
| `vars` (внутри группы)     | Переменные группы            |
| `children` (внутри группы) | Вложенные группы             |

#### Best Practices:

1. **Разделение инвентаря:** Вместо одного большого файла используйте директорию `inventory/`:

```Structure
inventory/
├── production/
│   ├── hosts.yml
│   └── group_vars/
├── staging/
│   ├── hosts.yml
│   └── group_vars/
└── requirements.yml
```

Запуск: `ansible-playbook -i inventory/production site.yml`


2. **Не храните секреты в инвентаре:** Используйте **Ansible Vault** для шифрования чувствительных данных (пароли, ключи).

```bash
ansible-vault encrypt inventory/group_vars/all/vault.yml
```

3. **Используйте `ansible_host`:** Если имя хоста в инвентаре отличается от реального IP или DNS, всегда явно указывайте `ansible_host`.


4. **Проверка инвентаря:** Перед запуском плейбуков проверяйте корректность подключения и переменных:

```bash
# Проверка подключения ко всем хостам
ansible all -m ping -i hosts.ini

# Просмотр переменных для конкретного хоста
ansible web01.example.com -m debug -a "var=hostvars[inventory_hostname]" -i hosts.ini

# Визуализация графа хостов
ansible-graph -i hosts.ini
```

   
5. **Выбор формата:**
    - Используйте **YAML**, если вы начинаете новый проект или у вас сложная структура переменных.
    - Используйте **INI**, если нужно быстро поправить конфиг или вы привыкли к старому синтаксису. Ansible отлично работает с обоими.




### Переменные:

В Ansible есть два основных типа "встроенных" переменных, которые вы можете использовать в плейбуках: **Facts (Факты)** и **Magic Variables (Магические переменные)**.

#### Facts (Факты):

Доступ к ним идет через префикс `ansible_`.

**Популярные факты:**

- `{{ ansible_default_ipv4.address }}` — Основной IP-адрес сервера.
- `{{ ansible_os_family }}` — Семейство ОС (например, `Debian`, `RedHat`). Удобно для условий `when`.
- `{{ ansible_distribution }}` — Название дистрибутива (Ubuntu, CentOS).
- `{{ ansible_distribution_version }}` — Версия ОС.
- `{{ ansible_processor_vcpus }}` — Количество ядер CPU.
- `{{ ansible_memtotal_mb }}` — Всего оперативной памяти в МБ.
- `{{ ansible_fqdn }}` — Полное доменное имя хоста.
- `{{ ansible_date_time.date }}` — Текущая дата на сервере.

**Просмотр фактов о конкретном хосте:**

```bash
ansible <имя_хоста> -m setup
```

#### Magic Variables (Магические переменные):

Эти переменные предоставляются самим Ansible в процессе выполнения плейбука. Они не приходят с сервера, а генерируются контроллером.

**Самые важные магические переменные:**

- `{{ inventory_hostname }}` — Имя хоста, как оно записано в файле инвентаря (важно, если оно отличается от реального DNS).
- `{{ ansible_hostname }}` — Короткое имя хоста (из фактов, то есть из ОС).
- `{{ groups }}` — Словарь всех групп и хостов в них.
- `{{ group_names }}` — Список групп, к которым принадлежит текущий хост.
- `{{ hostvars }}` — Доступ к переменным других хостов.
    - _Пример:_ `{{ hostvars['server1']['my_var'] }}`
- `{{ play_hosts }}` — Список всех хостов, участвующих в текущем плей.
- `{{ omit }}` — Специальная переменная. Если подставить её в аргумент модуля, этот аргумент будет проигнорирован (удобно для опциональных параметров).

#### Пользовательские переменные:

В Ansible переменные можно определять в десятках мест. Главное правило: **чем конкретнее место, тем выше приоритет**.

| Источник            | Описание                              | Приоритет         |
| ------------------- | ------------------------------------- | ----------------- |
| `extra_vars` (`-e`) | Переменные командной строки           | **Самый высокий** |
| Playbook `vars`     | Секция `vars` внутри плейбука         | Высокий           |
| Role `vars`         | Файл `roles/<role>/vars/main.yml`     | Высокий           |
| Host vars           | `host_vars/<hostname>.yml`            | Средний           |
| Group vars          | `group_vars/<group>.yml`              | Средний           |
| Inventory vars      | Переменные прямо в файле инвентаря    | Средний           |
| Role `defaults`     | Файл `roles/<role>/defaults/main.yml` | Низкий            |

**Переменные окружения (`group_vars` и `host_vars`)** - это лучшая практика для разделения настроек. Рядом с инвентарем создаются папки, имена которых совпадают с группами или хостами

```Structure
inventory/
├── production/
│   ├── hosts.yml
│   ├── group_vars/
│   │   ├── all.yml        # Для всех хостов
│   │   ├── webservers.yml # Только для группы webservers
│   │   └── dbservers.yml  # Только для группы dbservers
│   └── host_vars/
│       ├── web01.yml      # Специфика конкретного хоста
│       └── db01.yml
```

**Переменные командной строки (`extra_vars`)** имеют самый высокий приоритет. Удобно для CI/CD или разовых запусков.

```bash
ansible-playbook site.yml -e "deploy_version=1.2.3" -e "env=production"

# или через файл
ansible-playbook site.yml -e "@vars/deploy_params.yml"
```

### Управление секретами:

Управление секретами — одна из самых важных задач в Ansible. Для этого можно использовать встроенный инструмент **Ansible Vault**. Он позволяет шифровать любые текстовые файлы (плейбуки, инвентари, переменные) с помощью симметричного шифрования **AES-256**.

**Ansible Vault** — это утилита командной строки, встроенная в Ansible. Она шифрует файлы целиком.

- **Алгоритм:** AES-256 (симметричное шифрование).
- **Ключ доступа:** Пароль (одинаковый для шифрования и расшифровки).
- **Что можно шифровать:** Файлы переменных, плейбуки, инвентари, любые текстовые файлы.
- **Важно:** Ansible не умеет шифровать отдельные переменные внутри файла. Шифруется **весь файл целиком**.

**Все операции выполняются через команду `ansible-vault`.**

|Команда|Описание|Пример|
|---|---|---|
|`create`|Создать новый зашифрованный файл|`ansible-vault create secrets.yml`|
|`encrypt`|Зашифровать существующий файл|`ansible-vault encrypt vars.yml`|
|`decrypt`|Расшифровать файл (снять шифр)|`ansible-vault decrypt vars.yml`|
|`edit`|Открыть файл в редакторе для правки|`ansible-vault edit secrets.yml`|
|`view`|Просмотреть содержимое без расшифровки|`ansible-vault view secrets.yml`|
|`rekey`|Сменить пароль шифрования|`ansible-vault rekey secrets.yml`|
|`enc_str`|Зашифровать строку (для inline)|`ansible-vault encrypt_string 'mypassword'`|

#### Примеры использования

**Создание зашифрованного файла:**

```bash
ansible-vault create group_vars/all/vault.yml
# Вас попросят ввести пароль дважды.
# Откроется редактор (vim), где можно ввести секреты.
```

**Шифрование существующего файла:**

```bash
ansible-vault encrypt group_vars/all/vault.yml
```

**Просмотр содержимого:**

```bash
ansible-vault view group_vars/all/vault.yml
# Введите пароль, чтобы увидеть содержимое в терминале.
```

**Редактирование:**

```bash
ansible-vault edit group_vars/all/vault.yml
# Файл временно расшифровывается, открывается в редакторе,
# после сохранения снова шифруется.
```

Расшифровка происходит **в оперативной памяти** в момент выполнения плейбука.

#### Процесс работы (Step-by-Step):

1. **Запуск:** Вы запускаете `ansible-playbook site.yml`.
2. **Обнаружение:** Ansible сканирует файлы переменных. Если файл начинается с заголовка `$ANSIBLE_VAULT;1.1;AES256`, он помечает его как зашифрованный.
3. **Запрос пароля:**
    - Если пароль не передан явно, Ansible запрашивает его в терминале (`Vault password:`).
    - Если пароль передан (через файл или переменную), Ansible считывает его.
4. **Дешифрация в RAM:** Ansible использует пароль для расшифровки содержимого файла в оперативной памяти.
5. **Слияние переменных:** Расшифрованные переменные объединяются с остальными переменными (из инвентаря, `defaults`, `vars` и т.д.) согласно приоритетам.
6. **Рендеринг:** Когда задача выполняется, шаблонизатор Jinja2 подставляет значения переменных (уже расшифрованных) в задачи.
7. **Очистка:** После завершения работы процесса переменные удаляются из памяти.

#### Способы передачи пароля (Автоматизация):

Ввод пароля вручную неудобен для CI/CD. Есть несколько способов автоматизации.

##### Файл с паролем (Рекомендуемый для CI/CD):

```bash
echo "MySuperSecretPassword" > .vault_pass
chmod 600 .vault_pass  # Важно! Ограничьте права доступа
```

Запуск playbook-а

```bash
ansible-playbook -i hosts site.yml --vault-password-file .vault_pass
```

Или можно указать файл с паролем на глобальном уровне в `ansible.cfg`:

```vim
[defaults]
vault_password_file = ~/.vault_pass
```

##### Несколько паролей (Multi-Vault):

Если разные файлы зашифрованы разными паролями (например, `dev` и `prod`).

```bash
ansible-playbook site.yml 
  --vault-id dev@.vault_pass_dev 
  --vault-id prod@.vault_pass_prod
```

В файле инвентаря или плейбуке можно указать, какой ID нужен:

```yaml
---
- hosts: all
  vars_files:
    - vault.yml  # Без указания ID
    - vault_prod.yml@prod  # Явное указание ID
```

#### Отладка и безопасность:

**Как проверить, что файл зашифрован?**

```bash
head -1 group_vars/all/vault.yml
# Должно вывести: $ANSIBLE_VAULT;1.1;AES256
```

**Что если я забыл пароль?** Данные потеряны навсегда. AES-256 невозможно взломать перебором за разумное время. _Совет:_ Храните резервную копию пароля в менеджере паролей (KeePass, 1Password).

**Видны ли секреты в логах?** По умолчанию Ansible старается скрывать переменные, содержащие слова `password`, `secret`, `key`. Но для надежности используйте модуль `no_log`:

```yml
- name: Create database user
  postgresql_user:
    name: admin
    password: "{{ db_password }}"
  no_log: true  # Не логировать вывод этой задачи
```

**Проверка утечек.** Запустите плейбук с флагом `-vvv`, чтобы увидеть, что именно отправляется на сервер. Убедитесь, что зашифрованные переменные не "светятся" в логах в открытом виде.

### Рекомендуемая структура проекта:

Эта структура подходит для сложных проектов с ролями и несколькими окружениями.

```Structure
my_ansible_project/
├── ansible.cfg                 # Конфигурация Ansible
├── requirements.yml            # Зависимости (роли и коллекции из Galaxy)
├── .gitignore                  # Игнорируемые файлы (.vault_pass, *.retry)
├── .vault_pass                 # Пароль vault (НЕ коммитить в git!)
│
├── inventory/                  # Инвентари для разных сред
│   ├── production/
│   │   ├── hosts.yml
│   │   ├── group_vars/
│   │   └── host_vars/
│   └── staging/
│       ├── hosts.yml
│       ├── group_vars/
│       └── host_vars/
│
├── playbooks/                  # Точка входа (плейбуки)
│   ├── site.yml                # Главный плейбук
│   ├── webservers.yml
│   └── dbservers.yml
│
├── roles/                      # Локальные роли проекта
│   ├── common/
│   ├── nginx/
│   └── app/
│
├── collections/                # Локальные коллекции (если есть)
│   └── mycompany/
│
├── plugins/                    # Кастомные плагины
│   ├── filter/
│   ├── lookup/
│   └── callback/
│
├── vars/                       # Глобальные переменные
│   ├── common.yml
│   └── secrets.yml             # Зашифровано через ansible-vault
│
├── files/                      # Статические файлы (для copy)
│   └── configs/
│
├── templates/                  # Jinja2 шаблоны (для template)
│   └── app/
│
└── logs/                       # Логи выполнения (добавить в .gitignore)
```
