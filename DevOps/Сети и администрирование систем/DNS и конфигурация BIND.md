
**DNS** — это распределённая иерархическая система, которая преобразует читаемые доменные имена (например, `example.com`) в IP-адреса и обратно.

| Принцип              | Описание                                                                                       |
| -------------------- | ---------------------------------------------------------------------------------------------- |
| **Иерархия**         | Древовидная структура: корневые серверы → TLD → домены второго уровня → поддомены              |
| **Распределённость** | Нет единой базы данных; каждая зона управляется независимо                                     |
| **Кэширование**      | Результаты запросов кэшируются для снижения нагрузки и ускорения разрешения                    |
| **UDP/53**           | Основной транспорт — UDP порт 53 (TCP используется для больших ответов и зональных трансферов) |

### Типы и режимы работы DNS-серверов:

#### Авторитетный сервер (Authoritative):

Хранит и отдаёт «официальные» данные для определённой зоны (домена). Отвечает только на вопросы о записях, за которые несёт ответственность. Не выполняет рекурсию для клиентов.

| Подтип                | Описание                                                                          |
| --------------------- | --------------------------------------------------------------------------------- |
| **Master (Primary)**  | Исходный источник зональных данных, хранит файл зоны в редактируемом виде         |
| **Slave (Secondary)** | Получает копию зоны от мастера через AXFR/IXFR, служит для отказоустойчивости     |
| **Hidden Master**     | Мастер не публикуется в делегировании (нет в NS-записях), обновляет только слейвы |

**Пример конфигурации BIND (named.conf)**

```vim
// Master-сервер для example.com
zone "example.com" {
    type master;                          // Этот сервер — источник истины для зоны
    
    file "/var/lib/bind/zones/example.com.zone";  // Путь к файлу с записями зоны
    
    // 🔐 Контроль доступа к передаче зоны (AXFR/IXFR)
    allow-transfer { 
        192.168.10.2;     // IP слейва 1
        192.168.10.3;     // IP слейва 2
        key "slave-key";  // Или аутентификация по ключу TSIG
    };
    
    // 🔔 Уведомления слейвов об изменениях
    notify yes;                           // Включить отправку NOTIFY

    // Явно указать, кого уведомлять (помимо тех, кто в NS записях)
    also-notify {
        192.168.10.2;
        192.168.10.3;
        // Можно добавить скрытый бэкап-сервер, которого нет в NS
    };
    
    // 🔄 Динамические обновления (DDNS) — опционально
    // allow-update { key "ddns-key"; };
    // update-policy { grant ddns-key name example.com. A AAAA; };
    
    // 🗄️ Кэширование отрицательных ответов для этой зоны
    max-negative-ttl 3600;  // Переопределение глобального значения
    
    // 📝 Журналирование изменений (для отладки)
    // zone-statistics yes;
};

// Пример обратной зоны (PTR)
zone "10.168.192.in-addr.arpa" {
    type master;
    file "/var/lib/bind/zones/192.168.10.rev";
    allow-transfer { 192.168.10.2; 192.168.10.3; };
    notify yes;
    also-notify { 192.168.10.2; 192.168.10.3; };
};
```

```vim
// Объявление зоны как СЛЕЙВ
zone "example.com" {
    type slave;                           // Этот сервер получает зону от мастера
    
    file "/var/lib/bind/slaves/example.com.zone";  // Куда сохранить копию
    
    // От кого принимать зону (можно несколько для отказоустойчивости)
    masters { 
        192.168.10.1;     // IP основного мастера
        192.168.10.99;    // IP резервного мастера (опционально)
        // port 53;       // Если мастер слушает нестандартный порт
        // key "master-key";  // Аутентификация по TSIG
    };
    
    // Как часто проверять мастера, если NOTIFY не пришёл
    // (по умолчанию берётся из SOA Refresh, но можно переопределить)
    // refresh 3600;
    
    // ⏱️ Таймауты передачи
    max-transfer-time-in 120;       // Опционально: проверять наличие зоны даже 
									   без NOTIFY каждые N секунд
									// (по умолчанию берется из SOA Refresh)
    max-transfer-idle-in 60;        // Таймаут бездействия при передаче
    
    // Запретить этому слейву передавать зону дальше
    allow-transfer { none; };
    
};

zone "10.168.192.in-addr.arpa" {
    type slave;
    file "/var/lib/bind/slaves/192.168.10.rev";
    masters { 192.168.10.1; };
    allow-transfer { none; };
};
```

**Когда использовать?**

- Вы управляете доменом и хотите самостоятельно хостить его зону;
- Нужна отказоустойчивость через несколько авторитетных серверов;
- Требуется полный контроль над записями (включая DNSSEC).

#### Рекурсивный сервер (Recursive Resolver):

Принимает запросы от клиентов и самостоятельно «обходит» иерархию DNS, чтобы найти ответ. Кэширует результаты для ускорения повторных запросов. Не хранит зоны. Сам ищет ответ, начиная с корня  

```vim
options {
    directory "/var/cache/bind";
    
    // 🎧 Прослушиваемые интерфейсы
    listen-on { 127.0.0.1; 192.168.10.1; };
    listen-on-v6 { ::1; };
    
    // Кто может задавать вопросы
    allow-query { localhost; 192.168.10.0/24; };
    
    // Рекурсия: ВКЛЮЧЕНА (сервер будет сам ходить по цепочке DNS)
    recursion yes;
    allow-recursion { localhost; 192.168.10.0/24; };
    
    // ФОРВАРДЕРЫ НЕ УКАЗАНЫ → сервер будет делать полную рекурсию
    // (начинать с корневых серверов из named.root.hints)
    // Если хотите добавить форвардеры — раскомментируйте блок ниже:
    /*
    forwarders {
        8.8.8.8;
        1.1.1.1;
    };
    forward first;
    */
    
    // Запрет трансферов (нет зон → нечего передавать)
    allow-transfer { none; };
    
    // Агрессивное кэширование (цель caching-only — макс. скорость)
    max-cache-ttl 604800;   // До 7 дней для стабильных записей
    min-cache-ttl 600;      // Минимум 10 минут
    max-cache-size 4g;      // Выделить до 4 ГБ под кэш (если есть память)
    
    // Безопасность
    dnssec-validation auto;
    version "not disclosed";
    
    // Rate limiting (обязательно для публичных резолверов)
    rate-limit {
        responses-per-second 15;
        window 5;
    };
    
    // Логирование (включить при отладке)
    querylog yes;  // Записывать все запросы в лог (осторожно: много данных!)
};
```

**Как работает рекурсия (упрощённо):**

1. Клиент → Резолвер: "Какой IP у www.example.com?"
2. Резолвер → Корневой сервер: "Где .com?"
3. Корень → Резолвер: "Спроси у TLD .com: ns1.tld-servers.net"
4. Резолвер → TLD: "Где example.com?"
5. TLD → Резолвер: "Спроси у ns1.example.com"
6. Резолвер → Авторитетный: "Дай A-запись для www.example.com"
7. Авторитетный → Резолвер: "192.168.10.10"
8. Резолвер → Клиент: "192.168.10.10" + кэширует ответ

**Когда использовать?**

- Для локальной сети компании или дома
- Чтобы контролировать кэширование и фильтрацию (например, блокировка рекламы)
- Для повышения приватности (не использовать публичные резолверы)

#### Форвардер (Forwarder):

Не разрешает имена самостоятельно, а перенаправляет запросы другому рекурсивному серверу. Полезно для централизации, фильтрации или обхода ограничений.

**Конфигурация BIND: Forwarder и Caching-Only:**

```vim
options {
    // Рабочая директория для файлов кэша и логов
    directory "/var/cache/bind";
    
    // Сетевой интерфейс и порт для прослушивания
    // Можно указать несколько: interface 192.168.10.1 port 53;
    listen-on { 127.0.0.1; 192.168.10.1; };
    listen-on-v6 { ::1; };  // IPv6
    
    // Контроль доступа: кто может делать запросы
    // "any" — разрешить всем (осторожно в публичной сети!)
    allow-query { 
        localhost; 
        192.168.10.0/24;  // Разрешить только локальной сети
    };
    
    // РЕКУРСИЯ: разрешить рекурсивные запросы
    // Для форвардера это обычно ВКЛЮЧЕНО, иначе он не сможет отвечать клиентам
    recursion yes;
    
    // Кто может ЗАПУСКАТЬ рекурсию (важно для безопасности!)
    // Даже если recursion yes, без этого списка внешние не смогут использовать вас как open resolver
    allow-recursion {
        localhost;
        192.168.10.0/24;
        // 10.0.0.0/8;  // можно добавить другие доверенные сети
    };
    
    // ФОРВАРДИНГ: куда пересылать запросы
    forwarders {
        8.8.8.8;           // Google DNS
        8.8.4.4;           // Google DNS backup
        1.1.1.1;           // Cloudflare
        208.67.222.222;    // OpenDNS (опционально)
    };
    
    // Режим форвардинга:
    // "first" — попробовать форвардеры, если не ответили — делать рекурсию самому (по умолчанию)
    // "only"  — использовать ТОЛЬКО форвардеры, если они упали — вернуть ошибку
    forward first;
    
    // Запрет зональных трансферов (этот сервер не авторитетный!)
    allow-transfer { none; };
    
    // КЭШИРОВАНИЕ: настройки времени жизни записей
    // Переопределяют TTL из ответов, если они вне диапазона
    max-cache-ttl 86400;    // Максимум 24 часа в кэше
    min-cache-ttl 300;      // Минимум 5 минут (чтобы не кэшировать "мусор" с TTL=1)
    
    // Размер кэша в мегабайтах (опционально, для контроля памяти)
    // max-cache-size 2g;
    
    // DNSSEC: проверка подписей
    // "auto" — автоматически загружать root key и проверять, если есть подписи
    dnssec-validation auto;
    
    // Защита от атак и утечек информации
    version "not disclosed";           // Скрыть версию BIND в ответах
    hostname "not disclosed";
    server-id "not disclosed";
    
    // Rate limiting: защита от DNS amplification атак
    rate-limit {
        responses-per-second 10;   // Не более 10 ответов в секунду на один запрос
        window 5;                   // Интервал оценки в секундах
        log-only false;             // Не только логировать, но и блокировать
    };
    
    // Геолокация и Anycast (опционально, для сложных конфигураций)
    // topology { 
    //     192.168.10.0/24; 
    //     10.0.0.0/8; 
    // };
    
    // Статистика и мониторинг (опционально)
    // statistics-file "/var/cache/bind/named_stats.txt";
    // dump-file "/var/cache/bind/named_dump.db";
};
```

```vim
zone "partner.internal" {
    type forward;                         // Не хранить зону, пересылать запросы
    
    forward only;                         // "only" = только форвардеры, "first" = fallback к рекурсии

        10.20.30.40;     // DNS-сервер партнёра
        10.20.30.41;     // Резервный

        // tls yes;      // Включить TLS (требуется настройка tls {})
    };
    
    // 🔐 Ограничить, кто может запрашивать эту зону через вас
    allow-query { 192.168.10.0/24; };
    
    // 🔄 Кэширование ответов для этой зоны
    max-cache-ttl 1800;   // Кэшировать ответы не дольше 30 минут
};
```

#### Кэширующий сервер (Caching-only):

Не хранит собственные зоны, только кэширует ответы от других серверов. Частный случай рекурсивного сервера, но без авторитетных зон. Уменьшает нагрузку на сеть и ускоряет разрешение. 

Кеширующий сервер это не отдельный тип сервера, а режим работы сервера. Сервера **forwarder** и **recursive** могут быть кеширующими.
 1. Не имеет авторитетных зон (`named.conf.local` пустой);
2. Включает рекурсию и кэширование;
3. Может работать как с полной рекурсией, так и с форвардингом.


#### Разница между Forwarder, Recursive и Caching серверами:

| Характеристика                   | **Recursive**                                       | **Forwarder**                                  | **Caching-only**                                               |
| -------------------------------- | --------------------------------------------------- | ---------------------------------------------- | -------------------------------------------------------------- |
| **Хранит авторитетные зоны**     | ❌ Нет                                               | ❌ Нет                                          | ❌ Нет                                                          |
| **Как получает ответ**           | Сам ходит от корневых серверов → TLD → авторитетный | Пересылает запрос другому рекурсивному серверу | *Режим работы*, а не метод: может быть Recursive или Forwarder |
| **Кэширует ответы**              | ✅ Да                                                | ✅ Да                                           | ✅ Да (это его суть)                                            |
| **Начинает с корневых серверов** | ✅ Да                                                | ❌ Нет (делает форвардер)                       | Зависит от конфигурации                                        |
| **Может иметь `forwarders {}`**  | Опционально                                         | ✅ Обязательно                                  | Зависит от реализации                                          |
| **Аналогия**                     | «Я сам найду адрес в телефонном справочнике»        | «Я спрошу у коллеги, который уже искал»        | «Я только запоминаю найденное, сам не ищу»                     |


### Stub-зоны:

**Stub-зона** — это специальная конфигурация зоны в BIND, при которой сервер хранит **только три типа записей** для дочерней зоны:

|Тип записи|Зачем нужна|
|---|---|
|**SOA**|Метаданные зоны (серийный номер для проверки обновлений)|
|**NS**|Список авторитетных серверов для этой зоны|
|**Glue (A/AAAA)**|IP-адреса для NS-серверов, если они находятся внутри этой же зоны|
