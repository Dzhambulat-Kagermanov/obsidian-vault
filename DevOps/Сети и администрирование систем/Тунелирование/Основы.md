
Туннелирование (Tunneling) — это фундаментальная технология компьютерных сетей для **создания «виртуального кабеля» внутри другой сети.**

### Атрибуты при конфигурировании туннелей:

##### MTU:

Максимальный размер пакета (в байтах), который может быть передан через интерфейс без фрагментации. При инкапсуляции к исходному пакету добавляется заголовок туннеля. Если итоговый размер превысит MTU физического интерфейса (обычно 1500 байт), произойдёт:
- **Фрагментация** (потеря производительности, риск потери фрагментов)
- **Отбрасывание пакетов** (если установлен флаг DF — Don't Fragment)

##### TTL (Time To Live):

счетчик жизни пакета, который уменьшается при каждой пересылке

| Значение | Когда использовать                                   |
| -------- | ---------------------------------------------------- |
| `255`    | По умолчанию, для локальных/доверенных сетей         |
| `64`     | Стандартное значение для Интернет-трафика            |
| `1-10`   | Для туннелей внутри одного дата-центра (оптимизация) |

##### Keyexchange (для IPSec):

Определяет версию протокола **IKE (Internet Key Exchange)** (IKEv1 или IKEv2), который используется для установления защищённого соединения в IPsec.

**Сравнение версий IKE:**

| Параметр                  | IKEv1 (ISAKMP)              | IKEv2 (RFC 7296)                       |
| ------------------------- | --------------------------- | -------------------------------------- |
| **Год выпуска**           | 1998 (устаревший)           | 2014 (современный)                     |
| **Обмен сообщениями**     | 6-9 сообщений (2 фазы)      | 4 сообщения (объединённые фазы)        |
| **NAT Traversal**         | Требует отдельной настройки | Встроен по умолчанию                   |
| **Аутентификация**        | Только по сертификатам/PSK  | Поддержка EAP, XAuth, сертификаты, PSK |
| **Mobility (MOBIKE)**     | ❌ Нет                       | ✅ Да (смена IP без переподключения)    |
| **Устойчивость к атакам** | Ниже                        | Выше (защита от DoS, replay)           |
| **Совместимость**         | Старое оборудование         | Современные ОС и роутеры               |

##### AuthBy (для IPSec):

Метод аутентификации используемый в IPSec.

| Значение | Описание                                   | Безопасность                                  |
| -------- | ------------------------------------------ | --------------------------------------------- |
| `secret` | Pre-Shared Key (общий пароль)              | ⚠️ Средняя (удобно, но сложно масштабировать) |
| `rsasig` | Подпись сертификатом RSA                   | ✅ Высокая (рекомендуется для продакшена)      |
| `ecdsa`  | Подпись сертификатом ECDSA                 | ✅✅ Очень высокая (современный стандарт)       |
| `xauth`  | Дополнительная аутентификация пользователя | ✅ Для remote-access VPN                       |

##### Алгоритмы шифрования: ike / esp:

### Для чего нужно туннелирование?

**Безопасность и Конфиденциальность (Шифрование)**. Это самая частая причина использования туннелей в современном мире:
- **Проблема:** Данные в обычной сети (HTTP, FTP, Telnet) передаются в открытом виде. Любой провайдер или хакер в общественной Wi-Fi сети может их прочитать;
- **Решение:** Туннель (например, IPsec, WireGuard, OpenVPN) шифрует весь трафик перед отправкой. Даже если перехватят пакеты, внутри будет только «каша»;
- **Пример:** Корпоративный VPN, когда сотрудник работает из кафе и подключается к серверам офиса.

**Преодоление несовместимости протоколов (Инкапсуляция)**. Иногда нужно передать протокол через сеть, которая его «не понимает» или блокирует:
- **Проблема:** У вас есть старое оборудование, работающее только по IPv6, а провайдер дает только IPv4. Или вы хотите передать широковещательный трафик (Broadcast), который роутеры по умолчанию не пропускают через Интернет.
- **Решение:** Мы берем пакет одного протокола (например, IPv6 или Ethernet-кадр) и заворачиваем его в оболочку другого протокола (IPv4). Внешняя сеть видит только обычное IPv4-соединение, а на выходе туннеля оболочка снимается, и пакет летит дальше как родной.
- **Пример:** Протокол GRE (о котором мы говорили ранее) часто используют именно для этого — чтобы «протащить» маршрутизацию OSPF или multicast-видео через сеть, которая их блокирует.

**Объединение географически разнесенных сетей (Site-to-Site)**:
- **Проблема:** У компании есть офис в Москве и склад в Владивостоке. Нужно, чтобы они работали как одна локальная сеть (чтобы принтер в Москве видел компьютер во Владивостоке без сложных настроек).
- **Решение:** Туннель соединяет два шлюза. Для компьютеров внутри сети расстояние не имеет значения — они «думают», что находятся в одной комнате, хотя между ними тысячи километров и публичный Интернет.
- **Пример:** Сеть банка, где все филиалы объединены в единую защищенную инфраструктуру.

**Обход ограничений и цензуры**:
- **Проблема:** Провайдер или государство блокирует доступ к определенным ресурсам или портам.
- **Решение:** Туннелирование позволяет замаскировать трафик. Вы подключаетесь к удаленному серверу, где нет блокировок, и весь ваш запрос идет через этот туннель. Для провайдера видно лишь соединение с одним IP-адресом сервера, но не то, какие сайты вы посещаете внутри туннеля.
- **Пример:** Использование VPN-сервисов для доступа к заблокированным ресурсам.\

**Создание изолированных сред (Overlay Networks)**. В эпоху облаков и виртуализации это критически важно:
- **Проблема:** В большом дата-центре живут сотни клиентов. Их сети не должны пересекаться. У клиента А может быть подсеть `192.168.1.0/24`, и у клиента Б точно такая же. Как их разделить на одном физическом оборудовании?
- **Решение:** Создаются виртуальные туннели (VXLAN, GRE, Geneve). Каждый клиент живет в своем изолированном «пузыре». Пакеты одного клиента инкапсулируются с уникальным идентификатором (VNI), поэтому они никогда не попадут к другому клиенту, даже если IP-адреса совпадают.
- **Пример:** Docker, Kubernetes, VMware NSX, облачные провайдеры (AWS VPC, Azure VNet).

### Принцип работы:

1. **Инкапсуляция (Упаковка):** Исходный пакет (`Пакет А`) помещается внутрь нового пакета (`Пакет Б`):
    - _Заголовок Пакета Б_ содержит адреса начала и конца туннеля.
    - _Полезная нагрузка Пакета Б_ — это весь `Пакет А` целиком.
2. **Транспортировка:** Сеть видит только заголовок `Пакета Б` и доставляет его до конечной точки туннеля, игнорируя содержимое.
3. **Декапсуляция (Распаковка):** Получатель (конец туннеля) снимает заголовок `Пакета Б`, извлекает `Пакет А` и обрабатывает его так, будто он пришел напрямую.

**Без туннелирования** современный Интернет был бы небезопасным, фрагментированным и не смог бы поддерживать облачные технологии в том виде, в котором мы их знаем.
