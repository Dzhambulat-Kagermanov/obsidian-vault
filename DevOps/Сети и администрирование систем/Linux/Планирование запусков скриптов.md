|---|---|---|
|**Точность**|Минутная granularity.|Можно задать секунды, микросекунды.|
|**Зависимости**|Нет. Запускает слепо.|Может ждать сеть (`After=network.target`), монтирование дисков и т.д.|
|**Логирование|Через mail или файлы.|Интегрировано в `journalctl`. Удобно смотреть логи.|

|**Распространенность**|Есть везде (Unix, Linux, macOS).|Только в системах с systemd (Linux).|

### Cron:

**Cron** — это демон-планировщик задач в Unix-подобных системах (Linux, macOS, BSD). Его задача — запускать определенные команды или скрипты в заданное время автоматически.

#### Архитектура: Как это работает «под капотом»:

1. **Демон `crond`**: Это фоновый процесс, который запускается при старте системы. Он просыпается каждую минуту.
2. **Проверка**: Каждую минуту демон проверяет все файлы расписаний (crontabs) пользователей и системные конфиги.
3. **Сравнение**: Он сравнивает текущее время (минуту, час, день месяца, месяц, день недели) с правилами в файлах.
4. **Исполнение**: Если время совпадает, `crond` порождает дочерний процесс (shell `/bin/sh` по умолчанию), который выполняет указанную команду.
5. **Вывод**: Стандартный вывод (stdout) и ошибки (stderr) команды по умолчанию отправляются пользователю через локальную почту (mail subsystem). Если почта не настроена, вывод просто теряется (частая проблема!).

##### Синтаксис Crontab:

Файл расписания состоит из строк вида:

```
* * * * * command_to_execute
│ │ │ │ │
│ │ │ │ └── День недели (0–7, где 0 и 7 — воскресенье)
│ │ │ └──── Месяц (1–12)
│ │ └────── День месяца (1–31)
│ └──────── Час (0–23)
└────────── Минута (0–59)
```

##### Специальные символы:

- `*` — любое значение (каждую минуту/час/день...).
- `,` — перечисление значений. Пример: `1,15 * * * *` (запуск в 1-ю и 15-ю минуту каждого часа).
- `-` — диапазон. Пример: `1-5 * * * *` (с 1-й по 5-ю минуту).
- `/` — шаг. Пример: `*/10 * * * *` (каждые 10 минут).


**Примеры:**

```chron
# Каждый день в 3:00 ночи
0 3 * * * /usr/local/bin/backup.sh

# Каждые 15 минут
*/15 * * * * /usr/bin/check_health.sh

# Каждый понедельник в 9:00 утра
0 9 * * 1 /usr/bin/weekly_report.sh

# В 12:00 первого числа каждого месяца
0 12 1 * * /usr/bin/monthly_cleanup.sh
```

#### Где хранятся задания?

##### Пользовательские crontab (`crontab -e`):

Каждый пользователь имеет свой файл расписания.

- Команда редактирования: `crontab -e`;
- Просмотр: `crontab -l`;
- Удаление: `crontab -r` (осторожно, удаляет всё без подтверждения!)
- Расположение файлов (зависит от дистрибутива): обычно `/var/spool/cron/crontabs/username` или `/var/spool/cron/username`.

##### Системные crontab (`/etc/crontab` и `/etc/cron.d/`):

Используются для системных задач.

- Файл `/etc/crontab` имеет немного другой синтаксис: там есть дополнительное поле **пользователь**, от имени которого запускать команду.

```cron
# m h dom mon dow user command
17 * * * * root cd / && run-parts --report /etc/cron.hourly
```

#### Нюансы при работе:

##### Переменные окружения (Environment Variables):

Это самая частая причина, почему скрипт работает вручную, но не работает в cron. Cron запускает команды в «минималистичном» окружении.
- Переменная `PATH` обычно равна `/usr/bin:/bin`. Там нет `/usr/local/bin`, `/sbin`, `/home/user/bin`.
- Нет переменных `HOME`, `USER` (в некоторых реализациях), `LANG`.

Решение:

- Всегда указывай **полные пути** к исполняемым файлам в командах.
    - Плохо: `rsync -av ...`
    - Хорошо: `/usr/bin/rsync -av ...`
- Или задай `PATH` в начале crontab:
	```cron
	PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
	```

* Загружай профиль пользователя (если нужно):
	```cron
	* * * * * source /home/dzhambulat/.bashrc && /path/to/script.sh
	```

##### Рабочая директория (Working Directory):

Cron не гарантирует, что он запустит команду из домашней директории пользователя или из папки со скриптом. **Решение:** Всегда делай `cd` перед выполнением действий, зависящих от относительных путей.

```cron
* * * * * cd /opt/myapp && ./run_backup.sh
```

##### Логирование и вывод:

По умолчанию cron шлет почту. Если postfix/sendmail не настроен, ты не увидишь ошибок.

**Решение:** Перенаправляй вывод в лог-файл явно.

```cron
# Stdout и Stderr в один файл
* * * * * /path/to/script.sh >> /var/log/my_script.log 2>&1

# Только ошибки
* * * * * /path/to/script.sh > /dev/null 2>> /var/log/my_script_errors.log
```

##### Блокировка параллельных запусков:

Если скрипт выполняется дольше интервала cron (например, скрипт идет 15 минут, а cron стоит на `*/10`), запустится вторая копия. Это может убить базу данных или забить диск. **Решение:** Используй `flock` (file lock).

```cron
# Запускать раз в минуту, но не более одной копии одновременно
* * * * * /usr/bin/flock -n /tmp/my_script.lock /path/to/script.sh
```

Флаг `-n` (non-blocking) заставит cron пропустить запуск, если предыдущая копия еще работает.

##### Права доступа:

Файл crontab должен принадлежать пользователю. Если ты правишь `/etc/cron.d/`, файл должен принадлежать `root`.
