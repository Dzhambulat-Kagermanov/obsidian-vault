
**Keepalived** — это Linux-демон, обеспечивающий **высокую доступность (HA)** и балансировку нагрузки для кластеров на базе LVS (Linux Virtual Server). Он использует протокол **VRRP** (Virtual Router Redundancy Protocol) для управления **виртуальным IP-адресом (VIP)**, который автоматически перемещается между серверами кластера, обеспечивая бесшовный failover и непрерывную доступность сервисов.

**Два основных компонента Keepalived:**

|Компонент|Сетевой уровень|Описание|
|---|---|---|
|**IPVS** (Load Balancing)|Layer 4 (Transport)|Маршрутизация трафика на основе информации о TCP/UDP-соединениях|
|**VRRP** (High Availability)|Layer 3 (Network)|Обеспечение отказоустойчивости по IP и управление переходами active-passive<br><br>medium.com|

**Принцип работы:**

1. В группе маршрутизаторов (один **VRRP-инстанс**) совместно используется **виртуальный IP (VIP)**.
2. Сервер с **наивысшим приоритетом** становится **MASTER** и принимает трафик на этот VIP.
3. **BACKUP-узлы** постоянно обмениваются VRRP-пакетами (по умолчанию — каждую секунду) для проверки доступности MASTER.
4. Если MASTER не отвечает в течение **3 интервалов** (по умолчанию — 3 секунды), BACKUP с наивысшим приоритетом берёт на себя роль MASTER и назначает VIP своему интерфейсу.
5. При восстановлении MASTER он автоматически возвращает себе управление VIP (если настроено преобладание по приоритету)

> **Важно!** Только один сервер должен быть активным MASTER в любой момент времени. Если несколько серверов одновременно имеют VIP — это признак проблемы с коммуникацией VRRP (проверьте фаервол: должен быть разрешён multicast-трафик на `224.0.0.18` или порт `112` для unicast).

**Ограничения Keepalived:**

- Не обеспечивает балансировку нагрузки «из коробки» (только active-passive для одного VIP).
- Ограничение в **255 уникальных `virtual_router_id`** в одной широковещательной домене.
- Требует точной синхронизации времени и стабильной сетевой связности между узлами.

### Конфигурация MASTER сервера:

```haproxy
global_defs {
   # Идентификатор узла (для логов)
   router_id LB_MASTER
   # Отключение защиты от зацикливания (для локальных тестов можно оставить, в prod лучше включить)
   # vrrp_skip_check_adv_addr
   # Строгая проверка исходного адреса (безопасность)
   vrrp_strict
}

# Скрипт проверки здоровья сервиса (например, Nginx)
vrrp_script check_nginx {
    script "/usr/bin/curl -sf http://127.0.0.1/health >/dev/null"
    interval 2          # Проверка каждые 2 секунды
    timeout 1           # Ждем ответ 1 секунду
    fall 3              # 3 неудачи = узел считается нерабочим
    rise 2              # 2 успеха = узел снова рабочий
    priority -20        # При сбое скрипта уменьшаем приоритет на 20
}

vrrp_instance VI_1 {
    # Начальное состояние. 
    # В режиме Unicast с nopreempt оба узла могут быть BACKUP, 
    # но для классики оставляем MASTER здесь.
    state MASTER
    
    # Сетевой интерфейс, на котором будет жить VIP
    interface eth0
    
    # Уникальный ID группы (должен совпадать на всех узлах!)
    # Диапазон 0-255. Не должен пересекаться с другими кластерами в сети.
    virtual_router_id 51
    
    # Приоритет. У MASTER должен быть выше, чем у BACKUP.
    # Базовый приоритет 100. Ставим 150.
    priority 150
    
    # Интервал отправки анонсов (секунды). 
    # Чем меньше, тем быстрее фаиловер, но больше нагрузка на сеть.
    advert_int 1
    
    # --- НАСТРОЙКИ UNICAST ---
    # IP адрес текущего сервера
    unicast_src_ip 192.168.1.10
    # IP адрес соседа (куда слать пакеты)
    unicast_peer {
        192.168.1.11
    }
    # --------------------------
    
    # Аутентификация (простая, пакеты не шифруются, но защищают от случайных узлов)
    authentication {
        auth_type PASS
        auth_pass SecretPassword123
    }
    
    # Подключение скрипта проверки
    track_script {
        check_nginx
    }
    
    # Виртуальные IP адреса (можно указать несколько)
    virtual_ipaddress {
        192.168.1.100/24 dev eth0 label eth0:vip
    }
    
    # Скрипты, выполняемые при смене состояния
    notify_master "/etc/keepalived/notify_master.sh"
    notify_backup "/etc/keepalived/notify_backup.sh"
    notify_fault  "/etc/keepalived/notify_fault.sh"
}
```

### Конфигурация BACKUP:

Отличия от MASTER выделены комментариями:

```haproxy
global_defs {
   router_id LB_BACKUP  # Имя узла
   vrrp_strict
}

vrrp_script check_nginx {
    script "/usr/bin/curl -sf http://127.0.0.1/health >/dev/null"
    interval 2
    timeout 1
    fall 3
    rise 2
    priority -20
}

vrrp_instance VI_1 {
    state BACKUP          # [ОТЛИЧИЕ] Начальное состояние
    
    interface eth0
    virtual_router_id 51  # [ВАЖНО] Должен совпадать с MASTER
    
    priority 100          # [ОТЛИЧИЕ] Приоритет ниже, чем у MASTER (150)
    advert_int 1
    
    # --- НАСТРОЙКИ UNICAST ---
    unicast_src_ip 192.168.1.11  # [ОТЛИЧИЕ] Свой IP
    unicast_peer {
        192.168.1.10             # [ОТЛИЧИЕ] IP соседа (MASTER)
    }
    # --------------------------
    
    authentication {
        auth_type PASS
        auth_pass SecretPassword123 # [ВАЖНО] Пароль должен совпадать
    }
    
    track_script {
        check_nginx
    }
    
    virtual_ipaddress {
        192.168.1.100/24 dev eth0 label eth0:vip
    }
    
    # Уведомления (опционально)
    notify_master "/etc/keepalived/notify_master.sh"
    notify_backup "/etc/keepalived/notify_backup.sh"
}
```


### Разбор ключевых параметров конфигурации:

| Параметр            | Значение            | Зачем нужен                                                                                                                                      |
| ------------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `state`             | `MASTER` / `BACKUP` | Определяет начальное поведение при запуске сервиса. В режиме `nopreempt` оба узла могут быть `BACKUP`, и мастер выбирается только по приоритету. |
| `virtual_router_id` | `51` (пример)       | **Критично.** Это ID группы. Если на одном сегменте сети будут два кластера с одинаковым ID — возникнет конфликт IP.                             |
| `priority`          | `150` / `100`       | Кто важнее. При старте узел с большим приоритетом становится MASTER. Если скрипт `track_script` падает, приоритет уменьшается.                   |
| `advert_int`        | `1`                 | Частота «пингов» между узлами. Если за `3 * advert_int` нет ответа — считается, что мастер умер.                                                 |
| `unicast_src_ip`    | `IP`                | Адрес, с которого текущий узел отправляет пакеты VRRP.                                                                                           |
| `unicast_peer`      | `IP`                | Адрес соседа, которому нужно отправлять пакеты.                                                                                                  |
| `track_script`      | `name`              | Связывает проверку сервиса (скрипт) с экземпляром VRRP. Если скрипт возвращает ошибку, Keepalived снижает приоритет.                             |
| `nopreempt`         | (отсутствует)       | Если добавить эту директиву, то при восстановлении старого MASTER он **не отберет** роль у текущего, пока тот жив. Полезно для стабильности.     |

### Нюансы для Production:

**Фаервол (Firewall).** VRRP использует **протокол 112**. Если вы включите `firewalld` или `ufw`, кластер не заработает, пока вы не откроете этот протокол.

**Режим Nopreempt (Стабильность).** В классической конфигурации, если MASTER перезагружается, он сразу забирает VIP обратно (Preempt). Это вызывает лишний перепад трафика. Чтобы избежать «мигания» роли, добавьте `nopreempt` в конфиг **обоих** узлов и установите `state BACKUP` на обоих. Тогда мастером станет тот, у кого выше приоритет, но при восстановлении первого узла переключения не произойдет. Пример для стабильности:

```haproxy
vrrp_instance VI_1 {
    state BACKUP        # На обоих узлах
    nopreempt           # На обоих узлах
    priority 150        # На желаемом мастере выше
    # ...
}
```

**Скрипты уведомлений (Notify).** Вы можете отправлять алерты в Slack/Telegram при смене роли. Скрипт получает аргументы: `$1` (тип: MASTER/BACKUP/FAULT), `$2` (имя инстанса), `$3` (приоритет).

**Проверка работы.** Ищите строки: `Entering MASTER STATE` или `Entering BACKUP STATE`.

```bash
journalctl -u keepalived -f
```
