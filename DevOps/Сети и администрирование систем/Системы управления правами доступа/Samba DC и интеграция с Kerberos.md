### Основы:

**SAMBA** — это свободный программный пакет, который реализует протокол **SMB/CIFS** (Server Message Block / Common Internet File System) в Unix-подобных системах. Он позволяет Linux-серверам и рабочим станциям:

- Предоставлять файлы и принтеры для доступа из сетей Windows
- Интегрироваться в домены Active Directory
- Выступать в роли контроллера домена (начиная с версии 4.x)

#### Основные возможности:

|Функция|Описание|
|---|---|
|📁 Обмен файлами|Доступ к общим папкам из Windows, macOS, Linux|
|🖨️ Общий доступ к принтерам|Печать с любых устройств в сети|
|🔐 Аутентификация|Поддержка локальных пользователей, домена AD, LDAP|
|🌐 Интеграция с AD|Участие в домене Windows как клиент или контроллер|
|⚙️ Гибкая настройка|Детальный контроль прав доступа, шифрования, аудита|

#### Типичные сценарии использования:

1. Файловый сервер для смешанной сети;
2. Домашняя сеть: доступ к медиа с любого устройства:
	- Телевизор (DLNA через SAMBA);
	- Смартфон;
	- Ноутбук на Windows.
3. Корпоративная среда:
	- Интеграция Linux-сервера в Active Directory;
	- Централизованное управление пользователями и политиками;
	- Резервное копирование общих ресурсов.
4. Контроллер домена на базе Linux:
	- Domain Controller;
	- DNS-сервера;
	- Сервера политик групп (GPO — с ограничениями).

#### Kerberos:

**Kerberos** — это сетевой протокол аутентификации, который позволяет пользователям и службам доказывать свою личность друг другу **без передачи пароля по сети**.

В Kerberos есть центральный доверенный арбитр — **KDC (Key Distribution Center):**

- Клиент **никогда** не отправляет пароль серверу, к которому хочет подключиться (файловому шару, почте, веб-сайту);
- Клиент получает у KDC специальный криптографический билет (**Ticket**), который говорит: «Этот пользователь проверен, пустите его»;
- Сервер верит KDC, проверяет билет и пускает пользователя.

В среде Samba AD все эти роли объединены в одном процессе (`samba`), но логически они разделены:

1. **KDC (Key Distribution Center):** Центр распределения ключей. В Samba это часть контроллера домена. Состоит из двух частей:
    - **AS (Authentication Server):** Проверяет ваш пароль при входе и выдает первый билет;
    - **TGS (Ticket Granting Server):** Выдает билеты к конкретным сервисам (файлы, принтеры) на основе первого билета.
2. **Client:** Пользователь или служба, которая хочет получить доступ;
3. **Service (Server):** Ресурс, к которому нужен доступ (файловый сервер, веб-сайт);
4. **Realm:** Имя вашей области безопасности (обычно домен в верхнем регистре, например `OFFICE.SSA2026.REGION`);
5. **Principal:** Уникальное имя субъекта в Kerberos. Формат: `имя/тип@REALM`:
    - Пользователь: `administrator@OFFICE.SSA2026.REGION`;
    - Сервис: `host/dc-a.office.ssa2026.region@OFFICE.SSA2026.REGION`;
    - Сервис HTTP: `HTTP/webserver.office.ssa2026.region@OFFICE.SSA2026.REGION`.
6. **Keytab:** Файл, содержащий зашифрованные ключи сервиса. Позволяет сервису работать без ввода пароля (автоматическая аутентификация).

**Процесс аутентификации в Kerberos:**

1. Получение TGT (Ticket Granting Ticket):
	1. 1. Клиент отправляет запрос в KDC: «Я пользователь `admin`, хочу войти». (Пароль еще не отправлен!);
	2. KDC находит пользователя в базе, генерирует случайный сеансовый ключ;
	3. KDC шифрует ответ **паролем пользователя** (точнее, хэшем пароля). Внутри ответа лежит **TGT** и сеансовый ключ;
	4. Клиент получает ответ, просит у пользователя пароль, расшифровывает ответ:
	    - Если пароль верный -> расшифровка успешна -> у клиента есть TGT;
	    - Если пароль неверный -> ошибка «Preauthentication failed».
	5. TGT сохраняется в кэш клиента (в Linux это обычно файл `/tmp/krb5cc_<uid>` или keyring). Действует обычно 10 часов.
2. Получение Service Ticket (ST):
	1. Клиент обращается к KDC (к части TGS): «У меня есть TGT, дайте мне билет для доступа к сервису `cifs/fs01`»;
	2. Клиент отправляет TGT (зашифрованный ключом KDC) и запрос;
	3. KDC проверяет TGT (он может его прочитать, так как сам его зашифровал);
	4. KDC создает **Service Ticket** для `cifs/fs01`. Этот билет зашифрован **ключом самого сервиса fs01** (который хранится в его keytab или памяти). Клиент не может прочитать этот билет;
	5. KDC отдает билет клиенту.
3. Доступ к сервису:
	1. Клиент отправляет серверу `fs01` два вещи:
	    - Service Ticket (зашифрованный ключом сервера);
	    - Аутентификатор (timestamp, зашифрованный сеансовым ключом).
	2. Сервер `fs01` расшифровывает билет своим ключом (из keytab). Видит там имя пользователя и сеансовый ключ;
	3. Сервер расшифровывает аутентификатор сеансовым ключом. Сверяет время (защита от replay-атак);
	4. **Успех!** Сервер знает, кто вы, не зная вашего пароля. Сессия установлена.

#### Шаг 3. Настройка BIND9:

**Отключение chroot:**

```bash
control bind-chroot disabled
```

**bind-chroot** — это механизм безопасности, который запускает DNS-сервер BIND в изолированной среде ("chroot jail").

**Что такое chroot:**

- `chroot` (change root) — изменение корневой директории процесса;
- Процесс видит только файлы внутри указанной директории;
- Даже если злоумышленник получит доступ к процессу, он не сможет выйти за пределы этой "тюрьмы".

**Почему отключаем:**

- При использовании BIND9_DLZ плагина с Samba, BIND должен иметь доступ к файлам Samba (например, `/var/lib/samba/bind-dns/`);
- Эти файлы находятся вне стандартной chroot-директории BIND (обычно `/var/named/chroot`);
- Отключение chroot упрощает настройку интеграции, хотя немного снижает уровень безопасности.

**Отключение KRB5RACHETYPE:**

```bash
echo 'KRB5RACHETYPE="none"' >> /etc/sysconfig/bind
```

**Подключение плагина BIND9_DLZ:**

```bash
echo 'include "/var/lib/samba/bind-dns/named.conf";' >> /etc/bind/named.conf
```

**DLZ** (Dynamically Loadable Zones) — это модуль BIND, который позволяет загружать DNS зоны из внешних источников баз данных, а не из статических файлов.

**В случае с Samba AD DC:**

- BIND9_DLZ читает DNS записи напрямую из базы данных LDAP Active Directory;
- Изменения в AD автоматически отражаются в DNS;
- Не требуется ручная синхронизация между Samba и BIND.

**Сравнение backend-ов DNS для Samba AD DC:**

#### Шаг 4. Редактирование `/etc/bind/options.conf`:

```vim
key-gssapi-keytab "/var/lib/samba/bind-dns/dns.keytab";

allow-recursion { any; };
```

- **allow-query** - подсети, которым разрешены DNS-запросы;
- **allow-recursion** - подсети, которым разрешены рекурсивные запросы;
- **forwarders** - внешние DNS-серверы для пересылки запросов;
- **forward first** - сначала пересылать, затем кэшировать.

### Конфигурация Kerberos для Samba:

Конфигурация Kerberos для Samba AD DC — это не просто настройка одного файла. Это экосистема, состоящая из:

1. Глобального конфига клиента (`/etc/krb5.conf`) — чтобы система и утилиты знали, как общаться с KDC;
2. Внутренней конфигурации Samba `smb.conf` — где настраиваются политики работы KDC;
3. Keytab файлов — ключей для сервисов (BIND, SSH, HTTP);
4. Политик паролей и шифрования.
#### Настройка `/etc/krb5.conf`:

Этот файл читается всеми приложениями в Linux, использующими библиотеку Kerberos (`libkrb5`). При установке Samba DC через `samba-tool domain provision` он создается автоматически, но его **критически важно проверить и адаптировать**.

```vim
[libdefaults]
    # Имя вашего домена (Realm) ВСЕГДА в верхнем регистре
    default_realm = OFFICE.SSA2026.REGION

    # Разрешить поиск контроллеров домена через DNS SRV записи
    # Это критично для отказоустойчивости! Если один DC упадет, клиент найдет другой.
    dns_lookup_realm = false
    dns_lookup_kdc = true

    # Время жизни билетов
    ticket_lifetime = 24h
    renew_lifetime = 7d
    forwardable = true
    rdns = false

    # Типы шифрования. Samba по умолчанию предпочитает AES.
    # Избегайте weak crypto (rc4-hmac), если не требуется поддержка очень старых Windows (XP/2003).
    default_ccache_name = KEYRING:persistent:%{uid}
    permitted_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96

    # Синхронизация времени (допуск)
    clockskew = 300

[realms]
    OFFICE.SSA2026.REGION = {
        # Явное указание KDC (опционально, если dns_lookup_kdc = true)
        # kdc = dc-a.office.ssa2026.region
        # kdc = dc-b.office.ssa2026.region
        
        # Сервер для смены паролей
        admin_server = dc-a.office.ssa2026.region
        
        # Доменное имя (в нижнем регистре)
        default_domain = office.ssa2026.region
    }

[domain_realm]
    # Маппинг доменных имен на Realm
    .office.ssa2026.region = OFFICE.SSA2026.REGION
    office.ssa2026.region = OFFICE.SSA2026.REGION

[logging]
    # Логирование ошибок Kerberos (помогает при отладке)
    default = FILE:/var/log/krb5libs.log
    kdc = FILE:/var/log/krb5kdc.log
    admin_server = FILE:/var/log/kadmind.log
```

**Ключевые параметры и нюансы:**

- **`dns_lookup_kdc = true`**: Самая важная строка. Она заставляет клиент искать `_kerberos._tcp.OFFICE.SSA2026.REGION` в DNS. Без этого при падении основного DC вся сеть встанет, так как клиенты будут стучаться только на жестко прописанный IP.
- **`default_ccache_name`**: Определяет, где хранятся билеты пользователя.
    - `FILE:/tmp/krb5cc_%{uid}` — классический файл (исчезает после перезагрузки/логина).
    - `KEYRING:persistent:%{uid}` — хранение в ядре Linux (более безопасно и надежно между сессиями). Требует пакета `keyutils`.
- **`permitted_enctypes`**: Список разрешенных алгоритмов шифрования.
    - **AES256/AES128**: Стандарт де-факто. Используйте их.
    - **RC4-HMAC**: Устаревший, небезопасный. Включайте только если у вас есть legacy-системы (старые принтеры, сканеры, Windows XP/7 без обновлений). В Samba можно отключить globally в `smb.conf`.

#### Настройка `/etc/samba/smb.conf`:

Параметры работы Kerberos внутри Samba задаются в секции `[global]`:

```vim
[global]
    # Работа в режиме домена AD
    security = ADS
    realm = OFFICE.SSA2026.REGION
    workgroup = OFFICE

    # Интеграция с Kerberos
    kerberos method = secrets and keytab
    # "secrets" - использует внутренний storage Samba
    # "keytab" - использует системный keytab (/etc/krb5.keytab)
    # "dedicated keytab" - использует отдельный файл (редко нужно)

    # Принудительное использование локального KDC (для самого DC)
    # Обычно не требуется, если krb5.conf настроен верно через DNS.

    # Ограничение типов шифрования (Безопасность!)
    # Отключаем NTLMv1 и слабые методы
    ntlm auth = no-plaintext no-ntlm-v1
    server min protocol = SMB2_02
    
    # Запрет слабого шифрования Kerberos (RC4)
    # В новых версиях Samba это может быть, но лучше явно указать:
    tls enabled = yes # Для LDAP over SSL, косвенно влияет на безопасность
    
    # Настройка SPN (Service Principal Names)
    # Samba автоматически регистрирует SPN для себя, но для других сервисов нужно следить.
```

> Параметр `kerberos method = secrets and keytab` является рекомендуемым для контроллеров домена Samba. Он позволяет гибко управлять ключами.

#### Работа с Keytab (Таблицы ключей):

**Keytab** — это файл, содержащий пары `Principal:EncryptedKey`. Он нужен сервисам (даémon'ам), чтобы аутентифицироваться в домене без участия человека (без ввода пароля).

**Где хранятся ключи?**

- **Системный keytab:** `/etc/krb5.keytab` (обычно для хоста `host/fqdn`);
- **Специфичные keytab:** Например, `/var/lib/samba/bind-dns/dns.keytab` для BIND9.

**Просмотр содержимого keytab:**

```bash
klist -k /etc/krb5.keytab
# или
klist -kte /etc/krb5.keytab  # покажет типы шифрования и время создания
```

**GSS-API** (Generic Security Services Application Program Interface) — стандартный интерфейс для сервисов безопасности. Позволяет приложениям использовать различные механизмы безопасности (Kerberos, SPNEGO и др.)

**Как работает keytab в контексте Samba+BIND:**

1. Samba создает специальный principal для DNS: `DNS/dc-a.office.ssa2026.region@OFFICE.SSA2026.REGION`;
2. Генерируется keytab файл с ключом этого principal'а;
3. BIND использует этот keytab для аутентификации перед Samba/Kerberos;
4. Это позволяет BIND динамически обновлять DNS записи в Active Directory.


**Установка прав доступа:**

Keytab содержит секретные ключи! Доступ должен быть только у владельца сервиса.

```bash
chown root:root /etc/krb5.keytab
chmod 600 /etc/krb5.keytab

chown named:named /var/lib/samba/bind-dns/dns.keytab
chmod 600 /var/lib/samba/bind-dns/dns.keytab
```


##### Проверка сервиса (SPN):

**SPN** — это уникальное имя сервиса в сети Kerberos:
* Формат: `Сервис/ИмяХоста@ДОМЕН`;
* Пример: `HTTP/webserver.office.ssa2026.region@OFFICE.SSA2026.REGION`

Убедитесь, что у сервиса зарегистрирован правильный SPN в AD.

```bash
# Посмотреть все SPN у пользователя/компьютера
samba-tool spn list administrator

# Посмотреть SPN для компьютера
samba-tool spn list host/dc-a

# Посмотреть SPN для конкретного сервиса
samba-tool spn list HTTP/webapp

# Добавить SPN (например, для веб-сервера на аккаунте web_svc)
samba-tool spn add HTTP/web.office.ssa2026.region web_svc

# Удалить SPN
samba-tool spn delete HTTP/web.old.site web_svc
```

##### Частые проблемы и решения:

| Проблема                          | Причина                                                 | Решение                                                                                           |
| --------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **Clock skew too great**          | Время на клиенте и сервере отличается > 5 мин.          | Настройте `chrony` или `ntp`. Убедитесь, что часовой пояс верный.                                 |
| **Preauthentication failed**      | Неверный пароль.                                        | Перепроверьте раскладку, CapsLock. Сбросьте пароль в AD.                                          |
| **Cannot find KDC for realm**     | DNS не находит SRV записи.                              | Проверьте `/etc/resolv.conf` (должен указывать на DC). Проверьте `dns_lookup_kdc = true`.         |
| **Encryption type not supported** | Клиент хочет AES, сервер предлагает RC4 (или наоборот). | Проверьте `permitted_enctypes` в `krb5.conf` и настройки шифрования в AD.                         |
| **Success, но доступ запрещен**   | Билет получен, но нет прав в ACL или неверный SPN.      | Проверьте `samba-tool spn list`. Проверьте права на ресурс (файловую шару).                       |
| **Билеты быстро протухают**       | Неправильное время жизни в конфигах.                    | Увеличьте `ticket_lifetime` в `krb5.conf` и политики в AD (`samba-tool domain passwordsettings`). |

### Обеспечение отказоустойчивости и репликация:

Обеспечение отказоустойчивости (High Availability, HA) и репликация — это критически важные аспекты при построении инфраструктуры на базе Samba AD DC. В отличие от некоторых других систем, Samba не имеет встроенного механизма "мастер-слейв" с автоматическим переключением IP (как VRRP для самого сервиса AD), поэтому отказоустойчивость достигается за счет **множественности контроллеров домена** и правильной настройки клиентов.

Samba Active Directory использует модель **Multi-Master Replication** (многосторонняя репликация). Это означает:

- **Равноправие:** Все контроллеры домена (DC) равны. Нет понятия "главный" или "вторичный" DC для записи изменений;
- **Запись в любой узел:** Вы можете создать пользователя, изменить пароль или групповую политику на _любом_ доступном DC, и эти изменения будут реплицированы на остальные;
- **Отказ одного узла:** Если один DC падает, клиенты автоматически переключаются на другой (при правильной настройке DNS).

#### Репликация:

Samba использует протокол **DRS (Directory Replication Service)**, тот же самый, что и в Windows Server Active Directory.

**Типы репликации:**

1. **Intra-site (Внутри сайта):**
    - Происходит часто (по умолчанию каждые 15 секунд после изменения или каждые 3 минуты планово);
    - Использует сжатие данных;
    - triger-ится изменениями (Change Notification).
2. **Inter-site (Между сайтами):**
    - Если у вас несколько физических локаций (офисов), их можно объединить в разные "Sites" в AD;
    - Репликация происходит реже (по расписанию, например, раз в час) для экономии трафика;
    - Требует ручной настройки объектов "Site Link" в AD.

**Что реплицируется:**

- **LDAP Database:** Пользователи, группы, компьютеры, политики;
- **SYSVOL:** Скрипты входа, групповые политики (GPO). Для репликации SYSVOL в современных версиях Samba используется механизм **DFSR** (аналог Windows DFS-R) или **rsync** (в старых версиях, сейчас рекомендуется только DFSR);
- **DNS Zones:** Записи обновляются через безопасные динамические обновления (Secure Dynamic Updates) через BIND9_DLZ.

#### Добавление второго (резервного) контроллера домена:

##### Шаг 1. Установка пакетов:

```bash
apt-get update && apt-get install -y task-samba-dc bind bind-utils
```

##### Шаг 2. Присоединение к существующему домену:

```bash
samba-tool domain join office.ssa2026.region DC -U Administrator --dns-backend=BIND9_DLZ
```

**Разбор параметров:**

- `office.ssa2026.region`: Имя вашего домена (Realm).
- `DC`: Роль сервера (Domain Controller).
- `-Uadministrator`: Учетная запись администратора существующего домена.
- `--dns-backend=BIND9_DLZ`: Важно указать тот же бекенд, что и на первом сервере.

**Что происходит в этот момент:**

1. Сервер аутентифицируется на первом DC;
2. Скачивает всю базу данных AD (LDAP, Sysvol, DNS зоны);
3. Настраивает локальный `smb.conf` и `krb5.conf`;
4. Генерирует свои ключи Kerberos и регистрируется в DNS как новый DC.


##### Шаг 3. Запуск служб:

```bash
systemctl enable --now samba
systemctl enable --now bind
```

##### Шаг 4. Проверка репликации:

Через несколько минут (или сразу, если изменений мало) проверяем статус репликации на любом из контроллеров:

```bash
samba-tool drs showrepl
```

- **Success count:** Должен быть > 0;
- **Last attempt @ ... was successful:** Статус последней попытки должен быть успешным;
- **Naming Context:** Должны реплицироваться все контексты (Schema, Configuration, Domain);


> Если есть ошибки, проверяем связанность между серверами (порт 389, 636, 88, 53, 3268 и динамические порты RPC).


#### Обеспечение отказоустойчивости для клиентов:

**Samba не предоставляет "виртуальный IP" для домена.** Отказоустойчивость обеспечивается на стороне клиента через DNS. Клиенты (Windows, Linux) должны знать адреса **всех** контроллеров домена.

**Указание ip обоих DC в сетевых настройках:**

1. **В DHCP сервере:** Укажите в опции DNS (Option 6) адреса обоих контроллеров:

```Text
DNS Server 1: 192.168.1.10 (dc-a)
DNS Server 2: 192.168.1.11 (dc-b)
```

2. **Статическая настройка:** Пропишите оба DNS в сетевых настройках серверов и рабочих станций.

> _Порядок важен:_ Клиент попробует первый. Если он не отвечает (timeout), перейдет ко второму.

**Поведение клиента при отказе одного из DC:**

1. Клиент запрашивает у DNS запись `_ldap._tcp.dc._msdcs.office.ssa2026.region`;
2. DNS возвращает список всех доступных DC (и `dc-a`, и `dc-b`);
3. Клиент выбирает DC (обычно ближайший по метрике сайта или первый в списке);
4. Если выбранный DC не отвечает, клиент автоматически пробует следующий из списка;
5. После восстановления упавшего DC, клиенты постепенно начнут балансировать нагрузку или вернутся к нему (в зависимости от настроек сайта).

### Управление ролями FSMO:

В Active Directory есть 5 уникальных ролей, которые **должны выполняться только одним контроллером домена в любой момент времени**. Это не противоречит модели Multi-Master — эти роли нужны для операций, которые нельзя выполнять параллельно.

| Роль                      | Область действия | Описание                                                                                                                                                                                                                                                                                                 |
| ------------------------- | ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Schema Master**         | Лес (Forest)     | Единственный DC, который может изменять схему AD (добавлять атрибуты, классы). Критична при установке нового ПО (Exchange, SCCM и т.д.).                                                                                                                                                                 |
| **Domain Naming Master**  | Лес (Forest)     | Управляет добавлением/удалением доменов в лесу. Нужна редко, но критична при расширении инфраструктуры.                                                                                                                                                                                                  |
| **PDC Emulator**          | Домен (Domain)   | Эмулирует поведение старого PDC из Windows NT. Отвечает за:  <br>• Синхронизацию времени в домене  <br>• Обработку изменений паролей (немедленная репликация)  <br>• Блокировку учетных записей после неудачных попыток входа  <br>• Групповые политики (GPO) — редактирование по умолчанию идет на PDCe |
| **RID Master**            | Домен (Domain)   | Выдает пулы RID (Relative ID) другим DC. Каждый объект в AD имеет уникальный SID = Domain-SID + RID. Без RID Master новые пользователи/компьютеры не смогут создаваться.                                                                                                                                 |
| **Infrastructure Master** | Домен (Domain)   | Обновляет ссылки между объектами из разных доменов (в многодоменной среде). В однодоменной среде эта роль практически не используется.                                                                                                                                                                   |

##### Просмотр текущего расположения ролей:

```bash
samba-tool fsmo show
```

_Пример вывода:_

```Text
Role                   Owner
Schema                 dc-a.office.ssa2026.region
Domain Naming          dc-a.office.ssa2026.region
PDC                    dc-a.office.ssa2026.region
RID                    dc-a.office.ssa2026.region
Infrastructure         dc-a.office.ssa2026.region
```

##### Перенос ролей (Transfer) — плановая операция:

Используется, когда старый DC еще жив, и вы хотите перенести роли на новый (например, при замене оборудования).

```bash
# Перенести все роли на dc-b
samba-tool fsmo transfer --target-dc=dc-b.office.ssa2026.region

# Или по одной роли:
samba-tool fsmo transfer --role=PDC --target-dc=dc-b.office.ssa2026.region
samba-tool fsmo transfer --role=RID --target-dc=dc-b.office.ssa2026.region
```

##### Захват ролей (Seize) — аварийная операция:

Используется, когда старый DC **умер навсегда** и его нельзя восстановить. Это опасная операция — если старый DC потом включится, возникнет конфликт и возможное повреждение данных.

```bash
# Захватить все роли на dc-b
samba-tool fsmo seize --force

# Или по одной:
samba-tool fsmo seize --role=PDC --force
```

> **Важно!** Перед захватом убедитесь, что старый DC никогда не будет возвращен в сеть. Если он вернется — немедленно удалите его метаданные из AD:

```bash
samba-tool domain level raise --dry-run # проверить уровень
samba-tool computer delete "DC-A" --force # удалить запись компьютера
```

##### Лучшие практики по FSMO:

- Храните роли на самом производительном и надежном DC;
- Не держите все роли на одном DC, если у вас больше двух контроллеров;
- Регулярно проверяйте статус ролей (`fsmo show`) и репликации (`drs showrepl`);
- Документируйте, где находятся роли, и кто имеет право их переносить/захватывать.

### Сайт-администрирование (Active Directory Sites and Services):

##### Основные понятия:

Если контроллеры домена расположены в разных физических локациях (офисы, дата-центры, облака), **обязательно** нужно настроить сайты. Иначе клиенты могут подключаться к удаленному DC через медленный канал, а репликация будет забивать весь трафик.

- **Site (Сайт)** — логическая группа хорошо связанных сетей (высокая скорость, низкая задержка);
- **Subnet (Подсеть)** — IP-диапазон, привязанный к сайту. Клиенты автоматически определяют свой сайт по IP;
- **Site Link (Канал между сайтами)** — определяет, как часто и с какой стоимостью происходит репликация между сайтами;
- **Bridgehead Server** — выбранный DC в каждом сайте, который отвечает за межсайтовую репликацию.

#### Настройка сайтов через `samba-tool`:

##### Шаг 1. Создать сайты:

```bash
samba-tool sites create Site-Moscow
samba-tool sites create Site-Spb
```

##### Шаг 2. Создать подсети и привязать к сайтам:

```bash
# Подсеть Москвы
samba-tool sites subnet-create 192.168.10.0/24 Site-Moscow

# Подсеть Петербурга
samba-tool sites subnet-create 192.168.20.0/24 Site-Spb
```

Когда вы создаете объект подсети (`192.168.10.0/24`) и привязываете его к сайту `Site-Moscow`, происходит следующее:

1. Клиент в Москве (IP `192.168.10.55`) запрашивает контроллер домена;
2. В ответе от DNS содержится не просто список IP, а информация о сайтах;
3. Клиент смотрит на свой IP, понимает: «Я принадлежу подсети `192.168.10.0/24`, которая ассоциирована с сайтом `Site-Moscow`»;
4. Клиент фильтрует список и выбирает **только** те контроллеры, которые находятся в сайте `Site-Moscow`;
5. **Результат:** Клиент всегда подключается к локальному `DC-MOW`. Трафик не уходит в другой город.

##### Шаг 3. Привязать контроллеры доменов к сайтам:

```bash
# Предположим, dc-a находится в Москве, dc-b — в Петербурге
samba-tool sites move-server dc-a.office.ssa2026.region Site-Moscow
samba-tool sites move-server dc-b.office.ssa2026.region Site-Spb
```

##### Настройка сайт-линков (опционально):

По умолчанию создается линк `DEFAULTIPSITELINK` с интервалом 180 минут и стоимостью 100. Изменить интервал репликации (например, на 30 минут):

```bash
samba-tool sites link-set-options DEFAULTIPSITELINK --interval=30
```

Увеличить стоимость для медленного канала (чтобы репликация шла реже):

```bash
samba-tool sites link-set-options DEFAULTIPSITELINK --cost=200
```

**Как это влияет на работу:**

- **Клиенты**: Автоматически находят ближайший DC (по сайту) для аутентификации и запросов.
- **Репликация**: Внутри сайта — каждые 15 сек. Между сайтами — по расписанию (например, раз в 30 мин).
- **Трафик**: Межсайтовая репликация сжимается и планируется на ночное время (можно настроить).


##### Назначение Bridgehead (опционально):

В Samba (как и в Windows AD) есть два подхода: **автоматический** (по умолчанию) и **ручной** (Preferred Bridgehead).

**Вариант 1. Автоматический:**

По умолчанию служба **KCC (Knowledge Consistency Checker)** сама выбирает лучший сервер в качестве моста:

- Она учитывает загруженность сервера;
- Если выбранный мост падает, KCC автоматически выбирает нового;
- **Вам ничего делать не нужно.** Это самый надежный вариант для большинства сценариев.

**Вариант 2. Ручное назначение (Preferred Bridgehead):**

Используется если нужно жестко контролировать трафик. Например, у есть мощный сервер с выделенным каналом связи, и нужно чтобы репликация ходила только через него.

> **Важно!** Если назначить ручной _Bridgehead_ и этот сервер упадет, репликация между сайтами **остановится**, пока кто-то вручную не назначит новый. Автоматического переключения не будет!


Допустим, мы хотим сделать `dc-mow.office.ssa2026.region` предпочтительным мостом для сайта `Site-Moscow`.

**Шаг 1. Узнаем DN (Distinguished Name) сервера:**

```bash
# Найдем полный путь к объекту компьютера
samba-tool domain objectshow "dc-mow"
```

Находим строку `distinguishedName`. Обычно она выглядит так: `CN=DC-MOW,CN=Servers,CN=Site-Moscow,CN=Sites,CN=Configuration,DC=office,DC=ssa2026,DC=region`

**Шаг 2. Добавляем атрибут preferredBridgeheadServerFor:**

Нужно добавить этот сервер в список предпочтительных мостов для конкретного сайта. Редактируем объект самого **Сайта**:

```bash
samba-tool domain objectmodify "CN=Site-Moscow,CN=Sites,CN=Configuration,DC=office,DC=ssa2026,DC=region" --addattr="preferredBridgeheadServerFor:CN=DC-MOW,CN=Servers,CN=Site-Moscow,CN=Sites,CN=Configuration,DC=office,DC=ssa2026,DC=region"
```

**Что происходит:**

1. Когда выполняется команда `samba-tool domain objectmodify`, изменяется запись в базе данных LDAP (файл `sam.ldb`);
2. Эта запись хранится в разделе **Configuration**, который реплицируется на **все** контроллеры домена автоматически;
3. Как только выполнилась команда на `DC-1`, через несколько секунд (или минут, в зависимости от скорости репликации) это изменение появится на `DC-2`, `DC-3` и всех остальных.

Все контроллеры домена «видят» одну и ту же конфигурацию сайта. У них нет локальных настроек «кто тут главный», они все читают эту информацию из общей базы AD.

**Шаг 3. Принудительный пересчет топологии (опционально):**

Когда мы выполняем команду `samba-tool domain objectmodify`, изменяется запись в базе данных LDAP (файл `sam.ldb`). 

Обычно KCC срабатывает сам каждые 15 минут. Чтобы ускорить процесс на конкретном DC вводим:

```bash
samba-tool drs replicate <Имя_Текущего_DC> <Имя_Целевого_DC> --full-sync
```

**Шаг 4. Как удалить ручное назначение (вернуть автоматику):**

Если нужно чтобы Samba сама выбирала мост, нужно удалить все записи этого атрибута для данного сайта:

```bash
samba-tool domain objectmodify "CN=Site-Moscow,CN=Sites,CN=Configuration,DC=office,DC=ssa2026,DC=region" --delattr="preferredBridgeheadServerFor:<DN_Сервера>"
```

##### Резервное копирование и восстановление (Backup & Restore):

**Репликация — это НЕ бэкап!** Если случайно удалились _OU_ со всеми пользователями, это изменение мгновенно реплицируется на все DC, и везде потеряются данные.

**Что нужно бэкап-ить:**

1. База данных Samba (LDIF). Команда создает архив с:

- `sam.ldb.d/` — база данных LDAP;
- `sysvol/` — групповые политики и скрипты;
- `private/` — ключи, конфигурация.

```bash
samba_backup --destination=/backup/samba-backup-$(date +%F).tar.bz2
```

2. Конфигурационные файл:

```bash
tar czf /backup/samba-config-$(date +%F).tar.gz /etc/samba /etc/bind /etc/krb5.conf
```

3. Снимки файловой системы (если используете LVM/ZFS/Btrfs):

```bash
lvcreate -L 1G -s -n snap_samba /dev/vg0/lv_samba
```

**Авторитетное восстановление из бэкапа:**

**Шаг 1. Остановка службы Samba:**

```bash
systemctl stop samba
systemctl stop bind
```

**Шаг 2. Восстановление файлов из бэкапа:**

```bash
tar xjf /backup/samba-backup-2026-03-10.tar.bz2 -C /var/lib/samba/

# Убедитесь, что права доступа корректны
chown -R root:root /var/lib/samba/private
chmod 700 /var/lib/samba/private
```

**Шаг 3: Пометка объектов как «Авторитетных»:**

Active Directory использует механизм нумерации изменений — **USN (Update Sequence Number)**:
- Каждое изменение (создание пользователя, смена пароля) увеличивает счетчик USN на контроллере, где оно произошло.
- При репликации контроллеры сравнивают USN: «У меня версия 100, у тебя версия 105. Ок, я заберу у тебя новые данные».

Если просто восстановить файлы базы данных (`sam.ldb`) из бэкапа на работающем сервере без специальной подготовки, произойдет одно из двух:

1. **Восстановленные данные будут перезаписаны старыми данными** с других контроллеров домена (так как у них номер версии USN выше). Вы потеряете восстановленные объекты почти сразу;
2. Возникнет **конфликт репликации** («USN Rollback»), который может повредить весь домен.

Перед запуском службы нужно запустить утилиту `samba-tool dbcheck` со специальным флагом, которая пересчитает USN для восстановленных объектов, сделав их «новее» всех остальных в сети.

**Вариант 1. Восстановить ВСЁ (полный бэкап домена):**

Если откатывается весь контроллер до состояния бэкапа:

```bash
samba-tool dbcheck --cross-ncs --fix --attrs=uSNChanged,uSNCreated
```

- `--cross-ncs`: Проверять все разделы (Domain, Configuration, Schema).
- `--fix`: Исправлять несоответствия (в данном случае — повышать USN).
- `--attrs=uSNChanged`: Именно этот атрибут заставляет другие DC принять эти данные как новые.

**Вариант 2. Восстановить только удаленные объекты (точечное восстановление):**

Если база актуальна, но вы хотите вернуть конкретную удаленную OU или пользователя:

```bash
samba-tool dbcheck --cross-ncs --fix --attrs=uSNChanged --base="DN_ОБЪЕКТА"
```

> Пример: `--base="OU=Users,DC=office,DC=ssa2026,DC=region"`.


**Шаг 4. Запуск службы:**

Теперь запускаем службы. При старте Samba увидит, что USN у локальных объектов очень высокий, и при первой же репликации отправит эти данные на соседние контроллеры, перезаписав у них информацию о удалении.

```bash
systemctl start samba
systemctl start bind
```

**Шаг 5: Принудительная репликация (опционально):**

```bash
# С DC-Restore тянем данные на DC-Normal (или наоборот, в зависимости от топологии)
# Обычно достаточно просто подождать 1-2 минуты, но для уверенности:
samba-tool drs replicate DC-Normal DC-Restore --full-sync
```
