
**NTP (Network Time Protocol)** — это протокол прикладного уровня (UDP, порт **123**), предназначенный для синхронизации внутренних часов компьютера с серверами времени в сети.

#### Ключевые концепции:

- **Stratum (Стратум):** Иерархия точности.
    - **Stratum 0:** Высокоточные источники времени (атомные часы, GPS-приемники). Не подключены к сети напрямую.
    - **Stratum 1:** Серверы, напрямую подключенные к Stratum 0. Это основные серверы времени в интернете.
    - **Stratum 2:** Серверы, синхронизирующиеся со Stratum 1. Именно их обычно используют обычные пользователи и компании.
    - И так далее до Stratum 15. Stratum 16 означает, что время не синхронизировано.
- **Jitter (Джиттер):** Вариация задержки пакетов. Высокий джиттер ухудшает точность синхронизации.
- **Offset (Смещение):** Разница между локальным временем и временем сервера.

#### Выбор реализации:

| Характеристика       | `ntpd` (Classic NTP)                          | `chronyd` (Chrony)                                                               |
| -------------------- | --------------------------------------------- | -------------------------------------------------------------------------------- |
| **История**          | Старый, проверенный стандарт.                 | Более современный, разработан для систем, которые не работают постоянно.         |
| **Синхронизация**    | Медленнее сходится после запуска.             | Очень быстрая синхронизация.                                                     |
| **Плавающие часы**   | Хуже справляется с большими скачками времени. | Отлично компенсирует дрейф часов, даже если система была выключена.              |
| **Ресурсы**          | Потребляет немного больше ресурсов в простое. | Легковесный.                                                                     |
| **Где использовать** | Стабильные серверы, работающие 24/7.          | Виртуальные машины (VM), ноутбуки, контейнеры, системы с частыми перезагрузками. |

### Настройка Chrony:

Как ты спрашивал ранее, `chronyd` использует порт **323/UDP** для общения с клиентами (`chronyc`) и для внутреннего управления. Порт **123/UDP** используется для самого протокола NTP.

Установка пакетов:

```bash
sudo apt-get install chrony
```

Основной конфиг: `/etc/chrony/chrony.conf` (или `/etc/chrony.conf`)

```vim
# /etc/chrony/chrony.conf

# --- ИСТОЧНИКИ ВРЕМЕНИ (SERVERS) ---
# Используем пул. iburst ускоряет начальную синхронизацию.
# maxsources ограничивает количество активных источников из пула (по умолчанию 4).
pool ru.pool.ntp.org iburst maxsources 4

# Или использовать конктретный server. Можно указать доменное имя или ip

# Можно добавить конкретные надежные серверы (например, от провайдера или stratum-1)
# server ntp1.example.com iburst prefer
# prefer — отдает предпочтение этому серверу, если он доступен.

# --- ЛОКАЛЬНЫЙ РЕЖИМ (LOCAL REFERENCE) ---
# Если все внешние источники недоступны, chronyd будет считать себя источником времени.
# stratum 10 — достаточно низкий приоритет, чтобы клиенты понимали, что время не эталонное.
# Это критично, если этот сервер является NTP-сервером для локальной сети.
local stratum 10

# --- ДОСТУП КЛИЕНТОВ (ACCESS CONTROL) ---
# Разрешаем синхронизацию только доверенной подсети.
# По умолчанию chrony запрещает все входящие запросы, кроме localhost.
allow 192.168.1.0/24
allow 10.0.0.0/8

# Запрещаем конкретные IP (если нужно)
# deny 192.168.1.50

# --- КОРРЕКЦИЯ ЧАСОВ (CLOCK CORRECTION) ---
# makestep 1.0 3
# Если смещение больше 1 секунды, сделать резкий шаг (step) в первых 3 обновлениях.
# После этого chrony переходит в режим плавной подстройки (slew).
# Это важно для систем, которые долго были выключены или имеют сильное рассинхронизирование при старте.
makestep 1.0 3

# --- ДРЕЙФ ЧАСОВ (DRIFT FILE) ---
# Файл, где хранится информация о скорости дрейфа системных часов.
# Позволяет chrony быстрее синхронизироваться после перезагрузки.
driftfile /var/lib/chrony/drift

# --- КЛЮЧИ БЕЗОПАСНОСТИ (OPTIONAL) ---
# Для аутентификации NTP-пакетов (защита от подмены времени).
# keyfile /etc/chrony/chrony.keys

# --- ЛОГИРОВАНИЕ ---
# log tracking measurements statistics
# logdir /var/log/chrony
# В продакшене логи лучше писать аккуратно, они могут расти.
# tracking: смещение, частота, стратум.
# measurements: данные от серверов.
# statistics: статистика производительности.

# --- RTCSYNC (ТОЛЬКО ДЛЯ LINUX) ---
# Синхронизирует системное время с аппаратными часами (RTC) каждые 11 минут.
# Полезно, чтобы после перезагрузки BIOS/UEFI показывал правильное время.
rtcsync
```

##### iburst (Initial Burst):

Обычно NTP-клиент отправляет один пакет запроса серверу и ждет ответа. Если ответа нет или он идет долго, клиент ждет несколько минут перед следующей попыткой. Это стандартное поведение для экономии трафика и нагрузки.

С опцией `iburst` при **первом запуске** службы (или при потере связи с сервером) Chrony отправляет не один пакет, а **пакет из 8 запросов** с очень короткими интервалами.

**Зачем это нужно:**

1. **Ускорение синхронизации:** Без `iburst` полная синхронизация может занять до 10–15 минут. С `iburst` система может синхронизироваться за **10–20 секунд**.
2. **Быстрое восстановление:** Если сеть моргнула и связь с сервером пропала, при восстановлении канала `iburst` поможет быстрее вернуться в строй.

##### prefer (Предпочтительный):

Эта метка говорит алгоритму выбора источников Chrony: "Если этот сервер доступен и его данные не выглядят явно ошибочными (не являются 'falseticker'), отдавай ему предпочтение перед другими источниками".

Chrony собирает данные со всех настроенных серверов. Он использует алгоритмintersection algorithm (алгоритм пересечения), чтобы отсеять серверы, которые сильно врут.

- Если у тебя 3 сервера, и два показывают время `12:00:00`, а один `12:05:00`, Chrony отбросит третий как неисправный.
- Если у тебя есть сервер с флагом `prefer`, и его время находится в пределах допустимого диапазона (не является явной ложью), Chrony будет использовать именно его как основной источник для расчета системного времени, даже если другие серверы чуть-чуть точнее статистически.

**Зачем это нужно:**

1. **Локальный эталон:** У тебя есть свой локальный сервер времени (Stratum 1 или 2) внутри сети, который подключен к GPS/ГЛОНАСС. Ты хочешь, чтобы все машины в офисе синхронизировались именно с ним, а не ходили в интернет. Но если локальный сервер умрет, они должны переключиться на публичные пулы.

```vim
server 192.168.1.10 iburst prefer  # Локальный GPS-сервер
pool ru.pool.ntp.org iburst        # Резервный интернет-пул
```

`prefer` **не означает** "использовать только его". Если сервер с `prefer` начнет сильно врать (его offset выйдет за допустимые пределы), Chrony **отключит** его и переключится на другие источники. Флаг `prefer` работает только тогда, когда качество сигнала от этого сервера сопоставимо с остальными.

#### Управление и проверка:

```bash
sudo systemctl restart chrony
sudo systemctl enable chrony

# Проверка источников
chronyc sources -v

# Проверка статуса синхронизации
chronyc tracking
```


### Best-practice:

#### Выбор источников (Sources):

1. **Не используй только один сервер.** Всегда используй минимум 3–4 источника. Алгоритм NTP выбирает наиболее точные, отбрасывая "лжецов" (falsetickers).
2. **Географическая близость.** Используй пулы своей страны (`ru.pool.ntp.org`) или провайдера. Чем меньше задержка (latency), тем точнее синхронизация.
3. **Избегай публичных `pool.ntp.org` без региона.** Лучше `europe.pool.ntp.org` или `ru.pool.ntp.org`.

#### Безопасность:

1. **Ограничь доступ (`allow`).** Никогда не оставляй `allow 0.0.0.0/0`, если это не публичный сервис. Открытый NTP-сервер может быть использован для **NTP Amplification DDoS-атак**.
2. **Rate Limiting.** Chrony имеет встроенную защиту от флуда. Можно настроить `ratelimit interval 2 burst 8` в новых версиях, чтобы ограничить количество ответов одному клиенту.

#### Виртуальные машины (KVM/Proxmox/VMware):

1. **Отключи синхронизацию времени от гипервизора внутри гостя.**
    - В Proxmox/KVM: Не ставь галочку "QEMU Guest Agent" для синхронизации времени, если внутри стоит `chrony`. Пусть `chrony` работает независимо.
    - В VMware Tools: Отключи синхронизацию времени хост-машиной, если используешь внутренний NTP. Конфликт двух источников синхронизации приводит к "дрожанию" часов.
2. **Используй `local stratum 10`.** Если хост-машина теряет связь с интернетом, ВМ не должны терять синхронизацию друг с другом.

#### Мониторинг:

1. **Следи за `Root Dispersion` и `Offset`.**
    - `Offset` должен быть в пределах миллисекунд (< 10ms для обычных задач, < 1ms для высоких нагрузок).
    - Если `Offset` скачет, проверь нагрузку на CPU и сеть.
2. **Команда `chronyc tracking`.**
	
	```bash
	Reference ID    : 192.168.1.1 (ntp1.local)
	Stratum         : 3
	Ref time (UTC)  : Tue Apr 07 12:00:00 2026
	System time     : 0.000123456 seconds fast of NTP time
	Last offset     : +0.000012345 seconds
	RMS offset      : 0.000054321 seconds
	Frequency       : 12.345 ppm slow
	Residual freq   : +0.001 ppm
	Skew            : 0.123 ppm
	Root delay      : 0.012345 seconds
	Root dispersion : 0.001234 seconds
	Update interval : 64.0 seconds
	Leap status     : Normal
	```
	
    - `Leap status: Not synchronized` — проблема!
    - `Frequency` — насколько быстро убегают часы. Если значение большое (> 100 ppm), возможно, проблема с аппаратными часами или драйверами.

### Подводные камни:

#### "Скачок" времени (Step vs Slew):

- **Проблема:** Если время отличается сильно (например, на 5 секунд), NTP может либо резко изменить его (**step**), либо плавно подстроить (**slew**).
- **Риск:** Резкое изменение времени может сломать логи, базы данных (PostgreSQL чувствителен к назад идущему времени!), Kerberos (билеты станут невалидными) и распределенные системы.
- **Решение:**
    - Используй `makestep 1.0 3` в `chrony.conf`. Это позволит сделать резкий шаг только при старте (первые 3 обновления), если разница > 1 сек.
    - В дальнейшем chrony будет использовать только плавную подстройку.
    - Если время ушло сильно назад во время работы, лучше перезапустить службу или вручную сделать `chronyc -a makestep`.

#### Конфликт с `systemd-timesyncd`:

- **Проблема:** В современных Linux (Ubuntu 20.04+, Debian 11+) есть легкий клиент `systemd-timesyncd`. Если ты установишь `chrony`, они могут конфликтовать за управление часами.
- **Решение:**
	
	```bash
	sudo systemctl disable --now systemd-timesyncd
	sudo systemctl enable --now chrony
	```
	
	Проверь, кто управляет временем:
	```bash
	timedatectl status
	```
	Должно быть: `System clock synchronized: yes` и `NTP service: active` (именно chrony).

#### Сетевые задержки и асимметрия маршрутов:

- **Проблема:** NTP предполагает, что путь туда и обратно занимает одинаковое время. Если маршрут асимметричен (через разные провайдеры), точность падает.
- **Решение:** Выбирай серверы с минимальным и стабильным пингом. Используй `iburst`.

#### Порт 323/UDP:

- **Напоминание:** Как ты спрашивал ранее, `chronyd` использует порт **323/UDP** для общения с клиентами (`chronyc`) и для внутреннего управления. Порт **123/UDP** используется для самого протокола NTP.
- **Фаервол:** Если ты подключаешься к серверу с командой `chronyc` удаленно, нужно открыть **323/UDP**. Для обычной синхронизации времени клиентам нужен только **123/UDP**.

#### DNS-зависимость:

- **Проблема:** Если в `chrony.conf` указаны имена хостов (`pool ru.pool.ntp.org`), а DNS недоступен при старте, chrony не сможет разрешить имена.
- **Решение:**
    - Убедись, что DNS работает раньше, чем стартует chrony.
    - Или используй IP-адреса серверов напрямую (менее гибко, но надежнее в изолированных сетях).
    - В `chrony.conf` можно указать `offline` для источников, если сеть поднимается позже.
