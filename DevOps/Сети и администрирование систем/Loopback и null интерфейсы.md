### Loopback интерфейс:

**Loopback (петлевой) интерфейс** — это логический интерфейс, который всегда находится в состоянии **UP/UP**, пока работает само устройство (роутер или коммутатор). Он не зависит от состояния физических портов. Если у роутера отвалится все физические линки, loopback интерфейс останется активным.

**Для чего он нужен?**
1. **Идентификация устройства (Router ID):** Протоколы динамической маршрутизации (OSPF, BGP, EIGRP) используют IP-адрес loopback интерфейса в качестве идентификатора роутера (Router ID). Это гарантирует, что ID не изменится, если какой-то физический интерфейс упадет.
2. **Управление (Management):** Администраторы настраивают доступ по SSH/Telnet/SNMP именно на адрес loopback. Это позволяет управлять устройством через любой доступный физический путь, так как адрес loopback всегда достижим (при наличии маршрута), в отличие от адреса конкретного физического порта.
3. **Тестирование:** Используется для проверки стека TCP/IP на самом устройстве без выхода во внешнюю сеть.
4. **Стабильность туннелей:** При построении GRE/IPsec туннелей в качестве конечных точек часто используют адреса loopback, чтобы туннель не рвался при временной потере одного из физических каналов (если есть альтернативный путь).

Работа с этим интерфейсом имеет свои нюансы, о которых важно знать:
1. **Маска подсети:** Всегда используйте маску `/32` (для IPv4) или `/128` (для IPv6).
    - _Почему?_ Если вы зададите маску `/24`, операционная система может посчитать, что вся эта подсеть подключена напрямую к этому виртуальному интерфейсу. Это может привести к тому, что роутер начнет отвечать на ARP-запросы для всех IP-адресов этой подсети, вызывая конфликты.
2. **Маршрутизация:** Адрес loopback не появляется в таблице маршрутизации соседей автоматически. Вы должны явно анонсировать сеть loopback через протокол динамической маршрутизации (OSPF, BGP) или создать статический маршрут на других устройствах.
3. **Приоритет выбора:** Если у вас несколько loopback интерфейсов, протоколы маршрутизации обычно выбирают тот, у которого наибольший IP-адрес, если Router ID не задан вручную. Лучше задавать Router ID явно или использовать `loopback 0` с самым низким IP.

### Null интерфейсы:

**Null0** — это виртуальный интерфейс, который представляет собой "черную дыру". Любой пакет, отправленный на этот интерфейс, немедленно отбрасывается устройством без уведомления отправителя (по умолчанию) и без потребления ресурсов процессора на дальнейшую обработку.

**Для чего он нужен?**
1. **Административная дистанция (AD):** Статический маршрут на Null0 имеет стандартную AD = 1. Будьте осторожны: если у вас есть динамический протокол (например, OSPF с AD=110), статический маршрут всегда победит. Однако, если вы используете агрегацию внутри того же протокола (например, суммаризация в OSPF), маршрут на Null0 будет создан автоматически с той же метрикой, что и сумма.
2. **ICMP сообщения:** По умолчанию, когда пакет попадает на Null0, устройство может отправить отправителю сообщение ICMP "Host Unreachable". В условиях мощной DDoS-атаки генерация ответов ICMP может нагрузить процессор.
    - _Решение:_ На некоторых платформах можно отключить отправку unreachable сообщений для конкретных маршрутов или глобально, чтобы превратить Null0 в настоящую "тихую" черную дыру.
    - Команда (Cisco): `no ip unreachables` (вводится в режиме интерфейса, но для Null0 поведение может зависеть от версии IOS и платформы, часто требуется глобальное отключение или использование специфических флагов в маршруте).
3. **Логирование:** При отладке проблем (connectivity) легко упустить тот факт, что трафик уходит в Null0. Всегда проверяйте таблицу маршрутизации (`show ip route`), если пинги не проходят, но интерфейсы живы. Ищите запись с пометкой `via Null0`.

В операционных системах маршрутизаторов в качестве null интерфейса выступает Null0. Пример:
```Router
ip route 192.168.50.0 255.255.255.0 Null0
```

В linux системах роль "черной дыры" (куда пакеты исчезают бесследно) выполняет интерфейс с названием `blackhole`, но это не совсем обычный интерфейс, который отображается в списке `ip a` со своим IP-адресом. 

В linux-е не создается отдельный интерфейс, а вместо этого указывается тип отбрасывания для маршрута. Пример:
```bash
# Добавляет маршрут: весь трафик для 10.0.0.0/8 отбрасывать silently (как Null0)
sudo ip route add blackhole 10.0.0.0/8
```

**Типы отбрасывания в Linux:**
* `blackhole`: Пакет тихо удаляется (аналог Cisco Null0 по умолчанию). Отправитель не получает ответа;
* `unreachable`: Пакет удаляется, и отправителю уходит ICMP-сообщение "Host Unreachable";
* `prohibit`: Пакет удаляется, и отправителю уходит ICMP-сообщение "Communication Administratively Prohibited".
