### Терминология:

**Файл** — набор структурированной информации, относящейся к одному объекту. Файлом может быть изображение, аудиозапись, видео или конфигурационный текст. Каждый файл состоит из двух частей:
- **Данные** — полезное содержимое файла;
- **Метаданные** — данные о данных: разрешение изображения, частота дискретизации звука, кодировка текста и т.д.

**Файловая система (ФС)** — это механизм, с помощью которого операционная система организует, хранит, извлекает и управляет данными на носителях информации. Она решает, куда записать новый блок данных, как быстро найти нужный файл среди миллионов других и что делать, если питание пропало на середине записи. Для операционной системы диск без ФС — это просто длинная последовательность блоков (секторов) с номерами от 0 до N. ФС добавляет структуру, позволяющую:
- Группировать данные в **файлы**.
- Организовывать их в **каталоги (деревья)**.
- Хранить **метаданные** (права доступа, время создания, владелец).
- Эффективно управлять свободным местом.

**Суперблок (Superblock)** - Это «паспорт» файловой системы. Он хранится в начале раздела (и часто дублируется в других местах для надежности):
- **Что там:** Размер ФС, количество свободных и занятых блоков, размер блока, состояние ФС (чистая/грязная), ссылка на корневой inode.
- **Зачем:** При монтировании (`mount`) ядро читает суперблок, чтобы понять, как работать с этим разделом.

**Inode (Индексный дескриптор)** - это структура данных, описывающая файл, но не содержащая его имени. Содержит в себе следующую информацию:
- Тип файла (обычный, директория, symlink, устройство);
- Права доступа (rwx для owner/group/others);
- Владелец (UID) и группа (GID);
- Время создания, изменения, последнего доступа;
- Размер файла;
- Указатели на физические блоки данных на диске, где лежит содержимое файла.

Имя файла хранится не в **inode**, а в **директории**. Директория — это просто специальный файл, который связывает «Имя файла» -> «Номер inode».

**Таблица свободных мест (Bitmap / Free Space Map)** - структура, которая хранит битовую карту: какой блок свободен (0), а какой занят (1). Нужна, чтобы при создании нового файла быстро найти место.

**Журнал** - в современных ФС (ext4, XFS) есть механизм журналирования. Перед тем как записать данные в основную структуру, система пишет намерение в специальный журнал. Если питание пропадет во время записи, при перезагрузке система быстро «доиграет» операции из журнала, избегая повреждения структур ФС.

**Сектор** - самая маленькая физическая ячейка памяти на пластине диска. Стандартно **512 байт** (в новых дисках Advanced Format — **4096 байт** или 4К). Это физическая основа. Всё остальное строится из секторов

**Блок** - минимальная единица, с которой работает ОС и файловая система. Обычно **4 Кб** (это ровно один современный физический сектор) или больше (например, 8 Кб, 16 Кб). Если файл весит 1 байт, он всё равно займет целый блок, остальное пропадет. 

#### Разделы:

**Раздел (partition)** - логически выделенная **непрерывная** область на физическом диске. Нужно чтобы разделить один большой диск на независимые зоны. Раздел говорит ОС «Я занимаю секторы с 1000 по 5000 на диске А» (Жесткая привязка). После создания не может быть расширен. Раздел создается строго на одном диске и не может быть создан на основе двух физических дисков.

##### Для чего нужны разделы:

* **Разные файловые системы.** Может понадобиться, чтобы часть диска работала под **ext4** (для системы), другая часть под **XFS** (для баз данных), а третья под **ZFS** (для виртуальных машин). На одном разделе нельзя создать две разные ФС одновременно;
* **Изоляция сбоев и безопасности.** Если одна файловая система «упадет» (повредится структура данных из-за сбоя питания), остальные разделы могут остаться целыми. Пример: Отдельный раздел `/home`. Если пользователь случайно забьет диск логами или ошибками программы в `/var`, система (`/`) продолжит работать, потому что она на другом разделе.
* **Мультизагрузка (Dual Boot)**. Windows и Linux часто не совместимы с одной файловой системой. Windows ставится на NTFS, Linux предпочитает ext4. Чтобы поставить их на один диск, нужно разделить его на части.
* **Шифрование.** Можно зашифровать только раздел с данными (`/home` или `/secret`), оставив систему (`/`) незашифрованной для удобства загрузки. В LVM это делается еще гибче (шифрование всего VG), но база часто лежит на разделах.
* **Бэкапы и восстановление.** Легче делать образ системы (`/`), если данные пользователей лежат отдельно (`/home`). При переустановке ОС данные пользователей не затрагиваются.

##### Стандарты разметки разделов:

* **MBR (Master Boot Record):**
	* Структура MBR очень проста и занимает всего **512 байт** (первый сектор диска). В этих 512 байтах нужно уместить:
	    1. Код загрузчика (bootstrap code) — ~446 байт;
	    2. Подпись диска (Disk Signature) — 4 байта;
	    3. **Таблицу разделов**.
	- Под таблицу разделов отвели всего **64 байта**. Каждая запись о разделе занимает **16 байт**. - `64/16=4`, из чего следует что максимально по этому стандарту может быть 4 раздела;
	- Поддерживает диски размером до 2 терабайт;
	- Стандарт считается устаревшим и не рекомендуется использовать;
	- **Как обошли ограничение 4 разделов? (Extended Partition):**
		1. Из 4 доступных слотов один можно занять под **Расширенный раздел (Extended Partition)**.
		2. Расширенный раздел — это просто контейнер. Внутри него нельзя создать файловую систему напрямую.
		3. Внутри расширенного раздела создается своя цепочка записей, позволяющая создавать сколько угодно **Логических разделов (Logical Partitions)**.
		4. **Итог схемы MBR:**
		    - Либо 4 **Основных (Primary)** раздела.
		    - Либо 3 Основных + 1 Расширенный (в котором много Логических).
- **GPT (GUID (Globally Unique Identifier) Partition Table):**
	- Занимает гораздо больше места (несколько секторов в начале и в конце диска для надежности) в отличие от **MBR**;
	- Поддерживает до **128 разделов** по умолчанию в большинстве ОС (теоретически лимит зависит от размера таблицы, но он огромен);
	- Нет понятия «Основной» vs «Логический». Все разделы равноправны;
	- Поддерживает диски размером до 9.4 зеттабайт;
	- Хранит копию таблицы разделов в конце диска;


#### Тома:

**Том** — это единая доступная область хранения с единой файловой системой. это абстракция, созданная менеджером логических томов (LVM в Linux). Он отвязывает файловую систему от физической геометрии диска. **Том** говорит ОС: «Я занимаю 4000 блоков данных. Где они физически лежат? Мне всё равно, лишь бы менеджер (LVM) знал». Том можно спокойно расширять на нужное кол-во памяти и на разные диски. Перед созданием тома для LVM нужно отдать раздел или целый диск, чтобы том и раздел не были назначены на одни и те же сектора, чтобы файловые системы разделов и томов не конфликтовали между собой.

Том имеет все то же предназначение что и раздел, а также следующие цели:

* **Объединение дисков:** Ты можешь создать один том размером 10 Тб, который физически состоит из трех разных дисков по 4 Тб, 3 Тб и 3 Тб. Раздел так не умеет (раздел не может перепрыгнуть на другой диск);
* **Миграция на лету:** Если диск начинает сыпаться, ты можешь скопировать данные с него на новый диск (`pvmove`) без остановки процесса и без изменения имени тома для ОС. Для программы, работающей с диском, ничего не изменится. С разделом так нельзя — нужно копировать файлы вручную или делать дамп/восстановление.
* **Снимки (Snapshots):** Можно создать копию состояния тома за 1 секунду (технология Copy-on-Write). Например, перед обновлением базы данных или системы ты делаешь снапшот. Если всё сломалось — откатываешься назад за секунды. С разделами это невозможно без сложных внешних инструментов.
* **Тонкие тома (Thin Provisioning):** Можно создать том на 1 Тб, хотя физически у тебя есть только 500 Гб. Том будет расти по мере записи данных. Если данные не запишутся, место не потратится.

#### Раздел vs Том:

В 95% случаев **разделы** становятся лишним промежуточным звеном и многие администраторы отдают под LVM весь диск целиком, минуя создание таблиц разделов. **НО** есть несколько критических причин, почему разделы всё еще существуют и иногда **обязательны**:

1. **Загрузка системы (BIOS/UEFI и GRUB).** Компьютер должен как-то начать загружаться, прежде чем запустится ядро Linux и загрузятся драйверы LVM. Без хотя бы одного обычного раздела (для `/boot` или EFI) большинство систем просто не включатся. Загрузчики (GRUB, systemd-boot) и прошивки материнских плат (BIOS/UEFI) часто **не умеют** читать LVM напрямую (или умеют с большими ограничениями и сложной настройкой):
	* **UEFI** требует специальный раздел с файловой системой **FAT32** (ESP — EFI System Partition), чтобы найти файл `shimx64.efi` или `grubx64.efi`. На FAT32 нельзя создать LVM;
	* **Старый BIOS (Legacy)** часто ожидает найти загрузочный код в первых секторах обычного раздела (ext4/xfs), а не внутри сложной структуры LVM.
	
	Мы **вынуждены** создавать маленький раздел (обычно 500 Мб – 1 Гб) вне LVM:
	- `/dev/sda1` (раздел, FAT32) -> Монтируется как `/boot/efi`.
	- `/dev/sda2` (раздел, ext4) -> Монтируется как `/boot` (здесь лежат ядра и initramfs).
2.  **Совместимость и переносимость**. LVM — это технология специфичная для Linux (и некоторых Unix-систем).
	- **Двойная загрузка (Dual Boot):** Если нужно поставить рядом Windows и Linux:
	    - Windows **не видит** LVM. Для неё диск с LVM выглядит как неразмеченное пространство или мусор;
	    - Чтобы Windows увидела свой диск, он должен быть размечен обычными разделами (NTFS).
	- **Аварийное восстановление:**
	    - Если система сломалась настолько, что не грузится даже в Rescue-режиме с поддержкой LVM, тебе будет гораздо проще подключить диск к другому компьютеру и прочитать данные, если они лежат на обычном разделе ext4;
	    - С LVM нужно сначала активировать группу (`vgchange -ay`), разобраться с метаданными. В панике это лишний шаг.
	- **Внешние диски:** Если создается резервная копия на внешний USB-диск и планируется подключать его к разным машинам (возможно, не только Linux), обычные разделы надежнее.

### Как происходит считывание информации с диска:

Представим, что в терминале выполняется команда `cat /home/altlinux/config.txt`. Вот что происходит под капотом:

#### Шаг 1: Поиск пути (Path Resolution):

Ядро начинает с корневого inode (известен из суперблока).

1. Читает содержимое корневой директории (список имен и inode);
2. Ищет запись `home`, получает её inode;
3. Переходит в директорию `home`, читает её блоки, ищет `altlinux`, получает inode;
4. Переходит в `altlinux`, ищет `config.txt`;
5. Находит запись `config.txt` -> **Inode № 12345**.

#### Шаг 2: Чтение метаданных (Inode Lookup):

Ядро обращается к таблице inode на диске, загружает в память структуру inode № 12345.

1. Проверяет права доступа (есть ли у пользователя `altlinux` право на чтение?);
2. Узнает размер файла;
3. Получает список указателей на блоки данных.

#### Шаг 3: Чтение данных (Data Retrieval):

Теперь ядро знает номера физических блоков (например, блоки 500, 501, 505, 1024).

1. Ядро формирует запрос к **драйверу блочного устройства** (disk driver);
2. Драйвер преобразует логические номера блоков в команды для контроллера диска (SATA/NVMe);
3. **Кэш страниц (Page Cache):** Прежде чем идти на физический диск, ядро проверяет, нет ли этих блоков в оперативной памяти (Linux активно использует свободную RAM для кэширования дисковых операций). Если данные в кэше — чтение мгновенное;
4. Если данных нет в RAM, контроллер диска позиционирует считывающие головки (на HDD) или обращается к ячейкам NAND (на SSD) и считывает нужные блоки.

#### Шаг 4: Сборка и выдача:

Данные из считанных блоков собираются в буфер ядра, обрезаются по точному размеру файла (так как последний блок может быть заполнен не полностью) и передаются в пользовательское пространство программе `cat`.

### Фрагментация и дефрагментация:

**Фрагментация** — это состояние файловой системы, при котором файлы разбиты на части (фрагменты), записанные в не соседних физических блоках диска.

**Дефрагментация** — процесс перераспределения блоков файлов на диске таким образом, чтобы каждый файл занимал непрерывную область, а свободное пространство было собрано в конце (или больших кусках).

**Пример возникновения фрагментации:**

1. Ты записываешь файл `A` (100 Мб), затем `B` (50 Мб), затем `C` (100 Мб). Все стоит плотно.
2. Ты удаляешь файл `B`. Появляется дырка в 50 Мб посередине.
3. Ты создаешь новый файл `D` размером 80 Мб.
    - В дырку (50 Мб) он не влезает целиком.
    - Файловая система пишет первые 50 Мб файла `D` в освободившееся место.
    - Оставшиеся 30 Мб она вынуждена записать в следующее свободное место где-то дальше на диске.
4. **Итог:** Файл `D` теперь состоит из двух фрагментов. Если таких операций много, файл может состоять из сотен кусочков, разбросанных по всему диску.

#### Влияние фрагментации на разные типы носителей:

**Жесткие диски (HDD):**

- **Механика:** У HDD есть вращающиеся пластины и считывающая головка на подвижном коромысле.
- **Проблема:** Чтобы прочитать фрагментированный файл, головке нужно:
    1. Доехать до начала первого куска (**Seek Time**).
    2. Дождаться вращения пластины, пока нужный сектор окажется под головкой (**Rotational Latency**).
    3. Считать данные.
    4. **Переместиться** ко второму куску (снова Seek + Latency).
- **Результат:** При сильной фрагментации время механического перемещения головки начинает доминировать над временем чтения данных. Скорость линейного чтения падает с сотен МБ/с до единиц МБ/с. Система «тупит», загрузка ОС замедляется.
- **Вывод:** Для HDD дефрагментация **критически важна**.

**Твердотельные накопители (SSD / NVMe):**

- **Механика:** Нет движущихся частей. Доступ к любой ячейке памяти занимает одинаковое время (random access почти так же быстр, как sequential).
- **Проблема фрагментации:** Для скорости чтения фрагментация на SSD **почти не влияет**. Контроллер SSD может считывать разбросанные страницы параллельно по разным каналам.
- **Главная проблема SSD — ресурс записи:** У ячеек NAND-flash есть лимит циклов перезаписи (P/E cycles).
    - Процесс дефрагментации подразумевает чтение всех файлов и их перезапись в новом, непрерывном порядке.
    - Это создает огромный объем лишней записи (Write Amplification).
    - **Вывод:** Классическая дефрагментация для SSD **вредна**. Она изнашивает диск без заметного прироста производительности.

#### Алгоритм работы дефрагментатора:

1. **Анализ:** Сканирование ФС, построение карты фрагментов.
2. **Сортировка:** Определение оптимального расположения файлов (часто популярные системные файлы помещают в начало диска, где у HDD выше плотность записи и скорость).
3. **Перемещение:**
    - Найти свободный непрерывный участок.
    - Считать фрагментированный файл в оперативную память (кэш).
    - Записать его одним куском в новое место.
    - Обновить метаданные (inode): изменить указатели на блоки.
    - Освободить старые блоки.
4. **Сжатие свободного места:** Сдвиг всех файлов к началу диска, чтобы собрать всё свободное место в один большой кусок в конце.

#### Особенности фрагментации в Linux и Windows:

**Windows (NTFS).** Исторически склонна к фрагментации:
 
- В современных версиях (Win 7+) встроен планировщик оптимизации дисков;
- Для HDD запускает дефрагментацию, для SSD — отправляет команду TRIM.

**Linux.** Linux-файловые системы спроектированы так, чтобы минимизировать фрагментацию на лету:

1. **Отложенное выделение (Delayed Allocation):**
    - Когда приложение пишет файл, ext4 не выделяет блоки сразу. Она держит данные в кэше памяти.
    - Перед реальной записью на диск ФС видит полный размер файла и старается выделить под него один большой непрерывный кусок. Это резко снижает шанс фрагментации мелких файлов.
2. **Предварительное выделение (Pre-allocation):**
    - ФС выделяет места чуть больше, чем нужно сейчас, с запасом на рост файла.
3. **Расширение экстентов (Extents):**
    - В старых ФС (ext2/3) использовались блоки. В ext4 и XFS используются **экстенты**. Экстент — это запись в inode вида: «Начиная с блока N, занято X последовательных блоков».
    - Одна запись в inode может описывать мегабайты данных, если они лежат подряд. Это уменьшает размер метаданных и ускоряет работу с большими файлами.

**Нужна ли дефрагментация в Linux?**

- **Обычно нет.** Для десктопов и серверов с ext4/XFS она требуется крайне редко.
- **Когда нужна:** Если диск заполнен под завязку (>90-95%), алгоритмам становится трудно найти большие дырки, и фрагментация растет. Или на серверах баз данных, где файлы постоянно растут и удаляются хаотично.

### Удаление данных в Unix/Linux:

#### Логическое удаление:

Когда ты выполняешь `rm file.txt` в Linux:

1. **Ссылка удаляется:** Из директории удаляется запись, связывающая имя файла с его inode.
2. **Счетчик ссылок (Link Count):** В inode уменьшается счетчик ссылок (`i_nlink`).
    - Если он стал **0** и **никто не держит файл открытым** (нет активных file descriptors в процессах):
        - Блоки данных помечаются как свободные в битовой карте (bitmap).
        - Сам inode помечается как свободный.
        - **Данные физически остаются на диске!** Они просто становятся «невидимыми» для ОС и могут быть перезаписаны новыми данными в любой момент.
    - Если процесс (например, логгер или база данных) держит файл открытым в момент удаления, inode не освобождается, пока процесс не закроет дескриптор или не умрет. Файл исчезает из `ls`, но место на диске не освобождается.
        - _Как найти:_ `lsof | grep deleted`.
        - _Как восстановить место без перезапуска:_ Найти PID и выполнить `echo > /proc/<PID>/fd/<FD_NUMBER>`.

Данные лежат на диске, пока не будут перезаписаны сверху. Восстановление возможно специальными утилитами (testdisk, photorec), если блоки еще не затерты.

#### Принудительное (безопасное) удаление:

Если цель — сделать так, чтобы данные **невозможно было восстановить** (например, перед продажей диска или удалением чувствительных ключей), простого `rm` недостаточно.

**Методы для HDD:**

1. **Перезапись (Wiping):** Многократная запись случайных данных или нулей поверх файла/раздела.
    - Утилита: `shred -v -n 3 -z file.txt` (3 прохода случайными данными + 1 проход нулями).
    - Для всего диска: `dd if=/dev/urandom of=/dev/sdX ...` (долго и опасно, можно стереть не то).
2. **ATA Secure Erase:** Специальная команда, встроенная в прошивку диска. Она заставляет контроллер диска перезаписать все ячейки (часто напряжением, что быстрее программной записи).
    - Утилита: `hdparm --secure-erase /dev/sdX`.

**Методы для SSD:**

- **НЕ используй `shred` или многократную перезапись на SSD!**
    - Из-за выравнивания износа контроллер SSD может записать новые данные в другое физическое место, оставив старые копии нетронутыми в скрытых блоках.
    - Ты просто износишь ресурс диска зря.
- **Правильный метод:** Команда **TRIM** (или ATA Secure Erase / NVMe Format NVM).
    - При обычном удалении с поддержкой TRIM (`discard`) ОС сообщает контроллеру: «Эти блоки больше не нужны».
    - Контроллер SSD помечает их как свободные и **физически очищает** (сбрасывает напряжение в ячейках) в фоновом режиме или по команде.
    - Для гарантированного полного сброса используй: `blkdiscard /dev/nvme0n1` (мгновенно, сообщает контроллеру очистить всё) или `hdparm --secure-erase`.

#### Создание файловой системы для диска:

Если диск новый `lsblk -f` показывает, что у раздела (например, `/dev/sdb1`) нет типа файловой системы (FSTYPE), её нужно создать.

> Внимание! Это удалит все данные на разделе.

```bash
# Создать ext4 (стандарт для Linux)
mkfs.ext4 /dev/sdb1

# Или xfs (часто используется в enterprise)
mkfs.xfs /dev/sdb1
```

#### Создание LVM томов и группы томов а также их расширение:

Допустим нужно объединить `/dev/sdb1` и `/dev/sdc1` в одну группу `vg_data` и создать там том `lv_projects` размером 50 Гб. После чего нужно расширить том `lv_projects` на 20 Гб и расширить группу `vg_data` на 100 Гб.

##### Шаг 1. Передача разделов LVM для создания томов:

```bash
pvcreate /dev/sdb1 /dev/sdc1
```

> Вывод: `Physical volume "/dev/sdb1" successfully created`. Теперь эти разделы нельзя монтировать напрямую как обычные диски. Они стали частью системы LVM.

Просмотр информации о дисках использующихся в LVM:

```bash
pvs
# Или подробно:
pvdisplay /dev/sdb1
```

##### Шаг 2. Создание группы томов (VG):

```bash
vgcreate vg_data /dev/sdb1 /dev/sdc1
```

> Мы объединили два физических тома в одну группу.

Просмотр информации о группах томов или о конкретной группе:

```bash
vgs
# Или подробно:
vgdisplay vg_data
```

##### Шаг 3. Создание тома на основе группы томов:

Создадим том `lv_projects` размером 50 Гб из группы `vg_data`:

```bash
lvcreate -L 50G -n lv_projects vg_data
```

- `-L 50G`: Размер (можно указать в M, G, T).
- `-n lv_projects`: Имя тома.
- `vg_data`: Имя группы, из которой берем место.

Если нужно занять для тома все свободное место в группе томов:

```bash
lvcreate -l 100%FREE -n lv_projects vg_data
```

> Используется маленькая буква `-l` (extent), а не большая `-L`).

Просмотр информации о томах или о конкретном томе:

```bash
lvs
# Или подробно:
lvdisplay /dev/vg_data/lv_projects
```

##### Шаг 4. Расширение тома:

В группе `vg_data` должно быть свободное место для расширения:

```bash
lvextend -L +20G /dev/vg_data/lv_projects
```

> Знак `+` означает добавить 20Г к текущему размеру. Без знака `+` было бы "установить размер ровно 20 Гб", что уменьшило бы том!

##### Шаг 5. Расширение  группы томов:

Допустим к серверу подключен новый диск `/dev/sdd`, который нужно добавить в группу томов `vg_data` и расширить группу им:

```bash
pvcreate /dev/sdd

vgextend vg_data /dev/sdd
```

> Теперь в `vg_data` снова есть дополнительное место (размер нового диска) и можно расширять существующие тома через `lvextend`.


##### Удаление томов:

1. **Размонтируем том.** Нельзя удалять примонтированный том:

```bash
umount /dev/vg_data/lv_projects
```

2. Удаление тома:

```bash
lvremove /dev/vg_data/lv_projects
```

> Том исчезает из группы (`vgs` покажет увеличение свободного места `Free`). Данные теряются навсегда (восстановление возможно только сложными утилитами типа `testdisk`, но гарантий нет).

##### Удаление группы томов:

Группу можно удалить, только когда она **пустая** (в ней нет ни одного логического тома). После того как группа пустая ее можно удалить:

```bash
vgremove vg_data
```

##### Операции над томами и дисками:

Перемещение данных с диска на свободные диски в группе томов:

```bash
pvmove /dev/sdb1
```

Исключение диска из группы томов. Можно выполнить только если на диске нет данных:

```bash
vgreduce vg_data /dev/sdb1
```

Удаление диска из LVM среды чтобы диск стал чистым устройством:

```bash
pvremove /dev/sdb1
```

#### Ручное монтирование и размонтирование:

Монтирование — это «подключение» файловой системы диска к конкретной папке в дереве каталогов Linux.

**Шаг 1: Создать точку монтирования.** Это обычная пустая папка, куда «войдет» содержимое диска:

```bash
mkdir -p /mnt/data_disk
```

**Шаг 2: Монтирование диска.** 

```bash
mount /dev/sdb1 /mnt/data_disk
```

> Теперь всё, что будет записано в `/mnt/data_disk`, физически ляжет на диск `/dev/sdb1`.


**Шаг 3: Размонтирование.** Нельзя выдергивать диск и не перезагружать сервер, пока диск примонтирован! Сначала размонтируем:

```bash
umount /mnt/data_disk
# Или можно указать устройство:
umount /dev/sdb1
```

> Ошибка «target is busy» означает, что кто-то (процесс или пользователь) находится внутри этой папки. Нужно выйти из нее и попробовать снова.

#### Автоматическое монтирование диска при загрузке:

Чтобы диск подключался сам после перезагрузки, нужно добавить запись в файл `/etc/fstab` (File System Table).

> Важно! Никогда не редактируй `/etc/fstab` напрямую через `nano` без страховки. Одна опечатка — и сервер не загрузится (упадет в emergency mode).

1. Сделай бэкап: `sudo cp /etc/fstab /etc/fstab.bak`
2. Используй **UUID**, а не имя устройства (`/dev/sdb1`). Имя может измениться (стать `/dev/sdc1`) после добавления нового диска или перезапуска портов. UUID уникален и вечен.

**Шаг 1: Получение UUID-а.**

```bash
blkid /dev/sdb1
```

**Шаг 2: Редактирование fstab.**

```bash
nano /etc/fstab
```

Добавление строчки для автозагрузки диска

```nano
UUID=a1b2c3d4-e5f6-7890-...  /mnt/data_disk  ext4  defaults,noatime  0  2
```

**Разбор полей:**

1. `UUID=...` — Идентификатор диска;
2. `/mnt/data_disk` — Точка монтирования (папка должна существовать!);
3. `ext4` — Тип файловой системы;
4. `defaults,noatime` — Опции:
    - `defaults`: стандартные права (rw, suid, dev, exec, auto, nouser, async);
    - `noatime`: **Важно для производительности!** Запрещает записывать время последнего чтения файла. Без этого каждый прочитанный файл вызывает микро-запись на диск, что убивает SSD и замедляет HDD;
5. `0` — Флаг дампа (для бэкапа `dump`, обычно 0);
6. `2` — Порядок проверки диска (`fsck`) при загрузке:
    - `0`: Не проверять (обычно для не-корневых дисков или сетевых);
    - `1`: Проверять первым (только для корня `/`);
    - `2`: Проверять вторым (для остальных локальных дисков).

**Шаг 3: Проверка перед перезагрузкой.** Перед перезагрузкой нужно проверить синтаксис конфигурации:

```bash
mount -a
```


### Шифрование дисков через LUKS:

В Linux стандартом де-факто является **LUKS (Linux Unified Key Setup)**. Это не алгоритм шифрования сам по себе, а формат заголовка, который управляет ключами и настройками шифрования (обычно используется алгоритм AES).

1. **Физический диск** (`/dev/sdb`);
2. **LUKS-контейнер**: Создается поверх диска. Он шифрует всё содержимое. Без пароля данные внутри выглядят как белый шум;
3. **Виртуальное устройство** (`/dev/mapper/...`): Когда ты вводишь правильный пароль, LUKS «открывает» контейнер и создает виртуальный блочный интерфейс;
4. **Файловая система (или LVM)**: Создается уже **внутри** этого виртуального устройства.

LUKS шифрует весь блок устройства. Если ты используешь LVM, у тебя есть два пути:

- **Шифрование всего диска:** Диск -> LUKS -> LVM (PV) -> VG -> LV. (Рекомендуется: один пароль при загрузке открывает всё);
- **Шифрование отдельных томов:** Диск -> LVM -> LV -> LUKS -> ФС. (Нужно вводить пароль для каждого тома отдельно. Удобно для внешних дисков, неудобно для системных).

#### Настройка шифрования дисков:

##### Шаг 1. Установка пакетов:

```bash
apt-get update && apt-get install cryptsetup
```

##### Шаг 2. Создание LUKS контейнера:

Допустим, у нас есть чистый диск `/dev/sdb`. Мы хотим зашифровать его целиком, а внутри развернуть LVM.

```
cryptsetup luksFormat /dev/sdb
```

> После чего нужно придумать и ввести **парольную фразу** (Passphrase). Используйте длинную фразу. Запомнить её критически важно. Если потерять пароль — данные не восстановить никак.

> Важно! Все данные на `/dev/sdb` будут **безвозвратно удалены**.


##### Шаг 3. Открытие контейнера:

Теперь нужно «открыть» зашифрованный диск, чтобы система увидела его как обычное устройство.

```bash
cryptsetup open /dev/sdb my_secure_disk
```

- `my_secure_disk` — это имя для устройства. Оно станет именем устройства в `/dev/mapper/`.
- Система запросит пароль, который был задан на шаге 2.

> Появится устройство `/dev/mapper/my_secure_disk`. Для системы это теперь обычный «чистый» диск.


##### Шаг 4. Добавление зашифрованного диска в автозагрузку:

**Вариант 1. Ручной ввод пароля при загрузке (или через `cryptsetup open`):**

Чтобы диск открывался автоматически при загрузке (или чтобы ты мог открыть его одной командой без ввода параметров), нужно настроить файл конфигурации `/etc/crypttab`.

```bash
vim /etc/crypttab
```

```vim
my_secure_disk  UUID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx  none  luks
```

- `my_secure_disk`: Имя маппера (должно совпадать с тем, что использовалось в `cryptsetup open`).
- `UUID=...`: UUID физического диска (`/dev/sdb`). Узнай через `sudo blkid /dev/sdb`. **Не путать с UUID файловой системы!**
- `none`: Означает, что пароль будет запрошен интерактивно (или через keyfile, если указать путь).
- `luks`: Опция, говорящая, что это LUKS.

**Вариант 2. Ключ-файл (Автоматическое открытие):**

1. Создаем случайный файл-ключ:

```bash
dd if=/dev/urandom of=/root/keys/secure.key bs=512 count=4
chmod 0400 /root/keys/secure.key
```

2. Добавляем этот ключ в LUKS-контейнер (у контейнера может быть до 8 ключей):

```bash
cryptsetup luksAddKey /dev/sdb /root/keys/secure.key
# Сначала спросит старый пароль, потом создаст доступ по новому ключу.
```

3. Настройка `/etc/crypttab`:

```vim
my_secure_disk  UUID=xxxxxxxx...  /root/keys/secure.key  luks
```

Теперь при загрузке системы (или активации сервиса) пароль спрашиваться не будет.


**Сценарии когда может понадобится разблокировка по ключу:**

1. Каскадное шифрование (Зависимость дисков):
	* **Диск 1 (Системный, `/`):** Зашифрован LUKS. Пароль вводится руками при загрузке. На этом диске лежит ОС и... **секретный файл-ключ**.
	* **Диск 2 (Данные, `/data`):** Зашифрован LUKS. Настроен на авто-разблокировку через ключ-файл, который лежит на Диске 1.
	* **В чем смысл защиты?**
		- **Кража Диска 2:** Если вор выкрал только второй диск (с данными), он бесполезен. У него нет ключа, который лежит на первом диске. Данные защищены.
		- **Кража обоих дисков:** Да, если украли всё сразу — защита слабая (нужен пароль от первого). Но часто диски стоят в разных слотах, или второй диск может быть сетевым/внешним;
		- **Удобство:** Нужно ввести пароль только один раз (для системы), а все тома с данными открываются сами. Не нужно бегать к серверу вводить 5 паролей для 5 томов.
	- **Как это работает:**
		1. Включается сервер;
		2. Консоль просит пароль от Диске 1. Ты вводишь его;
		3. Система загружается, читает ключ-файл с уже разблокированного Диске 1;
		4. Автоматически разблокирует Диск 2;
		5. Сервисы (базы данных, файлы) стартуют без твоего участия.
2. Кластеры и Высокая Доступность (HA). В кластере (например, Proxmox HA, Kubernetes, базы данных с репликацией) серверы должны перезагружаться и восстанавливаться **автоматически** после сбоя питания, без вмешательства человека.
	* Если каждый сервер в кластере будет ждать ввода пароля вручную при каждой перезагрузке — кластер умрет. Время простоя (Downtime) будет огромным;
	- **Решение:** Ключи хранятся на защищенном внешнем хранилище (например, HashiCorp Vault, NFS с доступом только по IP внутри сети, или на загрузочной флешке, которая остается в сервере), либо используется каскадная схема;
	- Здесь шифрование защищает от **физической кражи диска из стойки** или утилизации старого диска, но не от компрометации всей инфраструктуры целиком.
3. Разделение обязанностей (Separation of Duties):
	* **Админ железа (Hardware Admin):** Имеет физический доступ к серверам, может менять диски, перезагружать их. Но он **не знает** паролей шифрования;
	- **Админ безопасности (Security Admin):** Хранит ключи в безопасном хранилище (Vault);
	- При перезагрузке скрипт автоматически запрашивает ключ у Vault (по сертификату сервера);
	- Если Hardware Admin украдет диск, он не сможет его прочитать, так как у него нет доступа к Vault, где лежат ключи. Ключ не лежит просто файлом на диске, он подтягивается по сети из защищенного источника.

**Когда ключ-файл — это плохо, а когда — хорошо?**

| Сценарий                         | Где лежит ключ?                                                   | Есть ли смысл?         | От чего защищает?                                                                    |
| -------------------------------- | ----------------------------------------------------------------- | ---------------------- | ------------------------------------------------------------------------------------ |
| **Плохо** 🔴                     | Ключ лежит в открытом виде **на том же диске**, который шифруешь. | **НЕТ**. Бессмысленно. | Ни от чего. Вор украл диск -> украл ключ -> прочитал данные.                         |
| **Хорошо** 🟢 (Каскад)           | Ключ лежит на **другом зашифрованном диске** (системном).         | **ДА**.                | От кражи второго диска по отдельности. Удобство (1 пароль на вход).                  |
| **Хорошо** 🟢 (Сеть)             | Ключ подгружается по сети из **Vault/NFS** при старте.            | **ДА**.                | От кражи диска, от несанкционированного запуска сервера вне сети компании.           |
| **Хорошо** 🟢 (Съемный носитель) | Ключ лежит на **USB-флешке**, которая вставлена в сервер.         | **ЧАСТИЧНО**.          | От кражи диска, если флешка осталась в сервере или у админа. (Защита от утилизации). |


#### Перевод диска в не зашифрованное состояние:

Для этой операции нужно **уничтожить заголовок LUKS**. Как только заголовок удален, система перестает видеть диск как зашифрованный контейнер. Для неё это снова просто кусок железа с непонятными данными (мусором). После этого ты можешь размечать его как обычный диск.

##### Шаг 1. Подготовка диска

**Размонтируем файловые системы**, которые находятся внутри LUKS

```bash
umount /mnt/my_secure_disk
```

**Деактивируем LVM** (если он был внутри):

```bash
vgchange -an
# Или конкретно группу:
vgchange -an vg_secure
```

**Закрываем LUKS-контейнер**:

```bash
cryptsetup close my_secure_disk
```

Проверка что устройство исчезло:

```bash
ls /dev/mapper/
# my_secure_disk там быть не должно!
```

##### Шаг 2. Уничтожение заголовка LUKS:

**Вариант 1. Быстрая очистка:**

Мы стираем только первые мегабайты диска, где лежит заголовок LUKS и метаданные. Остальные данные остаются лежать на диске в зашифрованном виде (но так как ключа нет, они просто мусор).

Используем утилиту `wipefs` (из пакета `util-linux`, стоит везде):

```bash
# Посмотри, что найдется на диске (для проверки)
wipefs /dev/sdb

# Удали ВСЕ подписи (LUKS, LVM, RAID и т.д.)
wipefs -a /dev/sdb
```

> Флаг `-a` (all) удаляет все сигнатуры файловой системы и криптографии.

**После этой команды:**
- `cryptsetup luksDump /dev/sdb` выдаст ошибку «Device is not a valid LUKS device»;
- Диск считается чистым для системы.

**Вариант 2. Полная очистка:**

```bash
# Затирка нулями. Это займет время, равное времени полной записи диска.
dd if=/dev/zero of=/dev/sdb bs=1M status=progress
```

##### Шаг 3. Проверка:

```bash
# Должно выдать ошибку, что это не LUKS
cryptsetup luksDump /dev/sdb

# Должно показать пустоту или только таблицу разделов (если она была)
lsblk -f /dev/sdb
```


#### Полезные команды управления LUKS:

|Команда|Описание|
|---|---|
|`cryptsetup luksDump /dev/sdb`|Показать информацию о контейнере (версия, слоты ключей, алгоритмы). **Очень полезно для диагностики.**|
|`cryptsetup luksAddKey /dev/sdb`|Добавить новый пароль/ключ в существующий контейнер.|
|`cryptsetup luksRemoveKey /dev/sdb`|Удалить пароль из контейнера. (Нужно ввести удаляемый пароль для подтверждения).|
|`cryptsetup luksChangeKey /dev/sdb`|Сменить пароль.|
|`cryptsetup close my_secure_disk`|Закрыть (заблокировать) контейнер. **Сначала размонтируй все ФС и деактивируй LVM внутри!**|
|`cryptsetup status my_secure_disk`|Показать статус открытого контейнера.|

#### Важные нюансы при работе с LUKS:

1. **Производительность:** Современное шифрование (AES-NI инструкции в процессоре) почти не влияет на скорость. Потеря производительности обычно составляет 3-5%, что незаметно для большинства задач. На старых CPU без AES-NI потеря может быть ощутимой.
2. **TRIM для SSD:** По умолчанию LUKS не передает команду TRIM на SSD (из соображений безопасности, чтобы можно было анализировать занятость блоков). Если тебе важна производительность SSD и ты готов немного снизить криптостойкость (теоретическая атака по анализу занятости), можно включить TRIM:
    - При открытии: `cryptsetup open --allow-discards ...`
    - В `/etc/crypttab`: добавь опцию `discard`.
    - _Внимание:_ Включай это только если понимаешь риски утечки метаданных о том, какие блоки заняты.
3. **Бэкап заголовка LUKS:** Заголовок LUKS содержит все ключи. Если он повредится (битый сектор в начале диска), ты потеряешь доступ ко всем данным, даже зная пароль. Сделай бэкап заголовка сразу после создания и храни этот файл в надежном месте (не на этом же диске!). Восстановить можно через `luksHeaderRestore`:
```bash
cryptsetup luksHeaderBackup /dev/sdb --header-backup-file /root/luks-sdb-header.img
```

4. **Связка с LVM и загрузкой:** Если ты шифруешь корневую систему (`/`), то GRUB должен уметь запрашивать пароль до загрузки ядра. В Debian это настраивается автоматически при установке, если выбрать опцию шифрования. В существующую систему это добавить сложно (нужно пересобирать `initramfs` и настраивать `cryptroot`). Для дисков с данными (`/home`, `/var`, базы) схема, описанная выше, подходит идеально.
