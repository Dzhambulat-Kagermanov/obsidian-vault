### Терминология:

**SCSI (Small Computer System Interface)** — это протокол для физического подключения и передачи данных между компьютерами и периферийными устройствами, такими как жесткие диски, стримеры и сканеры. Обеспечивает высокую скорость, надежность и позволяет объединять множество устройств на одной шине, что делает его популярным в серверах и рабочих станциях. В современных серверах чистый параллельный **SCSI (со шлейфами)** уже не используется. Его наследник — **SAS**. Когда говорят **«SCSI-диск»** в контексте современного железа, почти всегда имеют в виду **SAS** или виртуальный **SCSI в виртуализации**. Характеристика SCSI:
- **Назначение:** Корпоративный сегмент, серверы, СХД (системы хранения данных);
- **Архитектура:**
    - **Физический SAS:** Позволяет подключать несколько устройств к одному порту контроллера (через экспандеры), подобно тому, как USB-хаб работает с флешками, но на гораздо более высоком уровне надежности;
    - **Виртуальный SCSI:** В виртуализации (Proxmox, VMware, с которыми ты знаком) диск виртуальной машине часто представляется как устройство **VirtIO-SCSI** или просто **SCSI**. Это не физический интерфейс, а протокол обмена данными между гипервизором и гостевой ОС;
- **Очереди команд:** Поддерживают огромные очереди команд (до тысяч). Это критично для СУБД (PostgreSQL) и файловых серверов, где сотни запросов идут одновременно. Диск эффективно перестраивает их выполнение, минимизируя движение головок (для HDD) или обращений к ячейкам (для SSD);
- **Дуплекс:** Полнодуплексная передача (одновременная запись и чтение);
- **Надежность:** Диски рассчитаны на работу 24/7 под высокой нагрузкой (MTBF значительно выше, чем у SATA).

**SATA (Serial ATA)** — это современный последовательный интерфейс для подключения накопителей данных (HDD, SSD) и оптических приводов к материнской плате компьютера. Он пришел на смену устаревшему, медленному стандарту PATA) (IDE), предлагая более высокие скорости, удобные тонкие кабели и поддержку «горячей замены» (в большинстве реализаций). Характеристика SATA:
- **Назначение:** Массовый сегмент. Используется в домашних ПК, ноутбуках и недорогих серверах;
- **Архитектура:** Точка-точка (Point-to-Point). Каждый кабель соединяет контроллер и **одно** устройство;
- **Очереди команд:** Использует NCQ (Native Command Queuing), но очередь относительно небольшая (до 32 команд). Это значит, что диск сам оптимизирует порядок выполнения запросов, но при высокой нагрузке от многих пользователей он может стать «узким местом»;
- **Надежность и скорость:** Хорошие показатели для одиночных задач, но при одновременной работе множества потоков (как в базе данных PostgreSQL, с которой ты работаешь) производительность падает. Кабели тонкие, длина до 1 метра;
- **Как видит ОС:** Обычно как `/dev/sda`, `/dev/sdb`. Для ОС это просто блочное устройство.

**Сетевой диск** — это логическое представление удаленного ресурса хранения данных, доступ к которому осуществляется через сеть по определенному протоколу. Для операционной системы и приложений это выглядит как обычная папка или даже отдельный буквенный диск (в Windows, например, `Z:`), но физически данные находятся на другом компьютере или устройстве (NAS, SAN, файловом сервере). 

В отличие от локального диска, где ОС отправляет команды напрямую контроллеру (SATA/SCSI), здесь работает клиент-серверная модель на уровне файлов или блоков:
- **Файловый уровень (File-level)** (Для расшаривания файлов или директорий):
    - **SMB/CIFS:** Стандарт для Windows и смешанных сред. Монтируется общая папка (`\\server\share`), и она становится сетевым диском;
    - **NFS (Network File System):** Стандарт для Linux/Unix. Позволяет «пробросить» директорию с сервера на клиент так, будто она часть локальной файловой системы (`mount -t nfs server:/export/data /mnt/data`). Видит как файловую систему. Команды `open()`, `read()`, `write()` транслируются в сетевые пакеты;
- **Блочный уровень (Block-level)** (Для расшаривания блочных устройств):
    - **iSCSI:** Позволяет передавать SCSI-команды через TCP/IP сеть. Сервер «отдает» клиенту целый блок данных (LUN), который клиент видит как **физический диск** (например, `/dev/sdb`), хотя кабеля SATA к нему нет. На этом «диске» клиент создает свои разделы и файловые системы.
    - **FC (Fibre Channel):** Аналог iSCSI, но через специализированную оптоволоконную сеть.

**Ключевые отличия от локального диска:**

- **Задержка (Latency):** Доступ к сетевому диску всегда медленнее из-за времени прохождения сигнала по сети и обработки стека протоколов.
- **Зависимость от сети:** Если сеть упала, диск становится недоступным (в отличие от локального, который может работать даже при сбое ОС, если железо цело).
- **Блокировки:** Протоколы сетевого доступа (особенно SMB) управляют блокировками файлов, чтобы два пользователя не редактировали один файл одновременно и не повредили его. Локальная ФС делает это через ядро, сетевая — через согласование по сети.

### Настройка файлового сервера на Samba:

В мире Linux стандартом де-факто является пакет **Samba**, который реализует протокол SMB/CIFS. 

##### Шаг 1. Установка пакетов:

```bash
apt update && apt install samba samba-common-bin
```

`samba` — сервер samba; 
`samba-common-bin` — утилиты для управления (например, `smbpasswd`).

##### Шаг 2. Создание структуры папок и прав:

Допустим, мы создаем общую папку `/srv/samba/share`.

```bash
mkdir -p /srv/samba/share
chown -R nobody:nogroup /srv/samba/share
chmod -R 0775 /srv/samba/share
```

#####  Шаг 3. Конфигурация (`/etc/samba/smb.conf`).

Это главный файл конфигурации и стоит сделать бэкап перед правкой:

```bash
cp /etc/samba/smb.conf /etc/samba/smb.conf.bak
```

Редактирование `/etc/samba/smb.conf`:

```vim
[MyShare]
    comment = Shared Folder for Dzhambulat
    path = /srv/samba/share
    browseable = yes
    read only = no
    guest ok = no
    create mask = 0644
    directory mask = 0755
    valid users = @smbusers, altlinux
    force user = altlinux
```

**Детальный разбор параметров секции:**

| Параметр             | Значение по умолчанию / Пример | Описание                                                                                                                                                                           |
| -------------------- | ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`comment`**        | `Comment = Project Files`      | Текстовое описание, которое видно в списке сетей рядом с именем шары.                                                                                                              |
| **`path`**           | `path = /srv/samba/data`       | **Абсолютный путь** к реальной директории в файловой системе Linux. Без этого параметра шара не работает.                                                                          |
| **`browseable`**     | `yes` / `no`                   | Видна ли шара в списке при обзоре сети. Если `no`, подключиться можно только зная точное имя (`\\server\HiddenShare`), но в списке её не будет.                                    |
| **`read only`**      | `yes` / `no`                   | Запрещает или разрешает запись. `read only = no` равносильно `writable = yes`.                                                                                                     |
| **`guest ok`**       | `no` / `yes`                   | Разрешает ли вход без пароля (как пользователь `nobody`). Для безопасности лучше `no`.                                                                                             |
| **`valid users`**    | `user1, @devs`                 | Список пользователей (`user1`) или групп (`@devs`), которым **разрешено** подключение. Все остальные получат отказ.                                                                |
| **`invalid users`**  | `root, @banned`                | Список тех, кому доступ **запрещен** явно (даже если они есть в valid users).                                                                                                      |
| **`write list`**     | `user1, @admins`               | Более тонкая настройка: даже если шара `read only = yes`, пользователи из этого списка смогут писать.                                                                              |
| **`create mask`**    | `0644`                         | Права (chmod) для **новых файлов**, создаваемых через SMB. (Например, `rw-r--r--`).                                                                                                |
| **`directory mask`** | `0755`                         | Права для **новых папок**.                                                                                                                                                         |
| **`force user`**     | `altlinux`                     | **Важно!** Все операции записи будут выполняться от имени этого пользователя Linux, независимо от того, кто зашел по SMB. Решает проблемы с правами доступа между разными юзерами. |
| **`force group`**    | `developers`                   | Аналогично `force user`, но для группы. Все файлы будут принадлежать этой группе.                                                                                                  |
| **`vfs objects`**    | `acl_xattr`                    | Позволяет хранить ACL (списки контроля доступа Windows) в расширенных атрибутах файлов Linux. Нужно для тонких прав.                                                               |

**Что такое `MyShare`?**

- Это не имя папки на диске Linux (путь задается параметром `path`).
- Это имя, которое увидят пользователи в сети.
- Когда пользователь в Windows или Linux вводит адрес `\\server\MyShare` или `smb://server/MyShare`, он обращается именно к этой секции конфигурации.
- Ты можешь назвать её как угодно: `[Documents]`, `[DevProject]`, `[JambulatData]`. Имя может содержать пробелы, но тогда при подключении через CLI их нужно будет экранировать или брать в кавычки.

**Как задавать права для каждого пользователя отдельно?**

Способ 1. Разные шары для разных пользователей. Создаются разные секции в конфиге с разными путями и параметрами `valid users`:

```vim
[UserA_Private]
    path = /srv/samba/userA
    valid users = userA
    force user = userA
    read only = no

[UserB_Private]
    path = /srv/samba/userB
    valid users = userB
    force user = userB
    read only = no

[Common_ReadOnly]
    path = /srv/samba/common
    valid users = userA, userB
    read only = yes
    # Но userA может писать в свою подпапку через write list, если настроить аккуратно
```

Способ 1. Одна шара, разные права внутри (Продвинутый с ACL):

1. Включаем поддержку ACL в `smb.conf`:

```vim
[Project]
    path = /srv/samba/project
    valid users = userA, userB
    # Не используем force user, чтобы сохранялся реальный владелец файла
    read only = no
    create mask = 0660
    directory mask = 0770
    # Включаем обработку ACL
    vfs objects = acl_xattr
    map acl inherit = Yes
    store dos attributes = Yes
```

2. Настраиваем права на файлы и директории. Можно настроить через `chown` и `chmod`, а можно настроить через `setfacl`:

```bash
setfacl -R -m u:userA:rwx /srv/samba/project/FolderA # Полный доступ к папке A
setfacl -R -m u:userB:--- /srv/samba/project/FolderA # Явный запрет для B

setfacl -R -m u:userB:rwx /srv/samba/project/FolderB # Полный доступ к папке B
setfacl -R -m u:userA:--- /srv/samba/project/FolderB # Явный запрет для A
```

##### Шаг 4. Управление пользователями Samba:

Пользователи Samba **не зависят** напрямую от системных пользователей Linux в плане паролей. У них свой парольный файл (`tdbsam`). Чтобы добавить пользователя (он должен существовать в системе Linux):

```bash
# Создаем системного пользователя, если нет
useradd -M -s /usr/sbin/nologin smbuser1

# Добавляем его в базу Samba и задаем пароль
smbpasswd -a smbuser1
smbpasswd -a altlinux
```

##### Шаг 5. Проверка конфига и перезапуск:

```bash
testparm

# Если ошибок нет, перезапкаем службу:
systemctl restart smbd
systemctl enable smbd
```

##### Шаг 6. Firewall:

Если включен firewall, нужно открыть порты SMB:
* 139/tcp;
* 445/tcp;
* 137/udp;
* 138/udp.

##### Шаг 7. Подключение к SMB с Linux-клиента:

Для подключения тебе понадобится пакет `cifs-utils`:

**Способ 1. Подключение с терминала:**

```bash
apt-get update && apt-get install cifs-utils

# Создаем точку монтирования
mkdir -p /mnt/smb_share

# Монтируем
mount -t cifs //192.168.1.50/MyShare /mnt/smb_share -o username=altlinux,password=your_password,uid=1000,gid=1000
```

**Разбор опций `-o`:**

- `username`, `password`: Учетные данные.
- `uid`, `gid`: **Критично для Linux-клиента.** Эти параметры говорят ядру: "Все файлы на этой шаре считай принадлежащими пользователю с ID 1000". Иначе файлы могут принадлежать `root` или тому, кто монтировал, и обычный пользователь не сможет их читать/писать без `sudo`.
- `domain`: Если используется домен AD (например, `domain=MYCORP`).
- `vers=3.0`: Принудительно указать версию протокола (иногда нужно для совместимости).

**Безопасное введение пароля.** Чтобы не светить пароль в истории bash (`history`), можно использовать файл учетных данных (credentials):

_~/.smbcreds:_
```vim 
username=altlinux
password=SuperSecretPassword
```

```bash
chmod 600 ~/.smbcreds

mount -t cifs //192.168.1.50/MyShare /mnt/smb_share -o credentials=/home/altlinux/.smbcreds,uid=1000,gid=1000
```

Можно настроить автоматическое монтирование в /etc/fstab:

```vim
//192.168.1.50/MyShare  /mnt/smb_share  cifs  credentials=/home/altlinux/.smbcreds,uid=1000,gid=1000,iocharset=utf8,file_mode=0770,dir_mode=0770  0  0
```


**Способ 2. Подключение через GUI:** 

Можно подключиться как на Windows открыв файловый менеджер и введя путь до шары `smb://192.168.1.50/MyShare`. Система запросит логин/пароль и предложит запомнить их в связке ключей (Keyring).

### Настройка iSCSI:

#### Терминология:

- **Target (Цель):** Это **сервер** хранения. Устройство, которое «предоставляет» диск. У него есть имя (IQN) и он экспортирует логические тома (LUN);
- **Initiator (Инициатор):** Это **клиент**. Сервер или ПК, который хочет использовать удаленный диск. Он «находит» Target и подключается к нему;
- **LUN (Logical Unit Number):** Это сам «виртуальный диск», который Target отдает клиенту. Один Target может отдавать много LUN разным клиентам;
- **IQN (iSCSI Qualified Name):** Уникальное имя устройства в формате: `iqn.год-месяц.домен:имя` Пример: `iqn.2026-03.com.example.storage:disk1`.

#### Фундаментальная разница: Файл vs Блок

##### Файл-образ (`fileio`):

Создаётся обычный файл большого размера (например, 10 ГБ) внутри существующей файловой системы (ext4/xfs) на диске `/dev/sdb`.

- **Путь:** Физический диск -> Файловая система (ext4) -> Файл `disk1.img` -> iSCSI Target.
- **Как это видит ядро:** Для ядра это просто файл. Когда iSCSI получает запрос «записать блок №500», он транслируется в системный вызов `pwrite()` к файлу. Файловая система решает, куда именно на физическом диске положить эти данные.
- **Накладные расходы:** Двойная работа.
    1. iSCSI обрабатывает запрос.
    2. Файловая система (VFS) обрабатывает запрос, ищет inode файла, проверяет права, вычисляет смещение.
    3. Журнал ФС (journal) может записывать метаданные.
    4. Только потом данные пишутся на диск.
- **Фрагментация:** Так как файл лежит среди других файлов, он может быть фрагментирован. Головка диска будет прыгать, читая части файла из разных мест, что убивает скорость.
- **Когда использовать:** Тесты, демо-стенды, домашние лаборатории, где важна простота создания/удаления (просто `rm файл`), а не максимальная скорость.

##### Реальный блок (`block`):

Ты отдаешь iSCSI прямой доступ к разделу диска (например, `/dev/sdb1`) или всему диску (`/dev/sdb`).

- **Путь:** Физический диск -> iSCSI Target. **Файловой системы нет**;
- **Как это видит ядро:** iSCSI работает напрямую с блочным драйвером устройства. Запрос «записать блок №500» превращается в прямую команду контроллеру диска: «Запиши данные по LBA 500»;
- **Накладные расходы:** Минимальные. Нет слоя файловой системы, нет проверки inode, нет журналирования метаданных ФС;
- **Выравнивание (Alignment):** Данные пишутся ровно так, как просит клиент. Нет риска фрагментации файла;
- **Когда использовать:** Продакшен, базы данных (PostgreSQL), виртуальные машины, любые задачи, где важна скорость ввода-вывода (IOPS) и низкая задержка (latency).

#### Настройка iSCSI Target:

##### Шаг 1. Установка пакетов:

```bash
apt-get update && apt-get install targetcli-fb

systemctl enable --now target
```

##### Шаг 2. Создание Backstore (Хранилища):

Вход в оболочку:

```bash
targetcli
```

Команда для создания файла (fileio):

```targetcli
/backstores/fileio create disk1 /srv/iscsi_images/disk1.img 10G
```

**Что происходит:**
1. `targetcli` вызывает системную функцию для создания файла размером 10 ГБ по пути `/srv/iscsi_images/disk1.img`;
2. Если файл уже есть, он просто открывается;
3. В памяти создается объект backstore, который знает: «Я представляю этот файл»;
4. **Под капотом:** Файловая система резервирует место (или создает разреженный файл, если не указано иное). При записи данных iSCSI будет использовать буфер страницы ядра (Page Cache) для этого файла. Это значит, что часть оперативной памяти будет занята под кэш этого файла, конкурируя с другими процессами.


Команда для создания блока (block):

```targetcli
/backstores/block create disk1 /dev/sdb1
```

**Что происходит:**
1. `targetcli` проверяет, существует ли устройство `/dev/sdb1`;
2. Оно открывает устройство в режиме raw (сыром);
3. **Важно:** Оно **не создает** никакую файловую систему на этом разделе. Если там были данные, они могут быть перезаписаны при использовании. Обычно перед этим делают `fdisk /dev/sdb` и создают чистый раздел;
4. **Под капотом:** Создается привязка к блочному маппингу ядра. Когда приходят данные, они сразу отправляются в очередь запросов блочного устройства (Request Queue) драйвера диска. Обходится слой VFS (Virtual File System).

**Почему блок лучше?** Потому убирается посредника (ФС). В базах данных каждый лишний микросекунд задержки важен. Файловая система добавляет свою логику (кто владелец файла, когда последний доступ, журналы транзакций ФС), которая здесь абсолютно лишняя.

##### Шаг 3. Создание target цели: 

```targetcli
/iscsi create iqn.2026-03.local.dzhambulat:storage1
```

**Что происходит:**
1. В ядре загружается модуль `iscsi_target_mod` (если еще не загружен);
2. Создается новая структура данных в пространстве имен iSCSI с уникальным именем (IQN);
3. Автоматически создается **TPG (Target Portal Group)** — группа порталов. По умолчанию создается `tpg1`;
4. Внутри TPG создается слушатель на порту **3260** (стандартный порт iSCSI). Теперь ядро слушает TCP-соединения на этом порту и ждет пакетов с этим конкретным IQN;


##### Шаг 4. Привязка LUN (Логического устройства): 

```targetcli
/iscsi/iqn.../tpg1/luns create /backstores/block/disk1

# Или fileio если выбран файл-образ
/iscsi/iqn.../tpg1/luns create /backstores/fileio/disk1
```

**Что происходит:**
1. Мы говорим системе: «Возьми объект `disk1` (который мы создали в Шаге 1) и предъяви его миру как **LUN 0** внутри этой цели»;
2. **Под капотом:** Создается таблица соответствия. Когда клиент пришлет команду SCSI `READ(10)` с адресом блока X для LUN 0, ядро поймет: «Ага, это запрос к backstore `disk1`»;
3. Если это `block`, ядро пересчитает адрес блока и отправит команду прямо на `/dev/sdb1`;
4. Если это `fileio`, ядро пересчитает адрес блока в смещение внутри файла и вызовет чтение файла.

##### Шаг 5. Настройка контроля доступа:

**Ограничение по IQN:**

```targetcli
/iscsi/iqn.../tpg1/acls create iqn.1994-05.com.redhat:client-hostname
```

`client-hostname` — это уникальный идентификатор который генерируется на client-е.

iSCSI работает поверх обычной IP-сети. Любой, кто знает IP и порт, теоретически может попытаться подключиться.

**Что происходит:**
1. В конфигурации TPG создается запись: «Разрешить подключение только инициаторам с именем `iqn.1994...`».
2. Когда клиент пытается соединиться, он отправляет свое имя (Initiator Name) в процессе рукопожатия (Login Phase).
3. Ядро сверяет полученное имя со списком ACL. Если совпадения нет — соединение разрывается немедленно.
4. **Без этого шага:** По умолчанию многие конфигурации запрещают все подключения (демо-режим выключен), либо (если включен демо-режим) разрешают всем, что является дырой в безопасности.

ACL в iSCSI проверяет только совпадение строки и поэтому злоумышленник может подделать (спуфить) IQN, если у него нет другой защиты

**Парольная аутентификация (CHAP):**

- **Как работает:** Сервер отправляет клиенту «challenge» (случайное число). Клиент шифрует его своим паролем (секретом) и отправляет обратно. Сервер проверяет ответ. Сам пароль по сети не передается.
- **Типы:**
    - **One-way CHAP:** Клиент доказывает серверу, что он знает пароль. (Самый частый вариант).
    - **Mutual CHAP:** И клиент доказывает серверу, и сервер доказывает клиенту свою легитимность. (Максимальная защита от фейковых серверов).

**Настройка One-way CHAP в `targetcli`:**

```targetcli
cd /iscsi/iqn.2026-03.local.dzhambulat:storage1/tpg1
```

Включение аутентификации и задание пароля для всех IQN:

```targetcli
set attribute authentication=1
set attribute generate_node_acls=0
set attribute cache_dynamic_acls=0

# Задаем логин и пароль для доступа (пользователь и секрет)
set attribute chap_userid=mysecretuser
set attribute chap_password=SuperStrongPassword123
```

**Более надежный способ (через ACL).** В современных версиях лучше задавать CHAP для конкретного инициатора в ACL:

```bash
cd /iscsi/iqn.2026-03.local.dzhambulat:storage1/tpg1/acls
# Создаем ACL для клиента
create iqn.1994-05.com.redhat:client-hostname

# Заходим в созданную ноду ACL
cd iqn.1994-05.com.redhat:client-hostname

# Задаем параметры CHAP именно для этого клиента
set auth userid=mysecretuser
set auth password=SuperStrongPassword123
```


##### Шаг 6: Сохранение (`saveconfig`):

Вся настройка, которая выполнено, хранится **в оперативной памяти** ядра и в структуре данных `targetcli` из-за чего ее нужно сохранить.

```targetcli
saveconfig
```

**Что делает команда:**
1. Она берет текущее состояние дерева конфигурации из памяти;
2. Сериализует его в YAML-формат;
3. Записывает в файл `/etc/target/saveconfig.json` (путь может отличаться в разных дистрибутивах, но суть та же);
4. Настраивает службу `target` так, чтобы при следующей загрузке ОС она прочитала этот JSON и восстановила все backstores, targets и ACL в память.
