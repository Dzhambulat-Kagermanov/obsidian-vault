### Терминология:

**SCSI (Small Computer System Interface)** — это протокол для физического подключения и передачи данных между компьютерами и периферийными устройствами, такими как жесткие диски, стримеры и сканеры. Обеспечивает высокую скорость, надежность и позволяет объединять множество устройств на одной шине, что делает его популярным в серверах и рабочих станциях. В современных серверах чистый параллельный **SCSI (со шлейфами)** уже не используется. Его наследник — **SAS**. Когда говорят **«SCSI-диск»** в контексте современного железа, почти всегда имеют в виду **SAS** или виртуальный **SCSI в виртуализации**. Характеристика SCSI:
- **Назначение:** Корпоративный сегмент, серверы, СХД (системы хранения данных);
- **Архитектура:**
    - **Физический SAS:** Позволяет подключать несколько устройств к одному порту контроллера (через экспандеры), подобно тому, как USB-хаб работает с флешками, но на гораздо более высоком уровне надежности;
    - **Виртуальный SCSI:** В виртуализации (Proxmox, VMware, с которыми ты знаком) диск виртуальной машине часто представляется как устройство **VirtIO-SCSI** или просто **SCSI**. Это не физический интерфейс, а протокол обмена данными между гипервизором и гостевой ОС;
- **Очереди команд:** Поддерживают огромные очереди команд (до тысяч). Это критично для СУБД (PostgreSQL) и файловых серверов, где сотни запросов идут одновременно. Диск эффективно перестраивает их выполнение, минимизируя движение головок (для HDD) или обращений к ячейкам (для SSD);
- **Дуплекс:** Полнодуплексная передача (одновременная запись и чтение);
- **Надежность:** Диски рассчитаны на работу 24/7 под высокой нагрузкой (MTBF значительно выше, чем у SATA).

**SATA (Serial ATA)** — это современный последовательный интерфейс для подключения накопителей данных (HDD, SSD) и оптических приводов к материнской плате компьютера. Он пришел на смену устаревшему, медленному стандарту PATA) (IDE), предлагая более высокие скорости, удобные тонкие кабели и поддержку «горячей замены» (в большинстве реализаций). Характеристика SATA:
- **Назначение:** Массовый сегмент. Используется в домашних ПК, ноутбуках и недорогих серверах;
- **Архитектура:** Точка-точка (Point-to-Point). Каждый кабель соединяет контроллер и **одно** устройство;
- **Очереди команд:** Использует NCQ (Native Command Queuing), но очередь относительно небольшая (до 32 команд). Это значит, что диск сам оптимизирует порядок выполнения запросов, но при высокой нагрузке от многих пользователей он может стать «узким местом»;
- **Надежность и скорость:** Хорошие показатели для одиночных задач, но при одновременной работе множества потоков (как в базе данных PostgreSQL, с которой ты работаешь) производительность падает. Кабели тонкие, длина до 1 метра;
- **Как видит ОС:** Обычно как `/dev/sda`, `/dev/sdb`. Для ОС это просто блочное устройство.

**Сетевой диск** — это логическое представление удаленного ресурса хранения данных, доступ к которому осуществляется через сеть по определенному протоколу. Для операционной системы и приложений это выглядит как обычная папка или даже отдельный буквенный диск (в Windows, например, `Z:`), но физически данные находятся на другом компьютере или устройстве (NAS, SAN, файловом сервере). 

В отличие от локального диска, где ОС отправляет команды напрямую контроллеру (SATA/SCSI), здесь работает клиент-серверная модель на уровне файлов или блоков:
- **Файловый уровень (File-level)** (Для расшаривания файлов или директорий):
    - **SMB/CIFS:** Стандарт для Windows и смешанных сред. Монтируется общая папка (`\\server\share`), и она становится сетевым диском;
    - **NFS (Network File System):** Стандарт для Linux/Unix. Позволяет «пробросить» директорию с сервера на клиент так, будто она часть локальной файловой системы (`mount -t nfs server:/export/data /mnt/data`). Видит как файловую систему. Команды `open()`, `read()`, `write()` транслируются в сетевые пакеты;
- **Блочный уровень (Block-level)** (Для расшаривания блочных устройств):
    - **iSCSI:** Позволяет передавать SCSI-команды через TCP/IP сеть. Сервер «отдает» клиенту целый блок данных (LUN), который клиент видит как **физический диск** (например, `/dev/sdb`), хотя кабеля SATA к нему нет. На этом «диске» клиент создает свои разделы и файловые системы.
    - **FC (Fibre Channel):** Аналог iSCSI, но через специализированную оптоволоконную сеть.

**Ключевые отличия от локального диска:**

- **Задержка (Latency):** Доступ к сетевому диску всегда медленнее из-за времени прохождения сигнала по сети и обработки стека протоколов.
- **Зависимость от сети:** Если сеть упала, диск становится недоступным (в отличие от локального, который может работать даже при сбое ОС, если железо цело).
- **Блокировки:** Протоколы сетевого доступа (особенно SMB) управляют блокировками файлов, чтобы два пользователя не редактировали один файл одновременно и не повредили его. Локальная ФС делает это через ядро, сетевая — через согласование по сети.

### Настройка файлового сервера на Samba:

В мире Linux стандартом де-факто является пакет **Samba**, который реализует протокол SMB/CIFS. 

**Шаг 1. Установка пакетов.**

```bash
apt update && apt install samba samba-common-bin
```

`samba` — сервер samba; 
`samba-common-bin` — утилиты для управления (например, `smbpasswd`).

**Шаг 2. Создание структуры папок и прав.** Допустим, мы создаем общую папку `/srv/samba/share`.

```bash
mkdir -p /srv/samba/share
chown -R nobody:nogroup /srv/samba/share
chmod -R 0775 /srv/samba/share
```

**Шаг 3. Конфигурация (`/etc/samba/smb.conf`).** Это главный файл конфигурации.

Стоит сделать бэкап перед правкой:

```bash
cp /etc/samba/smb.conf /etc/samba/smb.conf.bak
```

Редактирование `/etc/samba/smb.conf`:

```vim
[MyShare]
    comment = Shared Folder for Dzhambulat
    path = /srv/samba/share
    browseable = yes
    read only = no
    guest ok = no
    create mask = 0644
    directory mask = 0755
    valid users = @smbusers, altlinux
    force user = altlinux
```

**Детальный разбор параметров секции:**

| Параметр             | Значение по умолчанию / Пример | Описание                                                                                                                                                                           |
| -------------------- | ------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`comment`**        | `Comment = Project Files`      | Текстовое описание, которое видно в списке сетей рядом с именем шары.                                                                                                              |
| **`path`**           | `path = /srv/samba/data`       | **Абсолютный путь** к реальной директории в файловой системе Linux. Без этого параметра шара не работает.                                                                          |
| **`browseable`**     | `yes` / `no`                   | Видна ли шара в списке при обзоре сети. Если `no`, подключиться можно только зная точное имя (`\\server\HiddenShare`), но в списке её не будет.                                    |
| **`read only`**      | `yes` / `no`                   | Запрещает или разрешает запись. `read only = no` равносильно `writable = yes`.                                                                                                     |
| **`guest ok`**       | `no` / `yes`                   | Разрешает ли вход без пароля (как пользователь `nobody`). Для безопасности лучше `no`.                                                                                             |
| **`valid users`**    | `user1, @devs`                 | Список пользователей (`user1`) или групп (`@devs`), которым **разрешено** подключение. Все остальные получат отказ.                                                                |
| **`invalid users`**  | `root, @banned`                | Список тех, кому доступ **запрещен** явно (даже если они есть в valid users).                                                                                                      |
| **`write list`**     | `user1, @admins`               | Более тонкая настройка: даже если шара `read only = yes`, пользователи из этого списка смогут писать.                                                                              |
| **`create mask`**    | `0644`                         | Права (chmod) для **новых файлов**, создаваемых через SMB. (Например, `rw-r--r--`).                                                                                                |
| **`directory mask`** | `0755`                         | Права для **новых папок**.                                                                                                                                                         |
| **`force user`**     | `altlinux`                     | **Важно!** Все операции записи будут выполняться от имени этого пользователя Linux, независимо от того, кто зашел по SMB. Решает проблемы с правами доступа между разными юзерами. |
| **`force group`**    | `developers`                   | Аналогично `force user`, но для группы. Все файлы будут принадлежать этой группе.                                                                                                  |
| **`vfs objects`**    | `acl_xattr`                    | Позволяет хранить ACL (списки контроля доступа Windows) в расширенных атрибутах файлов Linux. Нужно для тонких прав.                                                               |

**Что такое `MyShare`?**

- Это не имя папки на диске Linux (путь задается параметром `path`).
- Это имя, которое увидят пользователи в сети.
- Когда пользователь в Windows или Linux вводит адрес `\\server\MyShare` или `smb://server/MyShare`, он обращается именно к этой секции конфигурации.
- Ты можешь назвать её как угодно: `[Documents]`, `[DevProject]`, `[JambulatData]`. Имя может содержать пробелы, но тогда при подключении через CLI их нужно будет экранировать или брать в кавычки.

**Как задавать права для каждого пользователя отдельно?**

Способ 1. Разные шары для разных пользователей. Создаются разные секции в конфиге с разными путями и параметрами `valid users`:

```vim
[UserA_Private]
    path = /srv/samba/userA
    valid users = userA
    force user = userA
    read only = no

[UserB_Private]
    path = /srv/samba/userB
    valid users = userB
    force user = userB
    read only = no

[Common_ReadOnly]
    path = /srv/samba/common
    valid users = userA, userB
    read only = yes
    # Но userA может писать в свою подпапку через write list, если настроить аккуратно
```

Способ 1. Одна шара, разные права внутри (Продвинутый с ACL):

1. Включаем поддержку ACL в `smb.conf`:

```vim
[Project]
    path = /srv/samba/project
    valid users = userA, userB
    # Не используем force user, чтобы сохранялся реальный владелец файла
    read only = no
    create mask = 0660
    directory mask = 0770
    # Включаем обработку ACL
    vfs objects = acl_xattr
    map acl inherit = Yes
    store dos attributes = Yes
```

2. Настраиваем права на файлы и директории. Можно настроить через `chown` и `chmod`, а можно настроить через `setfacl`:

```bash
setfacl -R -m u:userA:rwx /srv/samba/project/FolderA # Полный доступ к папке A
setfacl -R -m u:userB:--- /srv/samba/project/FolderA # Явный запрет для B

setfacl -R -m u:userB:rwx /srv/samba/project/FolderB # Полный доступ к папке B
setfacl -R -m u:userA:--- /srv/samba/project/FolderB # Явный запрет для A
```

**Шаг 4. Управление пользователями Samba.** Пользователи Samba **не зависят** напрямую от системных пользователей Linux в плане паролей. У них свой парольный файл (`tdbsam`). Чтобы добавить пользователя (он должен существовать в системе Linux):

```bash
# Создаем системного пользователя, если нет
useradd -M -s /usr/sbin/nologin smbuser1

# Добавляем его в базу Samba и задаем пароль
smbpasswd -a smbuser1
smbpasswd -a altlinux
```

**Шаг 5. Проверка конфига и перезапуск.** 

```bash
testparm

# Если ошибок нет, перезапкаем службу:
systemctl restart smbd
systemctl enable smbd
```

**Шаг 6. Firewall.** Если включен firewall, нужно открыть порты SMB:
* 139/tcp;
* 445/tcp;
* 137/udp;
* 138/udp.

**Шаг 7. Подключение к SMB с Linux-клиента.** Для подключения тебе понадобится пакет `cifs-utils`:

```bash
apt-get update && apt-get install cifs-utils

# Создаем точку монтирования
mkdir -p /mnt/smb_share

# Монтируем
mount -t cifs //192.168.1.50/MyShare /mnt/smb_share -o username=altlinux,password=your_password,uid=1000,gid=1000
```

**Разбор опций `-o`:**

- `username`, `password`: Учетные данные.
- `uid`, `gid`: **Критично для Linux-клиента.** Эти параметры говорят ядру: "Все файлы на этой шаре считай принадлежащими пользователю с ID 1000". Иначе файлы могут принадлежать `root` или тому, кто монтировал, и обычный пользователь не сможет их читать/писать без `sudo`.
- `domain`: Если используется домен AD (например, `domain=MYCORP`).
- `vers=3.0`: Принудительно указать версию протокола (иногда нужно для совместимости).

**Безопасное введение пароля.** Чтобы не светить пароль в истории bash (`history`), можно использовать файл учетных данных (credentials):

_~/.smbcreds_
```vim 


username=altlinux
password=SuperSecretPassword
```
